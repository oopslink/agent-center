v2.7.1 当前v2.7 历史
A · 技术架构总览

agent-center 是一个用 Go 编写的多智能体编排平台，整体为「一个 server + N 个 worker」的拓扑：server 持有全部业务真相与读模型，worker 在任意机器上托管真正执行任务的智能体进程。所有跨进程通信走 server 暴露的 HTTP/SSE 控制信道；持久化默认落 SQLite，二进制内容落本地内容寻址 blob store。组合根（composition root）在 cmd/agent-center/main.go → internal/cli，并由 internal/cli + internal/webconsole/api + internal/workerdaemon 三处装配。

B · 主要机制设计

1 · 任务 / 议题流转（Task / Issue lifecycle）

项目管理（projectmanager）是工作真相源。议题 Issue 承载问题/讨论，任务 Task 承载可执行单元；两者各按 1:1 绑定一条会话（由 Conversation 的 ownerRef 持有，议题不存绑定字段）。v2.7.1 起，每个议题/任务在组织内分配单调的 I<n> / T<n> 引用号（orgNumber，#245）。

Issue： open→ in_progress→ resolved→ closed | reopened | withdrawn（终态）
Task： open→ assigned→ running→ blocked / reopened→ completed · verified · canceled（终态）

状态迁移由聚合内 CanTransitionTo 强制（issue.go:37 / task.go:40）；Task 的终态集合恰为 {completed, verified, canceled}（IsTerminal task.go:67），非终态集合即可观测性默认的「在飞工作」查询集。

2 · 智能体调度与 work item 派发

任务派给智能体后，智能体侧（agent）建立一条 AgentWorkItem，以 taskRef = pm://tasks/{id} 非拥有引用任务（ACL，不导入 pm 结构）。核心不变式：同一智能体同时只有一条活跃工作项（repository.go:49 HasActiveWorkItem）；重新派发会把旧工作项置 superseded、新建 queued。

queued→ active⇄ waiting_input→ done · failed · canceled · superseded

智能体本身的运行时生命周期与工作项是两个不同聚合：智能体生命周期 = stopped / running / stopping / resetting / error / failed（agent.go:52，failed 为崩溃熔断终态，需人工 Start/Reset 恢复）。

3 · 事件与可观测性（append-only events）

每个核心 BC 在状态变更的同一事务内追加领域事件，经 transactional outbox 可靠发布；可观测性（observability）是纯下游消费者（61 处引用），仅持有单一追加写聚合 Event 与投影读模型，对前端/运维以只读 OHS 暴露。真实事件名（点分小写）举例：

pm.task.createdpm.task.state_changedpm.task.assignedpm.issue.createdpm.issue.state_changedagent.lifecycle_changedagent.work_item_transitionedconversation.message_addedworkforce.worker.online

4 · 会话作为统一线程（Conversation）

会话（conversation）是统一的消息流 + 参与者。ConversationKind 恰为 4 个值：dm / channel / task / issue（types.go:56；旧的 adhoc/notification 已移除）。任务/议题各绑定一条会话（ownerRef = pm://tasks|issues/{id}，1:1）；参与者镜像项目管理的订阅真相（Conformist），由 participant_projector 投影。消息只在显式发布时入会话；智能体冗长的中间输出落 AgentActivityEvent 观测流，不污染会话线程。

5 · InputRequest 作为一等公民

当智能体需要人介入时，工作项进入 waiting_input，并产生一条携带 inputRequestRef 的消息（Message 没有 work_item_ref 字段，是 inputRequestRef，message.go:13）。人在会话线程中回应即解除阻塞，工作项回到 active。输入请求因此是会话线程上的一等结构，而非旁路通道。

6 · detached agent + survive-reattach

每个智能体由独立 detached supervisor 托管。worker daemon 重启或控制信道断连，supervisor 不退出；重连后重新附着到既有执行（survive-reattach）。控制信道为有序可重放日志：worker 主动消费并 ack offset（environment.Worker.lastAckedOffset），配合执行层幂等键实现双层去重；SSE 优先推送、轮询回退保证最终送达。
