v2.8 → v2.10.0 主要演进 — Plan DAG 编排 + 显式 org 路由 + 消息线程 + 看板/任务模型 + 三栏式 App Shell

Plan 编排（v2.9）：把工作组织成一张任务 DAG（一个 Plan）。在 draft 态拉入 backlog 任务为节点、连依赖（拒绝环 / 自环）；start 后由 plan orchestrator 投影器（internal/projectmanager/service/plan_orchestrator_projector.go）自动推进——每个节点状态是派生（derived, never stored）的 blocked / ready / dispatched / running / paused / done / failed（plan_view.go 的 NodeStatus）；上游任务一旦完成，ready 节点即被派发（work-delivery，非 chat @）给其指派智能体，全程无需手动 step。failed 节点阻塞其子树并唤醒 plan 创建者；rerun_failed_node / resume_paused_node 提供失败/暂停节点的恢复。Plan/Project 生命周期为 draft / running / done + 不可逆 archive（只读级联）。

显式 org 路由（v2.9）：所有 org-scoped API 迁到 /api/orgs/{slug}/...（path-explicit），消除从 session / ?org_slug= 推断的隐式「current org」；同一路径不再因隐藏状态返回不同数据，跨 org 访问以 404 拒绝（存在性不泄露）。前端在 /organizations/{slug}/... 下寻址。

消息线程 Message Threads（v2.9.1）：从任意消息派生一条线程，弹出式侧栏中扁平回复（Slack 风格单层，回复的 parent 永远是 root）；线程内 @mention 唤醒项目智能体且其回复落在线程内；Participants 侧栏列出会话内所有线程并标「新活动」未读标记。落库为 parent_message_id + 派生 root，服务端强制单层、拒绝跨会话 parent。

任务模型 / 看板（v2.9.1）：claimable ≠ assigned（指派 ≠ 认领）；每个项目一个常驻内建指派池（built-in assignment pool）——可认领任务的扁平列表。Work Board 三段式：Backlog（未排程，不可认领）· 指派池（内建，可认领）· 结构化 Plans（DAG 列）。任务状态机简化（ADR-0046）：blocked 不再是状态而是 running 任务上的可恢复标注（resume/unblock/complete/discard 自动清除），verified 移除（completed 即终态）；终态 / 已归档项默认列表隐藏。

三栏式 App Shell（v2.10.0）：Web Console 重构为四区桌面外壳——图标栏（顶层模块）→ 二级导航列 col② → 内容区 col③ → 按需上下文列 col④，沿用现行 IA（Workspace / Conversations / Members / System；Overview 移除），模块嵌套路由。二级导航靠注册表契约 SECONDARY_NAV_REGISTRY（web/src/shell/secondaryNav.tsx）按模块装配，每模块加自己的 *SecondaryNav.tsx + 一行注册，AppLayout.tsx 不被触碰（规避并发改同一文件的冲突）。

附件 / linkify / System 作者（v2.10.0）：任务 / issue 附件（project-member-gated，fail-closed，非成员 403/404）；消息引用 linkify 解析 task-<id> 与 T<number> org_ref（含终态 / done 任务，T62）为可点链接；system / scheduler 消息作者渲染为 “System”（不再 “(deleted)”，T75）。详见 CHANGELOG.md [v2.10.0]/[v2.9.1]/[v2.9] 与 docs/design/v2.10.0/。

核心机制 · Plan 编排投影器 + 唤醒/派发模型 — v2.9 → v2.10.0 新增的两条主轴

① Plan = 任务 DAG，节点状态派生。 Plan 在 draft 态编辑依赖（拒绝环 / 自环），start 转 running。节点状态从不落库、永远由 plan_view.go 派生：blocked（上游未完）→ ready（上游全完，可派发）→ dispatched（已交付指派智能体）→ running（智能体执行中，可临时 paused）→ done / failed。plan orchestrator 投影器消费 outbox 事件，上游 done 即把 ready 节点派发给其指派智能体——这是 work-delivery（工作交付），不是会话 @mention。failed 节点阻塞其整棵子树并唤醒 plan 创建者。恢复：rerun_failed_node 重跑失败节点、resume_paused_node 恢复暂停节点；陈旧释放的节点自动重派（retry-capped）。

② 智能体被「唤醒」的两条且仅两条路径。（a）工作交付：任务 / plan-node 派发到智能体的单活工作项（task-WorkItem-keyed 的 agent.wake）。（b）人类 @mention：人类在 DM / channel / task / issue / plan 会话中 @ 智能体（agent.converse）。关键 loop-break：见 internal/environment/service/wake_projector.go——只有 user: 发送者才唤醒智能体；智能体自身 @ 另一智能体不会唤醒对方（否则形成自激环）。system / agent 消息永不触发会话唤醒。

③ Plan/Project 生命周期与归档。 draft / running / done + 不可逆 archive（级联子任务、只读，running 任务存在时拒绝归档）；归档项从默认列表移出、单列「Archived」分组。任务状态机（ADR-0046）五态：open / running / completed / discarded / reopened，blocked 为可恢复标注、verified 已删除。

三栏式 App Shell · 二级导航注册表契约 — v2.10.0 前端外壳 web/src/shell/

Web Console 重构为四区桌面外壳：图标栏（顶层模块 Workspace / Conversations / Members / System）→ 二级导航列 col② → 内容区 col③ → 按需上下文列 col④，沿用现行 IA（Overview 移除），模块嵌套路由。

shell 与模块的职责切分（见 web/src/shell/secondaryNav.tsx）：AppLayout 拥有 col② 的 chrome（页眉、页脚 live / Light·Dark / Sign out、折叠、col④ 宿主）；col② 的 nav body 由各模块自有。模块任务契约只有两步：(1) 建 web/src/shell/nav/<Module>SecondaryNav.tsx（实现 ModuleSecondaryNav，接收 { orgBase }）；(2) 在 SECONDARY_NAV_REGISTRY 加一行 module: XxxSecondaryNav。这是唯一的共享改动（每模块一条不同的行 → git 自动合并），AppLayout.tsx 始终不被触碰——避免六个并发模块任务同改一文件（shared-state single-actor 命门）。无注册项的模块回落到 shell 默认 NavGroup。

每模块在外壳上重建：Conversations（col② channel / DM 导航，col④ 参与者 + 共享文件）、Workspace Projects + 项目详情（col② 项目子导航，列表卡显示 task / issue / plan / repo 计数）、Workspace Tasks / Issues（col④ 只读元数据面板）、全局跨项目 Plan 列表 + 详情（Chat / DAG / Task-list 三 tab）、Project Work Board、Members（col② Humans / Agents，Agent col④ 上下文面板）、System。

🆕 创建并派发任务 项目管理 → 智能体 → 会话

用户（控制台/CLI/MCP）→create_task→Task→pm.task.created
策略：派发→assign_task→Task · AgentWorkItem→pm.task.assignedagent.work_item_transitioned

🧭 Plan 自动推进 项目管理（orchestrator 投影器）→ 智能体（work-delivery）

用户 / PM 型 Agent→create_plan · add_task_to_plan · add_plan_dependency→Plan（draft · DAG）→pm.plan.updated
用户 / PM 型 Agent→start_plan→Plan（draft→running）→pm.plan.started
策略：orchestrator 投影器（上游 done → ready 节点派发）→PlanNode（ready→dispatched）· AgentWorkItem→pm.task.assigned · agent.work_item_transitioned
策略：节点失败阻塞子树→PlanNode（→failed）→唤醒 plan 创建者→rerun_failed_node / resume_paused_node
约束：节点状态（blocked/ready/dispatched/running/paused/done/failed）派生、从不落库；派发为 work-delivery（非 chat @）；依赖只在 draft 编辑、拒绝环/自环。

✅ 完成与领取 智能体 → 项目管理（v2.9.1 任务模型）

智能体→complete_task→Task（running→completed · 终态）→pm.task.state_changed
策略：内建指派池（claimable）→智能体→start_work（认领）→Task · AgentWorkItem→pm.task.state_changed
约束：指派 ≠ 认领（claimable = open ∧ assigned ∧ 处于已派发的 started plan 节点）；五态机 open/running/completed/discarded/reopened，blocked 为可恢复标注、verified 已删除；终态默认列表隐藏。

🧵 消息线程 + @mention 唤醒 会话 → 环境（WakeProjector）→ 智能体

用户（human）→post_message（parent_message_id + @agent）→Message（线程内）→conversation.message_added
策略：WakeProjector · 仅 user: 发送者→agent.wake / agent.converse→AgentWorkItem / running session→智能体回复落在同一线程
约束（loop-break）：只有人类 user: 发送者唤醒智能体；智能体自身 @ 另一智能体不唤醒对方；线程单层（reply 的 parent 恒为 root），拒绝跨会话 parent。

📁 文件上传与引用 环境 → 会话（v2.10.0 task/issue 附件）

智能体 / 用户→upload_file(scope=task) · attach_file→FileTransferSession + BlobStore→session open → completed
策略：project-member-gated（fail-closed）→download_file→FileReference→非成员 403 / 404
一 blob 多 scope 引用；ref_count=0 + 宽限期后 GC 回收。任务 / issue 详情面板列出 / 上传 / 下载附件，非项目成员一律 fail-closed。

💓 Worker 心跳与 Agent 可用性 环境 → 智能体

worker 守护进程→heartbeat / ack offset→Worker→workforce.worker.online
心跳超时→Worker→workforce.worker.offline→级联→agent → unavailable
Agent.availability 为只读派生（不落库），Worker 状态优先。supervisor detached，断连可 survive-reattach。
