## Binding principle — 验收独立于开发实现，验收忠于产品设计

Per @oopslink directive 2026-06-05 + `acceptance-plan-independence` memory:

**Acceptance source of truth = product design** (PR #168 §5 design proposal + the D1-D4 / R1-R3 / Q1 / `#`/`@` 4-layer lock + this plan's §0 anchors). **NOT** Dev implementation code, NOT Dev's branch, NOT what the unit tests cover. Tester / Tester2 acceptance gates are drafted **cold from spec, with zero Dev design participation and zero Dev-branch reference**, then validated only against the product design documents. Dev implementation is the **object under test**; acceptance assumes it could be implemented incorrectly and sweeps every entity behavior the design requires.

This prevents:
- (a) **Unit-green / real-broken** — testing what we built rather than what we should have built.
- (b) **Acceptance drift** — acceptance silently following code instead of leading it.
- (c) **Dev/Tester collusion** — gates softened to make implementation pass; design intent quietly erodes.

**Operational consequence** for filling Tester / Tester2 owned sections:
- Cite **`PR #168 §5.x` / `D1-D4` / `R1-R3` / `#/@` 4-layer lock / this plan's §0** when stating a gate. Do **not** cite Dev branch code, file paths, or function names as the gate's source.
- Tester2 UI gates such as "picker has `role=combobox` + `aria-expanded` + `aria-activedescendant`" cite `§5.3` + `§5.6` (design), **not** the DOM structure Dev's component happens to emit.
- Tester data gates such as "artifact CRUD `/api/artifacts?org_slug=` with versioning" cite `§5.4` + `§3.1` of this plan (which mirrors the design), **not** the route table Dev wires.
- This applies to every owner section in every phase. The scaffold §0 anchors below are the binding source; everything else descends from them.

(Tester #258 black-box acceptance and Tester2 #259 PilotDeck source-dive already followed this discipline. Codifying it here ensures every v2.8 phase reaffirms it.)

## 2. Phase 1 — Surface-agnostic MessageList (#264, D1+D2+R3)

### 2.1 What ships
- One `<ConversationView>` + shared `<MessageList>` / `<MessageRow>` / `<Composer>` component for **4 surfaces** (channel / DM / task-thread / issue-thread).
- `<MessageBlock>` interface frozen **in first 1-2 weeks of P1** (props: payload, isCollapsed, onExpand, onSendToSide) — feeds P2 design even under sequential ordering (Dev2 prep). **Internal checkpoint deliverable** (Dev2 cross-review): the contract MUST land as a tracked artifact (TypeScript interface in a referenceable location + a short markdown spec section) before P2 can begin design — `<MessageBlock>` "frozen" means the interface signature is committed and documented, not just verbally agreed.
- Backend: unread/read-position model `(conversation_id, user_id, last_read_seq)` tuple; sidebar badge count derived from this; thread follow/unfollow UI maps to existing CLI semantics.
- task-thread/issue-thread: work-item-as-message — discussion IS a thread on the work-item message, no separate "comments" model.

### 2.2 Tester data/API gates (owner: @AgentCenterTester)

> Applies the cross-phase invariants (§1) + the real-2-org seed model from #258. Gate = all rows PASS on a real install.

1. **Unread/read-position model (R3 tuple `(conversation_id, user_id, last_read_seq)`)**
   - badge count == actual unread (messages `seq > last_read_seq`) — exact: send N, mark-read at K, assert N−K.
   - mark-read **advances** last_read_seq; **idempotent** (re-mark = no change); **monotonic** (never regresses).
   - **per-(user,channel) isolation**: user A's read position ≠ user B's; channel X ≠ channel Y; no cross-tuple bleed.
   - new message increments unread for **all members except author**.
   - **Q-T1 LOCK (human-only)**: `user` = human identities only; **agents are NOT in the badge/unread model** (agent attention = directed-wake, D3). Verify an agent member shows **no** unread badge; @mention of an agent **wakes** it (does not accumulate unread).
2. **Thread follow/unfollow API**: follow → delivered/counted; unfollow → stops; state per-(user,thread); auto-follow on participate / @mention (verify the rule).
3. **work-item-as-message (D1)**: task/issue detail thread = a thread **on the work-item message** (same conversation+message model — **NOT a separate comments table**). Verify via API the thread messages live under the work-item message id; no parallel comments store.
4. **4-surface API same-source**: the same conversation returns an **identical** message payload for channel / DM / task-thread / issue-thread → one `<MessageList>` data contract, no per-surface divergence.
5. **`<MessageBlock>` contract frozen** (props payload / isCollapsed / onExpand / onSendToSide) at the P1 1-2-week checkpoint — verify it is stable + documented (feeds P2 design under sequential ordering).
6. **D2 hard-cutover (R1 = NO feature flag)**: the 4 surfaces switch to the unified MessageList **atomically** — verify **no half-migrated state** (no point where some surfaces use new + some old; no dual-write to two models; old component fully deleted in the same merge). Rollback = **`git revert <merge sha>`** — verify the revert path restores the prior per-surface state cleanly (there is no flag to flip back).
7. **run-real / computed-truth / false-alarm**: real install + real browser; send / scroll(pagination) / mention / react / attach across **all 4 surfaces** with real messages. Badge/unread read from API+rendered truth (not assumed). Any unread-count mismatch → rule out test timing / eventual-consistency / polling **before** flagging an off-by-one (§1.1).
