# v2.7 交付规范（delivery process SOP）— v1（待 @oopslink 过 → commit docs/）

> 起因：@oopslink 定分支/流程 + 问"流程能否改进/分阶段独立 owner"（2026-05-31）。团队（PM/Dev/Dev2/IntegrationDev/Tester）协同收敛。

## 0. 核心原则
1. **独立性**：建 ≠ 验 ≠ 收。建者不验收自己的活。质量的根。**双角色不破独立**：IntegrationDev 既是唯一合并人、又兼 Phase-2 feature 作者时——他自己的 patch **同受 PM diff-review + Tester 独立集成测才合**；"唯一合并人"= 机械合并权、**非 review 豁免**，合并键不让他自批自己的功能码。
2. **gate-with-owner**：每阶段一个明确 owner + 一个明确 gate（绿了才过）。
3. **shift-left 且保独立**：场景/边界思考前移到设计期，但不牺牲独立性。
4. **rigor 配 scope**：大重构/关键路径走全套；一行小改别上全流程。
5. **runtime 是真闸**：编译绿 / 单测绿 / code-review 绿 都 ≠ 真绿；真路径 runtime（集成测 + e2e）才是验收。本会话 7 条 review/单测漏的 bug 全靠独立 runtime 逮（HOL / stale-active / frozen-empty / L2-vs-B3 / FINDING-3 / #115 queued / #115 brief）。

## 1. 分支模型（@oopslink 定）
- **`v2.7` = 开发主干（dev mainline）**：特性分支从它拉、最终 promote 回它。
- **`v2.7-integration` = 集成主干**：IntegrationDev 维护；patch 先合这里、集成测过才 promote 回 v2.7。
- **特性分支**：每功能独立、从 v2.7 拉（如 `dev2/v27-107-fleet-repoint` / `agentcenterdev/v27-108-d5-sse`）。
- **migration 编号**：唯一递增、IntegrationDev 维护；drop-mig（0047+）占最后、原子。

## 2. 角色
| 角色 | 谁 | 职责 |
|------|-----|------|
| 项目管理 PM | @AgentCenterPD | 分工/序、设计评审（cut-point + diff review）、里程碑、§-1 把关、对外汇报 |
| 模块研发 | @AgentCenterDev / @AgentCenterDev2 (+IntegrationDev 兼 Phase-2、待 @oopslink 确认) | 特性分支 dev + 单测 + 切口设计 mini-matrix；完成提 PR |
| 集成 | @AgentCenterIntegrationDev | 合 patch→集成主干（rebase+full ./...+ff/squash）、热点 merge-queue、promote→v2.7、合并交接信号(@Tester+SHA) |
| 测试/验收 | @AgentCenterTester | 集成主干**集成测试** + 开发主干 **e2e**；独立验收场景-matrix |

## 3. 交付流程（pipeline）
```
特性分支(从 v2.7 拉) 开发+单测
  → [设计评审] PM cut-point/diff review + Tester 独立验收-matrix 并排
  → 绿 → push + PR 到 v2.7-integration（标注"动了 app.go Deps/wiring"）
  → IntegrationDev rebase 到 v2.7-integration HEAD + full ./... + ff/squash 合
  → IntegrationDev @Tester 给确切 SHA + 含哪些 patch/片
  → Tester 对 v2.7-integration 跑【集成测试】（多 patch 合一起、跨片交互、§-1、disconnect-FIXED）
  → 集成测绿 → IntegrationDev promote v2.7-integration → v2.7
  → IntegrationDev @Tester 给 promote 后 v2.7 SHA
  → Tester 对 v2.7 跑【e2e】（针对性/里程碑全套）
```

## 4. 四个 gate
1. **开发(+单测)** — Dev/Dev2：切口交「设计完整性 mini-matrix」(状态×路径×期望) build 前 fold；gate=自测绿 + 真码自查（无 dead code / 未接 wiring）。
2. **集成** — IntegrationDev：合前 fetch+rebase 到 v2.7-integration HEAD + full ./...（无半合、ff/squash）；**共享热点（NewRelay 注册 / app.go Deps / migration 号）合并时 rebase 解、merge-queue 串行**（append 为主、研发期不互锁、合并期 IntegrationDev 统一解）；gate=full suite 绿 + wiring 顺序对。
3. **集成测试** — Tester（独立、对 v2.7-integration）：多 patch 合并真路径 + 跨片 + §-1 + disconnect-FIXED；**promote 前必绿**。
4. **验收 e2e** — Tester（独立、对 v2.7）：产品端到端 / GATE 全套 / 真实用户流。

## 5. shift-left 核心机制 = 双独立 matrix 设计期并排对照
- **Dev 切口-matrix**：dev 设计完整性自检，build 前 fold 边界。
- **Tester 验收场景-matrix**：独立设计（spec/intent + PM），**不以 dev 的为输入基线**（否则继承盲点、独立性空）。
- 设计评审时并排 → 暴露"dev 假设 vs 独立枚举"差（本会话 re-emit-active-only / brief="" 那类漏在设计期现形，而非 commit+runtime 后）。
- 独立性红线：Tester matrix 不继承 dev 盲点 = shift-left 不反噬独立性。

## 6. 测试粒度（PM 定）
- **集成测试 = 按"集成增量"**（一个完整功能片：Phase-2 repoint 片 / D5 / E1；微改攒到片边界）。promote 前必绿。
- **e2e = 里程碑**（Phase-1 完✓ / Phase-2 全 repoint+drop 完 / D5 完 / E1 完 / tag 前 = 全套 GATE）+ 每 promote 跑针对性 e2e（动到的链）。

## 7. 当前特性分工
- @AgentCenterDev = **D5**（#108 SSE 流式控制下推）。
- @AgentCenterDev2 = **fleet repoint 收尾 → E1**（#105 UI）。
- @AgentCenterIntegrationDev = **集成 + Phase-2-bulk**（stats→projections→issue-API/discussion→taskruntime 执行模型 carve-out→drop-mig）【兼研发待 @oopslink 确认；纯集成则 Phase-2-bulk 归 Dev(D5后)】。
- @AgentCenterTester = 集成测 + e2e（双线 D5 + Phase-2）。

## 8. 集成 owner 升级触发（rigor 配 scope）
现 3 流（D5/E1/Phase-2-bulk）已配专职集成（IntegrationDev）。再涨（流 ≥4 / 热点频繁撞 / 集成 bug 信号：逐片绿合起来破）→ PM 评估加人或分集成域。

## 9. 反模式（避免）
- **纯瀑布**：4 gate 顺序硬串=慢+晚逮。靠双 matrix 设计期对照保并发思考。
- **信注释/单测当真绿**：runtime（集成测+e2e）才是闸。
- **过度流程**：小改别上全 4 gate。
- **自推开发主干**：研发不自 push v2.7；一切经 PR→集成主干→集成测→promote。

## v2.7.1 release-engineering 扩展（2026-06-04）

v2.7.1 周期沉淀的发布工程规范，**部分超越/补足 §1–§9**。下列规则适用于 v2.7.1 之后的所有 release/patch 周期。来源：`docs/rules/v271-retrospective.md` §9 + IntegrationDev/PD/Tester/Tester2/Dev2 复盘 + PR #145 整合。

### 10. 分支模型（v2.7.1 起，单分支 + 每 PR ff sync）

**supersedes §1 关于 v2.7-integration 中间分支的模型，仅对 v2.7.1+ 周期适用。** §1 内容作为 v2.7 周期历史保留。

- **`v2.7.x` = release 分支**（单分支，无 `v2.7.x-integration` 中间分支）。
- **每 PR 流程**：feature branch → PR 到 `v2.7.x` → IntegrationDev `merge --no-ff` → **立即 `merge --ff-only` 同步到 `main`**（per-PR sync，不留 main 滞后窗口）。
- **trunk 状态恒定**：`v2.7.x` = `main`（每 PR 合后两条都更新到同一 commit），未来 patch 或 v2.8 起点都清晰。
- **rationale**：v2.7 试过 `v2.7-integration` 中间分支，复杂度高 + 易漂移；v2.7.1 单分支模型干净、可预测；ceremony 时一次性大 ff 改成 per-PR 小 ff，易定位错误。

### 11. Tag 操作 — 显式 refspec 强制

**branch + tag 同名时（如 `v2.7.1` 是分支也是 tag），所有 git 操作必须用显式 refspec**。裸 `git push origin v2.7.1` 会撞 "matches more than one" 被 git 拦下——这是事实，第一次 retag 时被拦住，PR #145 ff sync 时再次被拦。

```bash
# Tag publish
git push origin refs/tags/v2.7.1

# Tag delete (destructive — 见 §12)
git push origin :refs/tags/v2.7.1
git tag -d v2.7.1                # 本地清理

# Branch push (when name collides with tag)
git push origin refs/heads/v2.7.1:refs/heads/v2.7.1

# Verify
git ls-remote --tags origin v2.7.1
git ls-remote --heads origin v2.7.1
```

**禁用**：裸 `git push origin <name>` / `git push origin :<name>` / `git push --delete origin <name>` 在 branch+tag 同名期间——必须 `refs/tags/` 或 `refs/heads/` 显式。

### 12. Destructive 操作授权协议

下列操作 **必须** owner（@oopslink 或当时的 release owner）**显式书面授权**才执行——不在 implicit context、之前同情境授权、PD/team 转述上自决：

- `git push origin :refs/tags/<tag>` （删已 push 的 tag）
- `git push --force origin refs/heads/<branch>` （force-push 远端分支）
- `git reset --hard origin/<branch>` 后再推（移动远端分支头）
- `git tag -f <tag> <new-commit>` 后再推（force-move tag）

**鉴别什么算"明示授权"**：
- 必须含明确动作动词（"撤 tag" / "force-push" / "delete branch"）
- 必须含明确目标（哪个 tag / 哪个 branch）
- 必须**当前**对话发出（不引用之前的同型授权）

**reference 事件**：第一次 v2.7.1 retag 时 @oopslink 写明 "确认无人使用 v2.7.1 → 撤 tag + 加 T<n>/I<n>"——含动词+目标+当下决定，符合标准。

### 13. Evidence-in-tagged-tree verify（最关键的 release SOP）

**声明 "X 是 commit-aligned with `<tag>`" 前必须 `git ls-tree -r <tag> <path>` 实证文件在 tagged tree 里**。

不算 evidence：
- 工作目录的文件（`ls`/`find` 看到）
- 刚 `Write` 完没 `git add`/`git commit` 的文件
- `git stash` 里的文件
- 别人转述说 "已 commit"

算 evidence：
- `git ls-tree -r <tag> <path>` 列出该文件（blob hash 存在）
- `git show <tag>:<path>` 可读出内容
- `git log <tag> -- <path>` 显示提交历史

**ship 帖发出前 checklist**（IntegrationDev own）：
- [ ] `git ls-tree -r <tag> <evidence-dir>/` 验报告 + 截图都在
- [ ] `git log <previous-tag>..<tag> --oneline` 验提交清单与 ship 帖列表一致
- [ ] `git rev-parse <tag>^{commit}` 与各方断言的 commit hash 一致

**chain-echo ≠ verify**：多角色 ack "X commit-aligned" 时，至少 IntegrationDev 必须运行上面命令实证。v2.7.1 evidence-archive gap = 五个角色 echo 同断言但没人 verify 的典型反例。

### 14. §-1 explicit signal discipline

PR merge 必须由 **PR 自身 thread** 上的明文 §-1 "approve + merge it" 信号触发。

**不算 §-1 PR approve**：
- 契约 approve（在 task thread 而非 PR thread）
- "tag waits on X+Y+Z" 类协调 message
- Implicit "应该可以合了" 推断
- 多个 PR 同时 approve 时不区分到具体 PR#

**reference 事件**：v2.7 PR #88 漏 thread approve / PR #97 误把契约 approve 等同 PR approve → 两次同型坑；v2.7.1 显式 §-1 模式 30+ PR 零事故。

### 15. PR rebase-before-merge 模式

IntegrationDev 每个 PR merge 前 check base SHA；若 ≠ 当前 `v2.7.x` head，rebase 后再合：

```bash
# 1. 查 PR base
gh pr view <N> --json baseRefName,headRefName,headRefOid

# 2. 若 base 落后，rebase
git checkout -b rebase-<N>-prep origin/<head-branch>
git rebase origin/v2.7.x
go build ./... 2>&1 | head -5         # 验 rebase 后 build 绿
git push --force-with-lease origin rebase-<N>-prep:<head-branch>

# 3. merge --no-ff (with 描述性 message)
git checkout v2.7.x && git reset --hard origin/v2.7.x
git merge --no-ff origin/<head-branch> -m "merge(v2.7.x PR #<N>): <description>"

# 4. ff sync main
git checkout main && git reset --hard origin/main
git merge --ff-only origin/v2.7.x
git push origin refs/heads/main:refs/heads/main
```

**关键约束**：rebase 用 `--force-with-lease`（不 `--force`），避免在 PR 作者刚 push 新提交时盖掉；rebase 后 build 失败立刻停，找 PR 作者改。

### 16. Re-tag 决策树

Tag 已 ship 后需变更时，按优先级选：

1. **(a) 接受当前 tag + 后补 commit + release note 指针**（默认 / non-destructive）
   - 适用：evidence/doc/小改 missed；tag 持续 immutable
   - cost：当前 tag tree 不字面含新增内容（与 commit-aligned 严格定义有差）
   - benefit：consumer 安全；tag 公认 immutable

2. **(b) Owner 授权 destructive retag**（限场景）
   - 适用：tag 推送后立即发现严重 scope 错误 + 无 consumer 拉过 + owner 明示授权（§12）
   - 必要 condition：window 小 + 无 consumer + owner 明示
   - reference：v2.7.1 第一次 retag（cacfcbd → bdc9818）符合条件，@oopslink 明示授权

3. **(c) Patch tag (v2.7.x → v2.7.x+1)**（推荐用于已有 consumer 的场景）
   - 适用：tag 已被 consumer 拉过 / 改动 substantive
   - 永不破 immutability + 给 consumer 清晰升级路径

**默认 (a)**；(b) 是例外路径；(c) 是 substantive 改动的正路径。"删 tag 重发" 不是常态。

### 17. 机器占用协议（端口 / prefix 模板）

并行/共享实例验收时固化端口与 prefix 模板：

- **端口**：center=`7101`，admin=`7051`，control-stream=`7301`。**避 `:7000`**（macOS AirPlay 默认端口，与 center 撞）。
- **prefix**：`/tmp/<round-name>`（如 `/tmp/acfull` / `/tmp/mb` / `/tmp/r3`）。
- **announce** 在 `#agent-center` 主频道（非 thread），含端口 + prefix + 预计时长。Tester2/Dev/Dev2 看见后避免端口冲突。
- **release** 完成在主频道公开 "机器释放"。

### 18. 单角色单域 owner（retag 期）

retag 期所有 ship-tail PR 必须 **单角色单域 owner**：每个 PR 由一人负责，不混 owner。理由：retag 期容错小、协调开销大。`merge 顺序` 应线性，最后合的 PR（如 README）能引用前面所有 commit 的 final 状态。

**适用范围**：小团队 + 高信任 + retag 阶段。常规 dev 期不强制（PR 间冲突低）。

### 19. Docs/非冲突 follow-up PR 用独立 git worktree（不共用 checkout）

多 agent 共享同一个 working copy 时，shell `cwd-reset` + 跨分支 `git checkout` 会让一个 agent 的 commit 落到别人当前 checkout 的分支上，连带 auto-push 把错误 commit 带去 origin。force-push 抢救又破 §12。**机制级隔离**比"靠自觉切 branch" 更稳。

**规则**：所有 docs/非冲突的 follow-up PR 在独立 worktree 里跑：

```bash
# 起 follow-up（新分支）
git worktree add /tmp/wt-<task> -b <branch-name> origin/main
cd /tmp/wt-<task>
# ... edit / commit / push ...
git push -u origin <branch-name>

# rebase 别人的 PR
git worktree add /tmp/wt-pr<N> origin/<head-branch>
cd /tmp/wt-pr<N>
git rebase origin/main
git push --force-with-lease origin HEAD:<head-branch>
cd <main-checkout>
git worktree remove /tmp/wt-pr<N>
```

**禁止**：在共享 checkout 上 `git checkout <other-branch>` + edit + commit。每次 cwd-reset 后都 verify 当前 branch（`git branch --show-current`）也不够保险，shell 状态可能随时变。

**reference 事件**：v2.7.1 retro follow-up 期 PR #146 case — Tester2 的 ux-standards commit (`cfd26f0`) 在共享 checkout 上被 IntegrationDev 的 delivery-process commit (`bf7755f`) 间接挤掉（IntegrationDev `git checkout -b` 后 cwd-reset 落回 tester2 branch，commit 落错位置，auto-push 带去 origin）。Tester2 没 force-push 抢救（守 §12），改用独立 worktree 重建为 PR #151。Dev 后续也对等自报多次共享 checkout 切 HEAD 是同型风险贡献源。

**Accidental commit on wrong branch — non-destructive recovery**（与 §11/§12 协同）：若已经在错的 branch 上 commit 并 auto-pushed，**不要** force-push 抹除。改为：
1. cherry-pick 该 commit 到正确分支
2. 在错误分支上 `git revert <commit>`（产生抵消 commit）
3. 推送 revert commit（净 diff = 0，history 可见）
4. 在 thread 透明 surface 给对应 owner

reference：v2.7.1 retro follow-up `bf7755f`（IntegrationDev 错误 commit）→ `87915a0`（revert on tester2 branch）→ 净 diff = 0、history 留 transparency 记录。
