## § -1. Ship-blocking 验收协议（最高优先级 / 不可跳）

> **教训（v2.6）**：v2.6 有验收计划、也"验收"了，但 build 仍挂（AppLayout build-break → v2.6.1 补）。根因：① 信 dev 的 "gates green" 报告、没亲自在干净环境复跑；② 没跑**生产构建**（dev 模式能过、`tsc -b`/`vite build` 会挂）；③ 没在 clean checkout 跑（增量缓存掩盖）；④ 没跑**真实发布产物**端到端。
>
> **v2.7 铁律**：以下每条由 **@AgentCenterPD 亲自执行**，**不信报告**，**贴出证据**，任一不过 = ship-block，**没有这些就没有 "ship-ready" 信号**：

1. **干净环境**：fresh clone / clean build（`git clean -xfd` 或全新 checkout），杜绝增量缓存掩盖。
2. **我亲自跑全门禁并贴输出**：`go build ./...`、`go test ./internal/... ./tests/integration/...`、`go test ./tests/e2e/... -count=1`、`make lint`、**`tsc -b --force`（生产类型检查）**、**前端生产构建 `vite build`（不是 dev server）**、`vitest`。
3. **跑真实发布产物**：`make release` → 全新 install → `start` → 在 mac 上走 §2 端到端主流程（不是 dev server / 不是源码态）。
4. **证据**：贴门禁命令输出 + 端到端走查证据（关键步骤命令输出/截图）。
5. ship-ready 信号 = 上述全部亲手通过 + 证据齐 → 才发；**绝不基于"应该没问题"**。

## § 1. 分域验收

### A0 — Conversation / File URI 基础
- [ ] Conversation.kind = `channel | issue | task | dm`（无 project_channel）；channel owner_ref=`id://organizations/{id}`、issue/task=`pm://...`、dm 无。
- [ ] Message.context_refs 带 work_item_ref/task_ref/agent_ref；UI 可按 work_item_ref 分段、不破。
- [ ] 统一 MessageAttachment（uri ac://files/{ulid}+filename+mime+size，图/文件不分）。
- [ ] FileURI 值对象 + resolver（hash(ulid) 分桶）；FileReference({scope,scope_id}→uri)；BlobStore 接缝（ULID 身份 + content sha256 元数据）。
- [ ] outbox 表 + 幂等投影器骨架（dedup by event_id）。

### B — ProjectManager 域
- [ ] 无全局/跨 Project Task·Issue（拒空 ProjectID）。
- [ ] Issue/Task 状态机强制（§2.2）；blocked 必带 reason；completed 由 assignee/agent；**verified 非本任务完成者（含另一 Agent，ErrSelfVerify）**。
- [ ] 嵌套端点 `/api/projects/{id}/{issues,tasks(+子动作),members,code-repos}`，requireProjectMember 门控；跨 Project→403/404。
- [ ] CreateTask/AssignTask 经 **outbox projector** 建会话 + 同步 participant（不内联）；reassign→旧 WorkItem superseded+新建。
- [ ] **OQ13**：unassign/reassign/reopen 出局 assignee **保留为 sticky 订阅者**，仅显式 Unsubscribe 移除；creator 不可退。
- [ ] CreateProject **不**自动建 channel；channel org 级、显式成员、可选 project_ref 软标签（无约束）。
- [ ] 前端：Issues/Tasks 内嵌 ProjectDetail tab，无全局页、无跨项目聚合。

### C — Agent BC
- [ ] AgentRun 不存在于 API；Agent 长驻模型。
- [ ] CreateAgent 必选 Worker、worker_id 不可变；Start/Stop/Restart/Reset(scope+二次确认)。
- [ ] **StopAgent 不置 WorkItem blocked**；**OQ11**：AgentWorkItem 无 blocked 态（Task blocked→取消 WorkItem；unblock→新建）。
- [ ] availability 派生 `available|busy|unavailable`，Worker 离线优先级最高。
- [ ] AgentWorkItem 多 AgentInteraction（wait→wake）；AgentActivityEvent 流；消息按 work_item_ref 分段。
- [ ] 前端：Agent list/建(选 Worker)/生命周期控制/WorkItem 队列/活动流/availability 徽章。

### D — Environment 域
- [ ] **D1**：一 Worker 控多 Agent（per-worker 独立 offset）；**重连按 offset 续传不重复破坏命令**（日志层：已 ack 不重投 + 命令带 idempotency_key）；Worker 掉线→Agent unavailable；worker-initiated + 复用 TLS/auth。
- [ ] **D2**：MCPHost 暴露 OQ4 工具集、**per-agent 域隔离**（own Project/Org、读限 own WorkItems、no self-verify、管理面 human-only）；两通道（stdin/stdout 驱动→AgentActivityEvent，**stdout 不进 Conversation** / MCP 反向操作）；显式人可见通信（post_task_message 唯一文本入会话路径；request_input/block_task/complete_task 携带文本）；唤醒闭环（OQ5：系统不判语义、Agent 自决、人+他 Agent 输入、忙缓存、同到合并）；**执行 cutover**（旧 taskruntime→新 AgentController）。
- [ ] **D5(#108)**：SSE stream-first 推送（同命令日志、断线回落 poll、保证同 poll 路径）。
- [ ] **D3**：FileTransferSession 上传/下载（create 返回 ac://files/{ulid}+transfer_uri、完成后可读）+ 可达性授权 + 引用计数 GC（7 天宽限，含孤儿）。
- [ ] **D2-g**：ClaudeCode 契约测试（stub claude）：start→注入→输出→request_input→调 MCP→stop/restart。
- [ ] **D2-g MCP 工具契约（per-tool 端到端，必须真实 admin 非 fake）**：16 个 OQ4/文件工具逐个 MCP call→真实 admin 端点→断言成功/正确副作用。**教训**：recording-fake 单测只证"转发了选定字段名"、证不了"字段名匹配 admin 契约"（b3 的 post_task_message `text`vs`content` bug 就是这盲区）——必须端到端机械验。

### E — 整合 / UI / Release

- [ ] **E1 补**：订阅 subscribe/unsubscribe 前端 UI（D2 起仅后端端点）；项目成员管理 UI；**derive-from-message（OQ12）**（多选消息→建 Task/Issue→forwarded 卡片入会话，零拷贝附件）。

## § 2. 跨 BC 端到端场景（mac 实操走查）

- [ ] **主流程**：建 Project → 建 Task → 指派 Agent → Agent 要输入(request_input) → 人回复 → Agent 续(新 AgentInteraction) → complete → creator/另一 Agent verify。
- [ ] **停 Agent**：active 工作中 StopAgent → 不误置 blocked，工作显示 paused（availability unavailable）。
- [ ] **重派分段**：Task 重派 A→B → 同一 Task 会话内按 work_item_ref 出现新段；A 仍在会话（sticky 订阅，OQ13）。
- [ ] **文件**：上传附件 → 拿 ac://files URI → Agent 经 URI 引用；删消息/任务 → 软删引用、blob 不立删、过宽限期 GC。
- [ ] **唤醒**：waiting_input 时人回复唤醒；忙时缓存；同到合并；他 Agent（编排者）也能回。
- [ ] **域隔离**：Agent 跨 Project/Org 操作被拒；Agent 不能验自己完成的 Task；管理面（建 Project/Member/Agent/Worker）Agent 不可达。
- [ ] **重连不重复破坏**：worker 断线重连，stop/reset 不二次执行（日志层 + worker 处理器 idempotency_key 双层）。

## § 4b. 已知 v1 限制（接受 + hardening follow-up）

- **MCPHost worker-token 注入**：agent 的 MCP server 子进程经 env 拿 **worker bearer**（权力大于单 agent，可操作该 worker 上任意 agent）；claude 模型运行时可见该 token → 被污染的 agent 理论上能冒充同 worker 上其它 agent。**v1 接受**（单用户、agents 均属用户、同信任域）。**Hardening（多用户/不可信 agent 前必做）**：注入 **per-agent 限域 token** 而非 worker bearer。决定 2026-05-30（@oopslink #1=B 后）。
