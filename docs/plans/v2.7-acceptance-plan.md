## § -1. Ship-blocking 验收协议（最高优先级 / 不可跳）

> **教训（v2.6）**：v2.6 有验收计划、也"验收"了，但 build 仍挂（AppLayout build-break → v2.6.1 补）。根因：① 信 dev 的 "gates green" 报告、没亲自在干净环境复跑；② 没跑**生产构建**（dev 模式能过、`tsc -b`/`vite build` 会挂）；③ 没在 clean checkout 跑（增量缓存掩盖）；④ 没跑**真实发布产物**端到端。
>
> **v2.7 铁律**：以下每条由 **@AgentCenterTester（验收 owner）亲自执行**，**不信报告**，**贴出证据**，任一不过 = ship-block，**没有这些就没有 "ship-ready" 信号**（纪律本身不变，仅执行者由 PM 改为验收 owner；@AgentCenterPD 保留设计复审 + 口径仲裁）：

1. **干净环境**：fresh clone / clean build（`git clean -xfd` 或全新 checkout），杜绝增量缓存掩盖。
2. **验收 owner 亲自跑全门禁并贴输出**：`go build ./...`、`go test ./internal/... ./tests/integration/...`、`go test ./tests/e2e/... -count=1`、`make lint`、**`tsc -b --force`（生产类型检查）**、**前端生产构建 `vite build`（不是 dev server）**、`vitest`。
3. **跑真实发布产物**：`make release` → 全新 install → `start` → 在 mac 上走 §2 端到端主流程（不是 dev server / 不是源码态）。
4. **证据**：贴门禁命令输出 + 端到端走查证据（关键步骤命令输出/截图）。
5. ship-ready 信号 = 上述全部亲手通过 + 证据齐 → 才发；**绝不基于"应该没问题"**。

## § 2. 跨 BC 端到端场景（mac 实操走查）

- [ ] **主流程**：建 Project → 建 Task → 指派 Agent → Agent 要输入(request_input) → 人回复 → Agent 续(新 AgentInteraction) → complete → creator/另一 Agent verify。
- [ ] **停 Agent**：active 工作中 StopAgent → 不误置 blocked，工作显示 paused（availability unavailable）。
- [ ] **重派分段**：Task 重派 A→B → 同一 Task 会话内按 work_item_ref 出现新段；A 仍在会话（sticky 订阅，OQ13）。
- [ ] **文件**：上传附件 → 拿 ac://files URI → Agent 经 URI 引用；删消息/任务 → 软删引用、blob 不立删、过宽限期 GC。
- [ ] **唤醒**：waiting_input 时人回复唤醒；忙时缓存；同到合并；他 Agent（编排者）也能回。
- [ ] **域隔离**：Agent 跨 Project/Org 操作被拒；Agent 不能验自己完成的 Task；管理面（建 Project/Member/Agent/Worker）Agent 不可达。
- [ ] **重连不重复破坏**：worker 断线重连，stop/reset 不二次执行（日志层 + worker 处理器 idempotency_key 双层）。

### § A.0 验收前置（每次执行都先固定并贴出）

- **真 claude 版本固定**：`claude --version` 必须贴出，基线 **claude 2.1.156**；版本漂移即重验（CLI 漂移是本 gate 存在的根因）。
- **环境**：测试机 oopslink@home（darwin arm64）；fresh install 空库；FLAG validation 模式（cutover 翻开关**前**的 additive 验证，不碰现网执行路径）。
- **argv 回归断言（GATE-5，每次跑前 assert）**：实际拉起的 `claude` argv =
  `claude --print --input-format stream-json --output-format stream-json --verbose --session-id <UUIDv5(agentID)> [--mcp-config <path>]`（dev 确认 2026-05-30；已过真 claude 2.1.156）
  - ✅ 必含 `--print`（D2-c-ii-C 抓到的回归：曾被错删 → 进程行为变化）。
  - ✅ 必含 `--input-format stream-json`（长驻注入通道①）。
  - ✅ **`--session-id` 必须是合法 UUID** —— claude 2.1.156 对非 UUID（如 raw ULID）直接报 `Invalid session ID. Must be a valid UUID.`，进程起不来 = 静默失效（真 claude gate 抓到）。**行为断言**：① 组装出的 `--session-id` 匹配 UUID 格式（正则 `^[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}$`）；② **同一 agent 跨重启得到稳定一致的 session-id**（会话连续性所需）。（实现参考：launcher 从 agent ULID 派生确定性 UUID，dev fix `2e04973`——仅交叉核对，非验收标准。）
  - **身份分离**：session 身份（派生 UUID）≠ 授权身份（真 agent ULID，经 `AC_MCP_AGENT_ID` 给 MCP 做域隔离授权）；二者解耦，e2e 断言时勿混。
  - **这是真 claude gate 抓到的第 3 个静默失效**（前两个：argv 删 `--print`；解析器对不上嵌套格式）。三者均 stub/单测看不见、只有真 claude 能抓 → 本门是 **GATE-0「会话能真启动」**（见 §A.5）。
  - ⚠️ **e2e 并发注意**：claude 拒绝**已在用**的 session-id。生产一 agent 一进程无碍；但并发 e2e/重跑须用**唯一/全新** session-id（或确保前一进程已退），否则撞 id 起不来。
  - ✅ 用 stdin 注入 user message，**无位置参数 prompt**、**无 `-p`**（旧 adapter 用 `-p` 是一次性模式，cutover 后是长驻）。
- **stdin 注入格式（dev 确认，已过真 claude）**：每行一条
  `{"type":"user","message":{"role":"user","content":[{"type":"text","text":"..."}]}}`
- 旧 `internal/agentadapter/claudecode/adapter.go` 是 taskruntime 期一次性 adapter（扁平 stream-json 解析、`-p` 一次性），**不是** cutover 后的 AgentController 路径；本 gate 验的是新 D2 路径（`internal/workerdaemon/claude_stream.go`），二者不可混用。

### § A.1 Round-trip 证据格式（强制 / 每个 D2-f/g 用例都产出）

> 目的：杜绝"门禁绿/stub 通过"的假性证据（v2.6 + D2-c-ii-C 两次教训）。**没有完整证据包 = 该用例视为未过**。

每个用例产出一个证据包目录 `docs/plans/reports/v2.7/d2-<case-id>/`，固定包含：

| 文件 | 内容 | 作用 |
|---|---|---|
| `meta.txt` | `claude --version`、执行时间、case-id、**完整 argv**、`mcp-config` 内容（token 脱敏为 `***`） | 证明跑的是真 claude + 正确 argv |
| `stdin.jsonl` | 实际注入 claude 的 stream-json **逐行**（任务简报 / 人回复 user message） | 通道①输入真凭 |
| `stdout.jsonl` | claude **真实** stdout **逐行原始**（未加工，含 system/assistant/user/result 嵌套包络） | 通道②输入真凭 |
| `parsed.jsonl` | 解析器产出的 AgentActivityEvent **逐行**；每行标注源 stdout 行号 | 解析正确性可逐行对账 |
| `assertions.md` | 期望 vs 实际逐条勾（PASS/FAIL）+ **Unknown 事件计数** | 判定依据 |
| `sideeffect.md` | （MCP 用例）AppService 端副作用前后对比：调用前后的 `GET` 响应 / DB 行 diff | 证"真打到 admin 且字段契约对"（非"转发了字段名"） |

### § A.2 通道① 注入用例（stdin → claude 收到）

> **OQ5 唤醒验收口径（PD 设计意图 2026-05-30，行为级 / 独立于实现）**：验收只断言**对外可观测行为**，不绑定游标/ULID/事件等内部实现（实现怎么保证是 dev 单测的事）。真相来源 = **持久化消息 + 该 agent 的读游标**；wake 信号只是降延迟优化，**不是真相来源**。
> - **唤醒面（先钉死）**：唤醒/注入只发生在 **agent 当前活跃 WorkItem 绑定的 task-owned 会话**内——**不是** agent 订阅的任意频道。两平面分清：**任务驱动平面**（唤醒注入 claude 会话）只在 task-owned 会话触发；**一般消息平面**（@提及/群聊）走 Slock 常规投递、**不**注入任务 claude 会话（与 e-i "只有 task-owned 会话发 message_added" 同源）。
> - **够格唤醒谓词（在唤醒面*内*筛 sender）**：该 task-owned 会话内一条消息够格唤醒 agent ⟺ `sender 是该会话参与者/订阅者（sticky 或 manual，OQ13）` **且** `sender ≠ agent 自己`。无关频道来的消息（即便 agent 订阅该频道、sender 够格）→ **不**唤醒任务会话。
> - **不丢（红线）**：投递派生自"持久化消息 + 读游标"。**丢弃全部 wake 信号后，agent 下次进入 `waiting_input` / 下次 poll 仍能从游标重算出全部未读够格消息**——不丢的保证不依赖任何 wake 送达。
> - **不重（严格定义）**：仅「**读游标已越过某消息、却再次投递**」算重复 = fail。游标推进（mark-seen）**之前**的 re-wake 重投"尚未 seen"的消息 = **设计内幂等，不算 bug**。
> - **合并 = wake-fire 时刻快照（非定时窗）**：触发那一刻所有未读够格消息进**同一批**；触发后才到的进**下一轮**。按"wake-fire 时是否已在未读集"断言，**不按固定时间窗**。
> - **交付 = 游标批量**：投"自该 agent 游标起的全部未读够格消息"，**按时间序合并、sender 轻标注**（`[user:alice]`/`[agent:bob]`，纯文本非 JSON），批内保持时间序。
> - **触发点 = `active → waiting_input` 跃迁**：非 waiting_input（working/stopped/unavailable）期间消息缓冲；**跃迁进 waiting_input** 才投；stopped/unavailable 期间不投，回到 waiting_input 由游标补齐（与"不丢"同源）。
>
> 实现参考（仅交叉核对，**非验收标准**）：D2-e 已落地（dev `cc53e43`）——`request_input` 同 tx `agent.awaiting_input`、`POST /admin/environment/agent/mark-seen`、read_state 支持 `agent:{id}`。**PD 拍 甲(push 优先 + poll 兜底，同一真相=游标) + (B) 中心 `WakeReconcileLoop`(60s 慢拍，仿 Pump/GCLoop) 2026-05-30**：D2-e-iii 从中心扫 `waiting_input` WorkItem、`flushUnread` 从游标重算、与 push **同一 cursor-gated flush + 同一 batch key** → push/poll 永不双投（C1-nodup(c)）。兜底由中心驱动、**不与 daemon/controller 同生死**（C1-noloss 失效面含 daemon 卡死）。这是 C1-noloss 红线可过的前提。次序：D2-e-iii → D2-f。

### § A.6 域隔离 / 幂等 / fail-closed 负向用例（OQ4/OQ6 边界，并入 §A.4 各行执行）

- [ ] **域隔离**：任一工具操作非本 Project/Org 的对象 → 拒（跨 Project read/write/assign/file 全覆盖）。
- [ ] **管理面 human-only**：建 Project/Member/Agent/Worker 对 Agent 不可达（无对应 MCP 工具 / 调用被拒）。
- [ ] **no-self-verify**：`verify_task` 自己完成的 Task → ErrSelfVerify。
- [ ] **读域**：Agent 读范围严格限 own WorkItems（`get_task`/`get_my_work` 越界 → 拒）。
- [ ] **幂等**：命令带 `idempotency_key`，重投/重连 → 副作用恰一次（与 §1-D D1「重连不重复破坏」合验）。

- [ ] **stdout 不入会话**：claude 啰嗦中间输出只沉 ActivityEvent；断言 Conversation 仅含经 `post_task_message`/`request_input`/`block_task`/`complete_task` 显式携带的文本。
