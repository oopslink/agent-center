**Known limitations** (filed below in [[v2.1-D: SSE per-conv subscribe]]):
- Real-time badge tick on `/channels` requires the SPA to call
  `/api/sse/subscribe` per visible conversation — not done in v2.1-C.
  Affects `conversation.message_added` SSE invalidation for the
  unread query (same gap as existing real-time message updates on
  ChannelDetail). Cold open + visit + return all work correctly via
  initial fetch + mutation invalidation; only the "background tab
  auto-tick" path is gated by this.
- `countUnread` SQL uses the existing `idx_messages_conv
  (conversation_id, posted_at)`, which seeks by conversation_id but
  scans the conversation linearly to evaluate `id > ?` as a residual
  predicate. With LIMIT 1000 cap, worst-case cost is O(min(conv_size,
  1000)) row fetches per query. Empirically 2-3ms for 500 messages
  on local sqlite. Becomes worth adding `(conversation_id, id)` as a
  covering index when channels routinely exceed 10K messages.

## v2.1-D: SSE per-conversation subscribe wiring

**Story**: SPA's `useSSE` hook opens the EventSource long connection
but never calls `POST /api/sse/subscribe` for any conversation. The
backend `Bus.Publish` filters events with a non-empty
`conversation_id` against `b.channels[userID]` (per-conv subscription
set); without subscribe calls, the set is empty and these events
never reach the frontend.

**Affected real-time updates**:
- `conversation.message_added` — live message push on ChannelDetail
  / DMDetail (existing pre-v2.1-C gap).
- `conversation.read_state.changed` — badge auto-tick on /channels
  / /dms list when another tab marks-seen (v2.1-C, surfaced today).
- `conversation.message_references_added` — carry-over refs panel
  refresh (existing).

**Workaround**: cold open + navigation re-fetches via react-query
invalidate-on-mount. Single-user single-tab workflows look fine.

**What's needed**:
1. Per-conv subscribe / unsubscribe hook (e.g.
   `useSSEConversationSubscribe(convId)` that POSTs on mount and
   unsubscribes on unmount).
2. Channels / DMs list pages: subscribe to every visible
   conversation (so badges auto-tick).
3. ChannelDetail / DMDetail: subscribe to the focused
   conversation (so live messages flow).
4. e2e regression: existing SSE specs already cover the subscribe
   path mechanically; we'd add a SPA-driven spec that verifies the
   subscribe is fired without a manual API call.

**Estimate**: ~1.5h.

**Priority**: medium — fixes a real UX regression but only manifests
in multi-tab or "leave page open" use cases. Single-user
single-active-tab workflows function correctly without it.

## v2.1-E: messages (conversation_id, id) covering index

**Story**: `countUnread` SQL filters by `WHERE conversation_id = ?
AND id > ?`. Current index `idx_messages_conv (conversation_id,
posted_at)` only seeks by conversation_id; the `id > ?` filter is a
residual predicate evaluated row-by-row. With LIMIT 1000 cap, cost is
bounded but linear in conversation size.

**Empirical** (v2.1-C smoke): 504 messages, query in ~3ms; 50
channels × 500 messages ≈ 150ms cumulative cold load — fine for v2.1
fresh installs.

**What's needed**: migration 0027 adding `CREATE INDEX
idx_messages_conv_id ON messages (conversation_id, id)` so the
`id > ?` predicate becomes a range seek (O(unread_count) vs
O(conversation_size)).

**Estimate**: ~30min (1 migration + round-trip test + verify EXPLAIN
QUERY PLAN flips to "SEARCH messages USING INDEX idx_messages_conv_id
(conversation_id=? AND id>?)" instead of SCAN).

**Priority**: low — file for when any single channel exceeds ~10K
messages, or when /channels list cold-load latency becomes a
complaint.
