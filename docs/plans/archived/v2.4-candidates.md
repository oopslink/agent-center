## § 1 候选详情

### C1 Conversation projection 加 project_id (read model)

**简述**：给 Conversation 的 read projection（不是 AR）加 `project_id`，按 parent_conversation_id 反查 Issue/Task AR 取，存 read model。让"跨 BC 按 project 查询 conversation"成为可能 —— 比如未来「这个 project 下所有 message 全文搜」「project 工作流 timeline」。

**产品价值**：
- 谁感知：开发者、未来的 cross-project 视图、message 全文搜场景
- 现状痛点：v2.3-#30 暴露 → Conversation AR 无 project_id → 跨 project 视图当时是 cosmetic chip → #31/#32 通过 BC-native endpoint 绕开了 Issue/Task 维度，但 **Conversation 维度的跨 project 视图仍然没有**
- 用户价值：中。**不是阻塞** —— Issues/Tasks 已经能 by-project filter（#32 完成）；只有想做"按 project 查 message"才会触发
- **本次为何 v2.4 而不是更晚**：作为 read model 加，无 schema 重大变更，**乘热打铁**比未来重新唤起上下文便宜

**技术成本**：~4-6h
- BC 涉及：Conversation（read projection）+ Discussion + TaskRuntime（拉 project_id 来源）
- 方案：在 Conversation 的 list/read query 里 join Issue/Task 表取 project_id（或 message_added 写入时同时落 read model 字段）
- **不是**给 Conversation AR 加字段 —— 维持 BC 内聚，走 read model（CQRS read side）
- 风险：read model 更新点要找全（message_added、derive、edit、participant change 都不影响 project_id 但要覆盖测试）

**触发条件**：
- ✅ **已部分触发**：v2.3-#30 audit § 5 + 跨 task 主题里已被点名
- ❌ 暂未触发：用户没有强烈"按 project 查 message"诉求 —— Issues/Tasks 已能 filter；纯 conversation 搜目前不是高频
- **建议节奏**：v2.4 早期做（趁 #31/#32 BC-native 模式还热）

**关联**：
- [v2.3-summary § 3.3 BC 边界主题](v2.3-summary.md)
- [v2.3-summary § 4 来源表](v2.3-summary.md)

### C2 admin token TTL + 审计事件

**简述**：admin token 当前是 revoke-only（无过期）+ create/revoke 不发事件。补 TTL（`expires_at` 字段）+ create/revoke/use audit events。

**产品价值**：
- 谁感知：部署运维、安全 review、合规
- 现状痛点：#28 audit § 5 明确推迟 —— "v0 minimal didn't ship"
- 用户价值：**条件性 P0**。
  - 单 Mac 自用：低（loopback 信任本机用户，token 几乎不轮换）
  - 多机部署（#27 ship 后）：**必做**。TCP 暴露后 token 漏出风险显著，无 TTL = 漏一次永远在线；无 audit = 漏了不知道
- **设计早就留好接口**：Service 层 `MarkUsedAsync` pump 已经在；audit 落 events 表是接 sink 即可，没有 architecture 改动

**技术成本**：~4h
- BC 涉及：admintoken
- TTL：schema 加 `expires_at`（迁移 0029），middleware 检查 → 401；create 接受 `--ttl`；list 显示剩余时间
- 审计事件：`admin_token.created` / `admin_token.revoked` / `admin_token.used` 类型加进 observability，落 events 表
- 风险：bootstrap token 需特例（不能过期，否则首次启动可能锁死 —— 或显式 require renew）

**触发条件**：
- ❌ 暂未触发（单机部署不痛）
- ✅ **#27 multi-host TCP transport ship → 立即触发**：TCP 暴露不带 TTL 是安全债
- **建议节奏**：作为 **#27 的成对工作**打包 v2.4，不要单独排（**避免 multi-host ship 后裸跑无 TTL**）

**关联**：
- [conventions § 0.4 Enforce 机制](../rules/conventions.md)
- [ADR-0026 SecretManagement plaintext-never-echo](../design/decisions/0026-user-secret-management-bc.md)（同思路）

### C3 AgentInstance.HomeDir 自动 scaffold

**简述**：当前 worker 启动真 agent（claude-code / codex / opencode）需操作员**手工**在 `~/.agent-center-worker/agents/<id>/` 建目录 + 填 `instructions.md` + `mcp_config.json`。这一步是 v2.3-3c 原本要做但没开。**做成 `agent-center agent init <id>` CLI + worker daemon 启动时自动 scaffold + 默认模板**。

**产品价值**：
- 谁感知：每个第一次跑真 agent 的用户、新用户上手 demo
- 现状痛点：**新用户上手最大 friction 点**之一。文档没写清 → enroll worker → spawn task → 失败 → 自查 home dir 路径 → 手 mkdir + cat 模板（如果知道哪里有模板）→ 重试
- 用户价值：**P0**。新用户体验、demo 流畅度直接受益；老用户每加一个 agent 都省事
- **现状是真痛**（#29 audit § 5 已明确列）

**技术成本**：~3-4h
- BC 涉及：Workforce（AgentInstance）+ Worker daemon
- 方案 A（**推荐**）：worker daemon 启动 agent 时检测 home_dir 是否存在 + 必需文件全 → 缺则自动 scaffold（模板从 worker 自带 assets/agent-defaults/<kind>/ 复制）
- 方案 B：CLI `agent-center agent init <id> [--kind=claudecode]` 显式触发
- 推荐 A + B 都做：A 解决"忘了"，B 解决"想预先准备"
- 模板内容：从 `assets/skills/` 现有素材衍生 + `examples/mcp_config.<kind>.json`
- 风险：模板演进时已 scaffold 的旧 agent 不会自动升级 —— 需要 `agent doctor` 工具能 diff（不在本次 scope，但留出口）

**触发条件**：
- ✅ **已发生**：#29 audit § 5 明确点名 + AgentCenterDev 暗示原 v2.3-3c 未开
- **建议节奏**：v2.4 早期。**优先级仅次于 C1**（C1 是架构清理，C3 是用户体验直击）

**关联**：
- [ADR-0024 AgentInstance 一等公民](../design/decisions/0024-agent-instance-first-class.md)

### C4 BlobStore GET endpoint

**简述**：#29 只接通了 `POST /admin/blob/put`（worker → center 上传 artifact blob）；**read path** 还是 server-local FS。要加 `GET /admin/blob/get?ref=<>` 让 web console / CLI 能读 artifact blob。

**产品价值**：
- 谁感知：Web 用户（看 task artifact，比如 agent 写的 markdown 报告）、CLI 用户
- 现状痛点：artifact 列表能看到 ref 但点不开（"看到挂在那但拉不下来"）—— Web console 看 task detail 时，artifact 区只能显示元数据，无法预览/下载内容
- 用户价值：**P1**。直接增强 Web 主入口的可用性
- **设计已就绪**：BlobStore 抽象已抽出，只差 read endpoint

**技术成本**：~2-3h
- BC 涉及：（admin transport 层）+ BlobStore service
- 方案：scope `blob:read`，路径校验同 put（防 OS escape）；content-type 简单嗅探或 octet-stream；web 端加 artifact 预览 UI（markdown / image / 文本类）
- 风险：大文件流式传输 —— v0 全量 buffer，文件大小硬 cap（如 10MB），后续按需做流式

**触发条件**：
- 🟡 **准备发生**：v2.3 后 artifact 体系打通，下个紧自然就是"能看到 artifact 内容"
- ❌ 暂未明确卡 user flow（用户没在 #agent-center 抱怨）—— 但**直觉式**痛点
- **建议节奏**：v2.4 中期。**优先级低于 C1/C3** —— 后两者直接卡用户、本条增强体验

**关联**：[#29 audit § 5](v2.3-audits/v23-3b-real-agent-dispatch-audit.md)

### C5 master_key destructive rotation 工具化

**简述**：当前 master_key（SecretManagement BC AES-256 加密 root key）的轮换走 **destructive rotation**（手工 + dangerous，需停服）。v2.4 把这个流程**工具化** —— `agent-center secret master-key rotate` 子命令 + 备份 / 校验 / 回滚步骤集成。

**产品价值**：
- 谁感知：长期部署的运维、需要符合密钥轮换合规的场景
- 现状痛点：流程**存在但 lossy** —— 操作员要手敲 sql + 手 backup + 手 verify。一旦 #27 ship → 多机部署 → master_key 泄漏面变大 → 轮换需求上升
- 用户价值：**条件性 P1**（与 C2 类似 —— #27 ship 后才急）
- **不是 v3 KMS-backed master key**（那是 roadmap 已 park 项）：本条只是把现有 destructive rotation 自动化 + safe，**不引入 KMS 依赖**

**技术成本**：~3-4h
- BC 涉及：SecretManagement
- 方案：CLI 子命令 → 强制要求 backup → 用新 key 重新加密所有 UserSecret rows → 校验 round-trip → 提交事务（失败回滚）
- 加 dry-run flag
- 风险：执行期间不能并发写 UserSecret（要求服务静默或锁）

**触发条件**：
- ❌ 暂未触发（单机自用极少轮换）
- ✅ **#27 ship 后高**
- **建议节奏**：和 C2 打包跟 #27 配套，否则单独排意义不大

**关联**：[docs/operations/master-key.md § 4](../operations/master-key.md) 已留 v3 候选位（但本条工具化是 v2.4 的 destructive rotation，不是 v3 的 KMS 替代）

### C7 SSE per-conv subscribe wiring (v2.1-D 残留)

**简述**：[v2.1-backlog.md "v2.1-D"](v2.1-backlog.md) 残留项。SPA `useSSE` 打开 EventSource 但从未 `POST /api/sse/subscribe`，导致 `b.channels[userID]` 是空集，per-conv SSE event 全不投到前端。冷启 + 重新访问通过 react-query invalidate 弥补；只在**多 tab / leave-tab-open** 场景挂掉。

**产品价值**：
- 谁感知：开多 tab 的 Web 用户、长时间挂着 channel 列表看 badge tick 的用户
- 现状痛点：单 tab 单活跃 workflow OK；多 tab 体验明显劣化
- 用户价值：**P1**。修一个长期挂着的 UX 回归
- **本条 v2.1 已识别 + 估时**，挪不动只是因为不阻塞 GA

**技术成本**：~1.5h（v2.1 backlog 原估时）
- BC 涉及：Conversation + 前端 SPA
- 方案：`useSSEConversationSubscribe(convId)` hook（mount 时 POST，unmount unsubscribe）；Channels / DMs list 对每个可见 conv 都 subscribe；ChannelDetail / DMDetail 对当前 conv subscribe
- 风险：subscribe 风暴（用户有 50 channel）—— v2.1 估时未考虑批量；可能要补 batch subscribe endpoint

**触发条件**：
- 🟡 **部分发生**：v2.1 ship 时已识别，未升级是因为单 tab 够用
- ❌ 用户尚未明确抱怨
- **建议节奏**：v2.4 后期，单独打 1 个小 ST 收尾

### C8 Issue projection 加 kind

**简述**：#31 audit § 5 明确推迟 —— Issue projection 故意省 `kind`（AR 不暴露）；要加要先扩 Discussion BC AR + 迁移。

**产品价值**：
- 谁感知：Web 用户（issue 分类筛选）、未来"按 issue 类别看板"场景
- 现状痛点：Web Issues 页只能按 status filter；不能按"agent_finding vs user_question vs ..."分类
- 用户价值：**P2**。增强不阻塞
- 当前 SPA 不需要 issue kind 也能用 —— 该字段是**未来视图能力的种子**

**技术成本**：~3h
- BC 涉及：Discussion（AR + 迁移 + service + projection）
- 方案：Issue AR 加 `kind` 字段（枚举 v0：user_question / agent_finding / decision / 其它）；迁移设默认值；projection / endpoint / SPA 三层各小动
- 风险：**枚举值定义是讨论点**，需 @oopslink + @AgentCenterDev 协商（哪些维度真有用）

**触发条件**：
- ❌ 暂未触发
- **建议节奏**：v2.4 之后（v2.5？）。**优先级低**，无明确用户需求驱动

### C9 messages 覆盖索引 (v2.1-E 残留)

**简述**：[v2.1-backlog.md "v2.1-E"](v2.1-backlog.md) 残留。`countUnread` SQL `WHERE conversation_id = ? AND id > ?` 走当前 `idx_messages_conv (conversation_id, posted_at)` —— `id > ?` 是 residual predicate，按行扫。加 `(conversation_id, id)` 覆盖索引让它变 range seek。

**产品价值**：
- 谁感知：单 channel 超过 10K message 的重度用户
- 现状痛点：v2.1 实测 504 message ~3ms，**完全没问题**
- 用户价值：**P3**。性能 hygiene，**没有压力**
- **本条诚实归类为"等触发"**

**技术成本**：~30min
- 方案：迁移 0029（或顺序号）加 index
- 风险：极低

**触发条件**：
- ❌ 暂未触发
- ✅ 单 channel 累计 message 数突破 10K 或 /channels 冷加载 > 200ms → 即做
- **建议节奏**：观察 → 触发后立即做（半小时收益）

### C10 测试基建：admin-write 紧凑场景 hardening

**简述**：#33 audit 暗示 —— 未来 test case 若在 ≤10ms 内发 ≥2 个 admin write（涉及 admintoken pump 节流）可能复现 SQLITE_BUSY race。当前**没问题**，但 test infra 没有自动 guard。

**产品价值**：
- 谁感知：开发自身（CI 稳定）
- 现状痛点：**没发生**，但有"埋伏" — 任何新 test 写得密了可能复现 flake
- 用户价值：**P3**。开发体验 hygiene
- 不是用户能感知的特性

**技术成本**：~2h
- 方案 A：每个 admin-write 集成 test 默认 `app.AdminTokenSvc.Close()`（已有模式）
- 方案 B：写 lint / 测试 helper 自动 detect → close
- 方案 C：在 prod 端用更宽松的 busy_timeout（不推荐，是 prod 改动）

**触发条件**：
- ❌ 暂未触发
- ✅ CI 在某次新 test 加入后再次出现 SQLITE_BUSY → 立即做
- **建议节奏**：守在 v2.4 backlog，被 CI 提醒时升优先级

## § 99 推荐打包

**v2.4 主线（PM 推荐）**：**C1 + C3 + (C2 + C5 + C6 跟 #27 一起) + C7**

| 主题 | 候选 | 估时 | 受益 |
|---|---|---|---|
| Conversation 边界收尾 | C1（projection 加 project_id）| ~4-6h | 用户准确性 |
| 真 agent 用户体验 | C3（HomeDir 自动 scaffold）| ~3-4h | 新用户、操作员 |
| 多机部署套件（**跟 #27 ship 同步 / 紧跟随**）| C2 + C5 + C6 | ~9-11h | 部署运维 |
| 实时 UX 收尾 | C7（SSE per-conv subscribe）| ~1.5h | 多 tab 用户 |

**总计 ~17-23h**（不含 #27 multi-host TCP 本身的 ~10h）

**理由**：
1. **C1 是 v2.3 主题最后一公里** —— BC 边界收尾，乘热做最便宜
