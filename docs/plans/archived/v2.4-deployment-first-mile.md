## § 2 Scope（窄）

### § 2.1 In scope

| 维度 | 内容 |
|---|---|
| **核心** | "解压好的目录里执行什么命令" + 这命令做什么 |
| **覆盖** | Center install + Worker install + 二者 upgrade |
| **操作系统** | macOS + Linux（amd64 + arm64） |
| **命令对称性** | 同一条命令既装也升级（自动检测 + 走对应路径） |
| **服务级安装** | systemd（Linux） / launchd（macOS）；开机自启 |
| **Worker enroll UX** | WC `/fleet` 加 "Add Worker" 入口 → Modal 显示一条 install 命令（含已填好 bootstrap token + fingerprint）→ 用户复制到已解压目录粘贴执行 → WC 实时高亮新上线 worker |
| **Upgrade 模型** | atomic symlink swap + 失败回滚（旧版本保留 N=3）+ DB migration 自动 |
| **失败兜底** | 装失败 / enroll 失败 / token 用过 / token 过期 / worker 连不上 center 都有明确文案 + recovery 路径 |

### § 2.2 Out of scope（推迟）

| 维度 | 推迟到 |
|---|---|
| **Tar 包从哪下、怎么分发** | v2.5 或 v3 deployment-as-product 主题 |
| **CI / release pipeline** | v3 主题 |
| **签名 / 完整性校验** | v3 主题 |
| **Dockerfile / docker-compose** | v3 主题 |
| **`agent-center diagnose` 自检工具** | v3 主题 |
| **Auto agent config**（claude-code instructions / mcp_config 模板填好）| v2.4 紧跟进（C3 候选），不进 v2.4 主线 |
| **Demo task 自动建好**（"装完看到 Hello World 任务"）| 不做 —— @oopslink 已明确"窄" |
| **Prometheus / OTel metrics endpoint** | v3 主题 |
| **多 user / 团队部署形态** | 团队故事 v3+ 整体重做 |

**重要边界**：@oopslink 2026-05-26 09:27 「tar 从哪里来不用管 默认有 tar 包并解压好了」。本文档**前提**就是：用户已经有 `agent-center-v<X>-<os>-<arch>/` 这个目录，在那个目录里执行命令。

## § 3 用户路径 — 三个 journey

### § 3.1 Journey A：第一次装 Center

**前提**：用户在 Mac / Linux 上，已经把 center tarball 解压到某个目录（如 `~/Downloads/agent-center-v2.4.0-darwin-arm64/`）。

**步骤**（用户体验角度）：
1. `cd` 进解压目录
2. 执行**一条命令**（具体名字工程定，假设 `./install`）
3. 命令做完后输出**一个 URL**（如 `http://127.0.0.1:7100`），告诉用户「在浏览器打开它继续」
4. 用户打开 URL → 进入空的 Web Console，看到 Home 页 / Fleet 页 / 各 nav 入口
5. **AgentCenter 装完跑起来了**

**成功画面（命令最终输出）**：
```
✓ AgentCenter v2.4.0 installed
✓ Service running on Mac launchd (auto-start enabled)
✓ Database initialized at <path>
✓ Bootstrap admin token created (saved to <path>)
✓ Web Console: http://127.0.0.1:7100

Open the URL above to continue setup.
```

**成功路径耗时目标**：解压 → 跑命令 → 看到 URL ≤ **30 秒**。

**失败画面 — 必须覆盖的几种**：
- 端口被占（7100 / 7099 / 7300 任一）→ 提示"端口 X 已被 Y 进程占用 + 杀掉它或换端口（`./install --port=...`）"
- 已有同版本安装（idempotent）→ 提示"已安装相同版本，无变化"
- 已有旧版本安装 → 切到 upgrade 路径（不报错，看 [§ 3.3](#-33-journey-cupgrade)）
- 权限不够（systemd unit 装不到 /etc/systemd/system 等）→ 提示要 sudo 或用 `--user-mode`
- DB 文件锁住（其它进程占）→ 提示具体冲突 + 怎么解

### § 3.2 Journey B：第一次加 Worker

**前提**：center 已装完跑起来，用户已在 WC 里。

**步骤**：
1. 用户进 WC `/fleet` 页 → 点 "Add Worker" 按钮
2. 弹出 Modal，显示**一条命令** + Copy 按钮 + "Waiting for worker..." 状态
3. 用户切换到目标机器（开发机 / VPS / 任何机器）—— 已经在那台机器上解压好 worker tarball
4. `cd` 进解压目录 → 粘贴命令执行
5. 命令完成后输出 "Worker enrolled, connected to center <host>"
6. 用户回到 WC，看到 Modal 状态变 "Worker connected ✓"，Fleet 表格里**新 worker 行实时高亮**
7. 关 Modal，看到 worker 在 Fleet 列表里正常显示

**WC Modal 长这样**（mockup）：
```
┌─ Add a Worker ──────────────────────────────────┐
│                                                  │
│ Make sure the AgentCenter tarball is extracted   │
│ on your worker machine. Then run this command    │
│ inside the extracted directory:                  │
│                                                  │
│ ┌─────────────────────────────────────────┐ Copy│
│ │ ./install worker \                       │     │
│ │   --bootstrap=tcp://abc1...:7300 \       │     │
│ │   --token=acat_xxxxxxxxx                 │     │
│ └─────────────────────────────────────────┘     │
│                                                  │
│ Token expires in 30 minutes.                     │
│ ⏳ Waiting for worker...                          │
│                                                  │
│                              [ Cancel ] [ Done ] │
└──────────────────────────────────────────────────┘
```

Token 用一次就 burn。fingerprint 嵌进 bootstrap URL（不另开字段）。

**成功路径耗时目标**：用户在 WC 点 "Add Worker" → 到目标机器跑完命令 → WC 看到 worker 上线 ≤ **2 分钟**（含切机器 + 复制粘贴）。

**失败画面 — 必须覆盖**：
- 用户跑命令但 token 已被另一台机器用过 → worker 端报"Token already used. Get a fresh one in the Web Console."；WC Modal 看到 worker 连接尝试 + 失败标记
- token 过期（30 分钟 cap）→ 同上，WC Modal 提示 "Token expired"，提供 "Generate new token" 按钮
- worker 机器连不上 center（防火墙 / 错的 host）→ worker 端报"Cannot reach center at <host>:7300. Check network / firewall."
- TLS fingerprint 不对（中间人？错连别的 center？）→ worker 端报"Center fingerprint mismatch. Possible MITM. Aborting."
- 同 worker name 已注册 → 用户在 WC Modal 选"添加" vs "覆盖"

### § 3.3 Journey C：Upgrade（Center 或 Worker）

**前提**：用户有新版 tarball（如 `agent-center-v2.4.1-darwin-arm64/`），已解压到任意目录（**不一定是原目录**）。

**步骤**：
1. `cd` 进新版本目录
2. 跑**同一条命令** `./install`（也可以加 `./install center` / `./install worker`，但默认无歧义时自动识别）
3. 命令检测：当前机器上已经安装了 v2.4.0 → 走 upgrade 路径
4. 工程实现：将 v2.4.1 装到 `/opt/agent-center-v2.4.1/`（保留 v2.4.0 不删）→ atomic 切 `/opt/agent-center` symlink 指向 v2.4.1 → restart service → 校验 health
5. 校验失败 → 自动回滚（symlink 切回 v2.4.0 + restart）+ 报错"Upgrade failed, rolled back to v2.4.0. Reason: <X>"
6. 成功 → 报"Upgraded v2.4.0 → v2.4.1. Service running."
7. 旧版本路径保留最近 N=3 个（v2.4.0、v2.3.x、v2.3.x-1），更老的需用户手动 GC（写在文档）

**成功路径耗时目标**：跑命令到 service 恢复 ≤ **15 秒**（含 DB migration + service restart）。

**Worker upgrade**：完全同形态，无 enroll 重复（已经 enrolled 的 worker 直接 binary swap + restart）。

**跨 minor 版本 upgrade**（v2.x → v2.y）：自动 migration 包含
**跨 major 版本 upgrade**（v2 → v3）：v3 ship 时单独 guide，不承诺自动

## § 5 失败 & 恢复矩阵

| Journey | 失败模式 | 用户看到 | 推荐 recovery |
|---|---|---|---|
| Install center | 端口被占 | "Port 7100 in use by PID <X>" | 杀进程 or `--port=...` |
| Install center | 已装相同版本 | "Already installed v2.4.0" | 无 action 退出 |
| Install center | DB 锁 | "DB file locked by PID <X>" | 杀冲突 or 停服后重跑 |
| Install center | 系统权限不够 | "Need sudo for systemd unit install" | 用 `sudo` 或加 `--user-mode` |
| Install worker | center 不可达 | "Cannot reach <host>:7300" | 检查网络 / 防火墙 / center 是否跑 |
| Install worker | token 已 burn | "Token already used" | WC 重新 mint |
| Install worker | token 过期 | "Token expired (30 min cap)" | WC 重新 mint |
| Install worker | TLS fingerprint 不匹配 | "Center fingerprint mismatch" | 检查 center URL / 防 MITM |
| Install worker | 同 worker 名已存在 | "Worker 'mbp-1' already enrolled" | WC 选 "Add another" or 改 worker name |
| Upgrade | 启动失败 | "Upgrade failed, rolled back to v<old>" + 具体错 | 看 reason → 修 → 重试 |
| Upgrade | DB migration 失败 | "Migration v<X> → v<Y> failed" + 具体错 | 看 migration log → 修数据 → 重试 |
| Upgrade | 磁盘空间不够（旧版本累积）| "Disk full, free up space" | 手动 GC 旧版本路径 |

## § 7 验证标准

v2.4 ship 前**必须**：

1. **@AgentCenterPD 在 Mac 上走通**（**本期 acceptance 委托给 PM；2026-05-26 @oopslink 在 task #34 thread 决定**）：
   - 解压 → `./install` → 30 秒内拿到 WC URL → 打开看到 Fleet 空状态
   - 在第二个工作目录（或同机另一个用户）：解压 worker tarball → 复制 WC 的 install 命令 → 粘贴 → 2 分钟内看到 worker 在 Fleet 高亮上线
   - 重新解压新版本 → 跑 `./install` → 15 秒内看到 service 恢复
   - 故意打断 upgrade（kill 中间步骤）→ 看到 rollback 成功
2. **失败模式至少抽样验证 5 条**（[§ 5](#-5-失败--恢复矩阵)）—— 端口冲突、token burn、token expire、fingerprint mismatch、升级失败回滚
3. **平台覆盖**：**本期只验 mac arm64**（runtime 当前所在平台）。mac amd64 抽样可选；**Linux 本期不验 / 不声称 ship-ready** —— A2 systemd 代码仍写但未经验证，留作 next iteration 单独验。

**ship 标准 = @AgentCenterPD 走完 1-2 + 平台覆盖判定**「丝滑、产品级」，给 ship 信号 → @oopslink 拿 release notes 决定何时切版本。不是 "测试通过 + 文档齐"。
