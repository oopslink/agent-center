## § 1. Auth 流程验收（12 项）

### 1.1 Fresh install → /signup

| 步骤 | 期望 |
|---|---|
| 1. 完全清库（rm -rf ~/.agent-center） | DB 为空 |
| 2. `agent-center start` | center 启动 OK，无 panic |
| 3. 浏览器访问 `http://localhost:8080/` | 重定向到 `/signup` |
| 4. 访问 `/anywhere`（如 `/organizations/default`）| 重定向到 `/signup` |

### 1.2 Signup 表单字段校验

| 输入 | 期望 |
|---|---|
| 空 display_name | inline 表单错误 |
| display_name > 40 字符 | inline 表单错误 |
| 5 位 passcode | inline "请输入 6 位数字" |
| 7 位 passcode | inline 错误 |
| passcode 包含字母 | inline 错误 |
| confirm_passcode 不匹配 | inline 错误 |
| 空 first_organization_name | inline 错误 |
| organization_slug 含大写 | inline "slug 只能 [a-z0-9-]" |
| organization_slug 长度 < 3 | inline 错误 |
| organization_slug 含 `_` 或空格 | inline 错误 |
| organization_slug = `--invalid` | inline 错误 |
| 全部合法字段 | 提交按钮可点 |

### 1.3 Signup 成功

| 步骤 | 期望 |
|---|---|
| 1. 提交合法 signup form (例：display=`Hayang`, passcode=`123456`, org_name=`My Workspace`, org_slug=`my-workspace`) | 跳 `/organizations/my-workspace/` |
| 2. 查 DB | `identities` 1 行 (kind=user, account_status=active, passcode_hash 非空) + `organizations` 1 行 (slug=`my-workspace`) + `members` 1 行 (role=owner, status=joined) |
| 3. 查 cookie | `session` cookie 存在，HttpOnly+Secure+SameSite=Lax |
| 4. 查 events 表 | `identity.created` + `organization.created` + `member.added` + `auth.signed_in` 各 1 行 |

### 1.4 Signin 流程

| 步骤 | 期望 |
|---|---|
| 1. logout（或清 cookie）| 跳 `/signin` |
| 2. 输错 passcode 3 次 | 每次报 `ErrPasscodeInvalid`；无锁定（v2.6 不出 lockout）|
| 3. 输正确 passcode | 跳回 last visited org 或 `/organizations/{slug}/` |
| 4. 检查 cookie | 新 session JWT |
| 5. 查 events | `auth.signin_failed` 3 行 + `auth.signed_in` 1 行 |

### 1.5 Signin display_name 不存在

| 输入 | 期望 |
|---|---|
| 输入不存在 display_name + 任意 passcode | 报 `ErrPasscodeInvalid`（**不能** 暴露"用户不存在"，防 enumeration）|

### 1.6 Signin Identity disabled

| 步骤 | 期望 |
|---|---|
| 1. owner 把另一个 user 改 account_status=disabled（CLI / DB 模拟）| OK |
| 2. 用该 user 试 signin | `ErrPasscodeInvalid`（不暴露 disabled 原因）|

### 1.7 Logout

| 步骤 | 期望 |
|---|---|
| 1. 已登录状态，点 `/me` → Logout（或调用 POST `/signout`）| 跳 `/`；cookie 清除 |
| 2. 再访问任意 org 页 | 跳 `/signin` |
| 3. 查 events | `auth.signed_out` 1 行 |

### 1.8 JWT 过期跳 /signin

| 步骤 | 期望 |
|---|---|
| 1. 已登录状态，手动改 cookie exp 或 wait 7d | 任意请求 401 |
| 2. SPA 拦截 401 | 全屏跳 `/signin`（不出 modal） |

### 1.9 DS-4 即时失效（admin disable Identity）

| 步骤 | 期望 |
|---|---|
| 1. user A 已登录有效 JWT | OK |
| 2. owner 把 A 改 account_status=disabled | OK |
| 3. A 的浏览器立即下一个请求 | 401 → 跳 `/signin`（DS-4 fail-safe） |

### 1.10 Cookie 缺失 / 篡改

| 输入 | 期望 |
|---|---|
| 无 session cookie | 跳 `/signin` |
| 篡改 cookie 值 | 401 + 跳 `/signin` |
| cookie 用其它密钥签名 | 401 |

### 1.11 Passcode 改密码（user）

| 步骤 | 期望 |
|---|---|
| 1. `/me` 页输入旧 + 新 + confirm | 成功 |
| 2. 用新 passcode signin | OK |
| 3. 用旧 passcode signin | 失败 |
| 4. 查 events | `identity.passcode_changed` 1 行 |

### 1.12 Signin display_name 跨 kind 唯一性

| 步骤 | 期望 |
|---|---|
| 1. 已存在 user `Hayang` | OK |
| 2. owner add agent display_name=`Hayang` | OK（不强制唯一）|
| 3. signin 输入 `Hayang` | 找到 user kind 那个（partial unique on kind=user）|

## § 2. Organization CRUD 验收（10 项）

### 2.1 Org switcher 顶 bar

| 步骤 | 期望 |
|---|---|
| 1. signup 创建 `Org A` (slug=`a`) → signup 再创建 `Org B` (slug=`b`)（通过 "Create new organization" 入口）| 两 Org 都建立；当前 Identity 都是 owner |
| 2. Top Bar Org switcher 下拉 | 显示 `Org A` + `Org B` + "+ Create new organization" |
| 3. 在 Org A 的 `/channels` 点 Org B | 跳到 `/organizations/b/channels` |

### 2.2 多 tab 多 org 并发

| 步骤 | 期望 |
|---|---|
| 1. tab 1 打开 `/organizations/a/channels` | OK |
| 2. tab 2 打开 `/organizations/b/channels` | OK |
| 3. 两 tab 操作互不干扰 | 各自的 channel 列表独立 |

### 2.3 Create Organization

| 步骤 | 期望 |
|---|---|
| 1. Org switcher "+ Create new organization" | Modal 弹出 |
| 2. 输入 name + slug | 提交 OK |
| 3. 自动 redirect 到新 Org | `/organizations/{new-slug}/` |
| 4. 查 DB | `organizations` +1 + `members` +1 (current Identity role=owner) |

### 2.4 Slug 唯一性

| 步骤 | 期望 |
|---|---|
| 1. 已有 Org `slug=alpha` | OK |
| 2. 创建另一个 Org slug=`alpha` | 422 + `ErrOrganizationSlugTaken` |

### 2.5 Slug 格式校验

| 输入 | 期望 |
|---|---|
| `Alpha` (含大写) | 拒绝 |
| `-alpha` (首字符 -) | 拒绝 |
| `alpha-` (尾字符 -) | 拒绝 |
| `a` (太短) | 拒绝 |
| `a` × 41 (太长) | 拒绝 |
| `al--pha` (连续 -) | 拒绝 |
| `alpha-1-beta` | 接受 |

### 2.6 Org Settings UI

| 步骤 | 期望 |
|---|---|
| 1. 访问 `/organizations/{slug}/settings`（owner） | 显示 name / slug / description / created / Delete 按钮 |
| 2. 改 name + 保存 | OK + emit `organization.updated` |
| 3. 改 slug + 保存 | OK + URL redirect 到新 slug + emit `organization.updated` |
| 4. admin 访问 settings | name + description editable；slug 不可编辑 |
| 5. member 访问 settings | 全只读 |

### 2.7 Organization 软删

| 步骤 | 期望 |
|---|---|
| 1. owner 点 Delete | 二次确认 modal |
| 2. 确认 | 软删；跳出该 org；emit `organization.deleted` |
| 3. 查 DB | `organizations.deleted_at` 非空 + 所有 Members.status=disabled |
| 4. 等几秒（事件驱动级联） | Workforce / Conversation / SecretManagement 软删该 org 资源 |
| 5. 再访问 `/organizations/{deleted-slug}/...` | 404 |
| 6. Create 新 Org slug=deleted-slug | OK（slug 释放）|

### 2.8 Org Settings admin 权限拒绝

| 步骤 | 期望 |
|---|---|
| admin 在 settings 页点 Delete | 按钮 disabled / 403（**owner-only**）|
| admin 改 slug | 403 |

### 2.9 Org switcher 不显示已软删的

| 步骤 | 期望 |
|---|---|
| 软删 Org B 后 | switcher 下拉只剩 active Orgs（不显示 B）|

### 2.10 没有 Member 的 Org 进不去

| 步骤 | 期望 |
|---|---|
| user A 访问 user B 创建的 `/organizations/b/...` | 403 + "Not a member of this Organization" |

## § 3. Member CRUD 验收（12 项）

### 3.1 Sidebar Members group

| 步骤 | 期望 |
|---|---|
| 1. 访问 `/organizations/{slug}/`（任意 member）| 左 sidebar 有 Members group，可展开 |
| 2. 展开 Members | 显示 Humans 子项 + Agents 子项 |
| 3. 点 Humans | 跳 `/organizations/{slug}/members/humans` |
| 4. 点 Agents | 跳 `/organizations/{slug}/members/agents` |

### 3.2 Humans tab

| 步骤 | 期望 |
|---|---|
| 1. 访问 `/organizations/{slug}/members/humans` | 列出所有 kind=user 的 Member（含 disabled）|
| 2. owner 看到 + Add User 按钮 | OK |
| 3. admin 看到 + Add User 按钮 | OK |
| 4. member 看不到 + Add User 按钮 | OK（只读）|

### 3.3 Add User

| 步骤 | 期望 |
|---|---|
| 1. owner 点 + Add User | Modal 弹出 |
| 2. 输入 display_name + temp_passcode + role | 提交 OK |
| 3. 查 DB | `identities` +1 (kind=user) + `members` +1 (role=指定) |
| 4. 新 user 用 temp_passcode signin | OK |
| 5. 新 user 走 `/me` 改 passcode | OK |

### 3.4 Agents tab

| 步骤 | 期望 |
|---|---|
| 1. 访问 `/organizations/{slug}/members/agents` | 列出 kind=agent 的 Member |
| 2. 每行显示 "running on {worker}" | JOIN agent_instances 取 worker name |
| 3. owner/admin 看到 + Add Agent | OK |

### 3.5 Add Agent

| 步骤 | 期望 |
|---|---|
| 1. owner 点 + Add Agent | 跳 `/organizations/{slug}/members/new?kind=agent` |
| 2. 输入 display_name + description + role + worker_id (optional) | 提交 OK |
| 3. 查 DB | `identities` +1 (kind=agent, passcode_hash IS NULL) + `members` +1 |
| 4. 如选了 worker | `agent_instances` 关联到 worker + identity_id 指向新 Identity |

### 3.6 Change role

| 步骤 | 期望 |
|---|---|
| 1. 选 Member 行 "···" → Change role | 子菜单显示 owner/admin/member |
| 2. 升 admin → owner（admin 操作）| 拒绝（admin 不能 promote 到 owner）|
| 3. 升 admin → owner（owner 操作）| OK + emit `member.role_changed` |
| 4. 降 owner → admin（owner 操作）| OK |
| 5. 查 events | payload 含 old_role + new_role |

### 3.7 [[organization-min-owner]] —— 最后 owner 守门

| 步骤 | 期望 |
|---|---|
| 1. Org 只有 1 owner | OK |
| 2. owner 把自己降级 admin | 422 `ErrLastOwnerCannotChangeRole` |
| 3. owner disable 自己（admin 操作）| 422 `ErrLastOwnerCannotDisable` |
| 4. 加 owner B 后再降级 A | OK |

### 3.8 [[organization-min-owner]] 并发 race（DS-2 关键测试）

| 步骤 | 期望 |
|---|---|
| 1. Org 有 2 owner (A, B) | OK |
| 2. 写脚本并发 disable A + disable B（同时发请求）| 其中一个 OK，另一个 422 `ErrLastOwnerCannotDisable` |
| 3. 查 DB | Org 仍有 ≥1 active owner（**M2 invariant 保住**）|

### 3.9 Disable Member

| 步骤 | 期望 |
|---|---|
| 1. admin 选某 user Member "···" → Disable | OK + emit `member.disabled` |
| 2. 该 user 用 cookie 访问 | 仍能访问（Identity.account_status=active），但 / organizations/{slug}/ 部分 403 |
| 3. 该 user 切到别的 org | OK（Member.status 是 org 局部） |

### 3.10 Re-enable Member

| 步骤 | 期望 |
|---|---|
| 1. admin 选 disabled member → Re-enable | OK + emit `member.re_enabled` |
| 2. 查 DB | status=joined + disabled_at IS NULL |

### 3.11 Permission Matrix 完整 walkthrough（§ 8 of v2.6-design）

按 Permission Matrix 表格逐角色逐操作 walk through，列入 sanity check appendix。

### 3.12 Identity self-disable UI 不存在（OQ-4 closed）

| 步骤 | 期望 |
|---|---|
| 访问 `/me` 页 | 无 "Delete Account" 按钮 |

## § 10. ship-block / non-block 分级

### Ship-block（必须全部 ✓）

- § 1 Auth 流程
- § 2 Organization CRUD
- § 3 Member CRUD
- § 4 RBAC matrix
- § 5 Schema migration
- § 6 Supervisor cut（无残留）

- § 8 全部 events 发射

### Non-block（高优 polish）

- § 9 Performance smoke
- UI 微调 / 空状态 / error state

### Acceptance 流程

1. dev 报"Phase 1-4 全部完成"
2. PM 按本计划单逐节验收
3. 任一 ship-block 项失败 → 反馈 dev → 修 → 重测
4. 全部 ✓ → PM 报 @oopslink "ship-ready"
5. @oopslink 拍 ship → 打 v2.6.0 tag
