# v2.7 Domain Refactor Plan

> Branch: `design/v2.7-domain-refactor`
> Date: 2026-05-29
> Scope: ProjectManager + Agent + Environment + Conversation/File URI foundation

## 0. Position

v2.6 shipped Identity, Organization, Member, authentication, and Organization-scoped routes. v2.7 should use that foundation to correct the core domain model around project management and long-running agents.

This plan is not an implementation spec yet. It is the planning and boundary document for the next design pack. After review, the accepted parts should be split into tactical docs and ADRs before coding starts.

v2.7 does not need backward compatibility. Dropping and rebuilding tables is acceptable when it gives a cleaner domain model.

## 1. End State

AgentCenter should model four separate responsibilities:

| Responsibility | Bounded Context | Core truth |
|---|---|---|
| Work management | **ProjectManager** | Projects, project members, issues, tasks, task/issue subscription truth, task/issue state transitions |
| Agent product model | **Agent** | Agent profile, skills, runtime binding, lifecycle controls, work queue, activity stream |
| Runtime environment | **Environment** | Workers, AgentController, local process/session control, worker-initiated center connection, FileTransfer |
| Message stream | **Conversation** | Conversations, messages, participants as message-layer projection, read state, attachments by URI |

The important boundary is:

- ProjectManager decides what work exists and who should be notified.
- Agent decides which Agent is assigned what work and tracks the logical work item.
- Environment starts, monitors, and talks to local processes.
- Conversation carries human/agent communication but does not infer task or issue state.

## 2. Locked Decisions

### 2.1 ProjectManager

- The bounded context name is **ProjectManager / 项目管理域**.
- A Task must belong to exactly one Project.
- An Issue must belong to exactly one Project.
- There are no global Tasks and no cross-Project Tasks.
- A Task may be independent or derived from an Issue.
- Task and Issue each bind one stable Conversation.
- ProjectMember exists in the first version. Its first responsibility is "who is in this Project" and project message delivery scope, not fine-grained permission control.
- Subscription truth for Task/Issue lives in ProjectManager.
- Creator and current assignee are always effective subscribers.
- Manual subscribers are additive.
- Conversation participants mirror effective subscribers.
- On reassignment, the old assignee is removed from effective subscribers unless they are also creator or manual subscriber.
- Creator/current assignee cannot be removed from subscription while they still hold that role. Creator mute is future scope.

### 2.2 Task and Issue status

Issue states:

```text
open -> in_progress -> resolved -> closed
open/in_progress -> withdrawn
resolved/closed -> reopened -> open
```

`triaged` is final-shape only, not v2.7.

Task states:

```text
open -> assigned -> running -> blocked -> running
running -> completed -> verified
open/assigned/running/blocked -> canceled
completed/verified -> reopened -> open
assigned -> open  (unassign)
```

Rules:

- `blocked` is explicit business state and requires a Conversation message/reason.
- `completed` is set by assignee/agent.
- `verified` is set by any authorized identity in the Project who is **not** the task's completing assignee. This may be the creator, a project manager role, or **another Agent**. An identity can never verify a Task it completed itself (no self-verification; enables Agent peer-review). Decided 2026-05-29, see §10 OQ4.
- Task/Issue state never changes by system inference from Conversation messages.

### 2.3 Agent

- Agent is an independent bounded context.
- Agent is logically long-running.
- There is no core **AgentRun** concept.
- Agent includes:
  - profile: name, description, model, CLI, environment variables
  - skill list
  - runtime config: selected Worker, runtime home, memory data
  - lifecycle controls: start, stop, restart, reset
  - observability: totals, heartbeat/status, activity events
- CreateAgent must choose a Worker.
- Agent runtime Worker binding is immutable in v2.7.
- Worker migration / cloud Agent entity is roadmap.
- Agent degraded state is out of the first version.

AgentRuntimeConfig first version:

- No configurable `work_dir`.
- Worker-local home:

```text
~/.agent-center/workers/{worker_id}/agents/{agent_id}/
```

Under home:

```text
config/
logs/
tmp/
memory/
workspace/
```

Recommended default current working directory is `{home}/workspace`, not the home root.

### 2.4 AgentWorkItem and AgentInteraction

- AgentWorkItem is a logical work item, not a process/run.
- Task assignment to an Agent creates an AgentWorkItem.
- Reassignment creates a new AgentWorkItem and marks the old one `superseded`.
- A Task keeps one stable Conversation across reassignments.
- Messages in the Task Conversation use `context_refs.work_item_ref` to distinguish attempts/segments.
- AgentWorkItem may have multiple AgentInteractions.
- A user reply in a waiting-input Task Conversation can trigger another AgentInteraction on the same AgentWorkItem.
- First version Message needs `work_item_ref`; `interaction_ref` can stay in AgentActivityEvent unless UI needs it.

AgentWorkItem status:

```text
queued -> active -> waiting_input -> active
active -> blocked
active -> done
active -> failed
queued/active/waiting_input/blocked -> canceled
queued/active/waiting_input/blocked -> superseded
```

### 2.5 Environment

- The bounded context name is **Environment**.
- Worker belongs to Environment.
- One machine may run multiple Workers.
- One Worker controls multiple Agents.
- Center to Worker control is worker-initiated long-lived stream or poll.
- Worker MCP exposes AppService tools only, not direct database access.
- AgentLauncher should be named **AgentController**.
- Worker internals:
  - ControlLoop
  - AgentController
  - MCPHost
  - FileTransferClient
  - LocalStore
- Worker.status is separate from Agent lifecycle.
- Agent availability is derived from Worker.status plus Agent lifecycle.

StopAgent:

- Operational stop must not automatically set active AgentWorkItem to blocked.
- Blocked remains explicit business state requiring message/reason.
- Stopped Agent makes active work appear paused/unavailable via Agent availability.

ResetAgent:

- Reset has scopes.
- Every reset requires second confirmation.

UpdateRuntimeConfig:

- Does not automatically restart.
- New config is applied on next restart.

### 2.6 ClaudeCode headless integration

The first Environment implementation targets headless long-lived ClaudeCode directly. The integration has **two channels, each with one job** (re-confirmed 2026-05-29):

**Channel 1 — local control channel: stream-json over stdin/stdout (drives the conversation)**

- AgentController launches `claude` headless and long-lived with `--input-format stream-json --output-format stream-json`; the process does not exit between turns.
- **stdin (in)**: the task brief and any human/Agent reply are injected as user messages. Conversation turns are delivered by stdin injection — not pulled.
- **stdout (out)**: assistant text, tool-use events, and results stream out as JSON lines. AgentController parses stdout into **AgentActivityEvent** (the observation stream).
- **stdout does NOT auto-post to Conversation.** The model's verbose intermediate output lives only in AgentActivityEvent. A human who wants detail looks at the activity stream.

**Channel 2 — business capability channel: MCP AppService tools (lets the model act on AgentCenter)**

- The Worker MCPHost exposes the §10 OQ4 tool set to the claude process via `--mcp-config`. Tool calls hit Center AppService (never the DB) and are bound by §10 OQ6 domain isolation.

**Human-visible communication is explicit-only.** Because stdout does not reach humans, anything a human must see is carried by an explicit message:

- `post_task_message` is the **only** way Agent text enters the Task Conversation.
- `request_input(question, ...)` posts `question` to the Task Conversation and sets the WorkItem `waiting_input`.
- `block_task(reason, ...)` posts `reason` to the Conversation (consistent with §2.2 requiring a message for `blocked`).
- `complete_task(summary?, ...)` may carry a delivery summary into the Conversation.

Net effect: the Task Conversation contains only what the Agent deliberately says plus state-change system messages; everything else sinks into AgentActivityEvent.

**Lifecycle and sessions**

- AgentController owns start/stop/restart/reset. The process is long-lived; per-interaction process spawn is **not** the default.
- A WorkItem may have multiple AgentInteractions = multiple turns within the same long-lived session.
- The wait→wake loop (see §10 OQ5): the model calls `request_input` → the turn ends and the process idles → a human/Agent reply is injected as a new user message on the same session's stdin → a new AgentInteraction resumes with full context.
- AgentInteraction is a logical turn. Environment may internally track the claude session id / process (AgentRuntimeSession), but the Agent bounded context exposes Agent lifecycle/status/activity, not ClaudeCodeSession as a business aggregate.

**Runtime**: cwd = `{home}/workspace` (§10 OQ7); memory under `{home}/memory/`; environment variables from the Agent profile.

ClaudeCode headless contract has been verified manually. v2.7 needs **contract tests** to protect against CLI drift, covering: start → inject task → receive output → `request_input` → call an MCP tool → stop/restart.

### 2.7 File URI, BlobStore, and FileTransfer

> Revised 2026-05-29 (see §10 OQ8). File storage is a **horizontal, generic module** decoupled from any business context. Identity is an opaque ULID; content is not hashed for addressing.

**Identity and addressing**

- Do not introduce `file_id` as business reference. Files are referenced by URI.
- File identity is an opaque, **server-generated ULID**. Stable File URI format:

```text
ac://files/{ulid}
```

- The File URI is returned at upload-session create time (the ULID is known immediately), so callers always have the final URI up front.
- Transfer URI format:

```text
ac://transfers/...
```

**BlobStore (storage layer)**

- File storage is a generic content store that knows nothing about Task/Issue/Agent/Conversation.
- Physical layout buckets by `hash(ulid)`, e.g. `~/.agent-center/files/objects/{h1}/{h2}/{ulid}`, where `{h1}{h2}` is a prefix of `hash(ulid)`. Hashing the ULID (whose prefix is time-ordered) avoids directory hot-spotting.
- A blob is write-once. Same-name uploads never overwrite — each upload gets a fresh ULID, so collisions are impossible by construction.
- No automatic content dedup in v1 (identical content uploaded twice = two ULIDs = two blobs). Acceptable for the single-user scope and recoverable later via the optional content checksum (below) without changing the URI scheme.
- Metadata stores a **content checksum (sha256)** for integrity verification and future opt-in dedup. The checksum does **not** participate in identity or addressing.
- First storage backend is the Center local filesystem; the resolver maps `ac://files/{ulid}` to the physical bucket path. The backend is replaceable without changing URIs.

**Placement (reference layer)**

- "Where a file is used" is expressed by **reference records**, not by the URI. A reference links `{scope, scope_id} -> file_uri`, where scope ∈ `task | issue | project | conversation | agent | tmp`.
- References are many-to-one: one blob can be referenced from any number of scopes. Sharing a file = adding references, never copying the blob.
- Filename, mime type, size, and display name live on the **reference/attachment metadata**, not on the blob identity. The same blob can appear under different display names in different places.

**Authorization (reachability)**

- URI is a reference, not authorization.
- Download/upload still go through Environment/FileTransfer AppService authorization.
- A caller may read a blob only if it can reach at least one reference to that blob that lies within the caller's Org/Project domain (reachability-based authorization, consistent with §2.1 / §10 OQ6 domain isolation).

**Retention and GC**

- Deleting a business object (Task/Issue/Project/Conversation or a message) **soft-deletes its references**, not the blob.
- A blob is physically removed by an **asynchronous GC** when its live-reference count reaches zero and a grace period has elapsed (default grace period: **7 days**).
- Cross-reference safety: a blob referenced from another Org/Project is not collected while any live reference remains.
- Orphan uploads (uploaded but never referenced; e.g. an abandoned transfer) are collected the same way: expired transfer session + zero references + past grace period.

### 2.8 Conversation

- Conversation remains independent bounded context.
- Conversation.kind:

```text
project_channel | issue | task | dm
```

- Conversation owner_ref uses URI:

```text
pm://projects/{project_id}
pm://issues/{issue_id}
pm://tasks/{task_id}
```

- Message attachments store one unified attachment structure with URI and metadata.
- Do not split file/image in domain structure.
- UI display type is derived from metadata.

A0 minimum attachment fields:

```text
uri: ac://files/...
filename
mime_type
size
```

Later metadata may add:

```text
media_kind
preview_uri
width
height
```

## 3. Boundary Model

### 3.1 ProjectManager aggregates

| Aggregate | Responsibility |
|---|---|
| Project | project identity, name, description, lifecycle |
| ProjectMember | members in a project; delivery scope candidate |
| Issue | project-scoped problem/discussion item and its state |
| Task | project-scoped unit of work and assignment state |
| TaskSubscriber | manual subscriber truth for Task |
| IssueSubscriber | manual subscriber truth for Issue |
| CodeRepoRef | repository reference attached to Project |

Task and Issue should not be stored inside Project as child arrays. Project is not a giant aggregate.

### 3.2 Agent aggregates and streams

| Aggregate/Stream | Responsibility |
|---|---|
| Agent | profile, skills, runtime config, lifecycle intent |
| AgentWorkItem | assignment queue item for an Agent |
| AgentActivityEvent | append-only activity/progress/status observation |

AgentWorkItem references Task by URI/ID but does not own Task state.

### 3.3 Environment aggregates and streams

| Aggregate/Stream | Responsibility |
|---|---|
| Worker | deployed always-on process on a machine |
| AgentRuntimeSession | internal/runtime record for Agent process/session, if needed |
| BlobStore | horizontal content store; blob identity = ULID, metadata holds content sha256 (see §2.7, §10 OQ8) |
| FileReference | placement record `{scope, scope_id} -> file_uri`; many references to one blob |
| FileTransferSession | upload/download session and transfer state |
| WorkerControlEvent | command/ack/status stream between Center and Worker |

AgentRuntimeSession is not exposed as a product aggregate unless the user needs to operate on it directly. BlobStore is a generic module that knows nothing about Task/Issue/Agent/Conversation; business contexts attach files only through FileReference and File URI.

### 3.4 Conversation aggregates

| Aggregate | Responsibility |
|---|---|
| Conversation | message stream metadata, kind, owner_ref |
| Message | content, sender, context_refs, attachments |
| ConversationParticipant | message-layer participation projection |
| ReadState | unread/read tracking per participant |

ConversationParticipant is not the business truth for Task/Issue subscription. It is a projection synced from ProjectManager for Task/Issue Conversations.

## 4. Cross-Context Flows

### 4.1 Create Task

Application Service: `ProjectManager.CreateTask`

Flow:

1. Validate actor is Organization member and Project member.
2. Create Task in ProjectManager.
3. Create TaskSubscriber defaults: creator, optional assignee.
4. Create Conversation with owner_ref `pm://tasks/{task_id}`.
5. Sync Conversation participants from effective subscribers.
6. Emit `task.created`.

Transaction rule:

- ProjectManager state and subscriber truth should commit atomically.
- Conversation create/participant sync should be same transaction if implementation keeps one database.
- If this crosses storage/process boundary later, use outbox with idempotent projection.

### 4.2 Assign or reassign Task to Agent

Application Service: `ProjectManager.AssignTask`

Flow:

1. Validate Task belongs to Project.
2. Validate target Agent is available for this Organization/Project.
3. If old assignee exists:
   - mark old AgentWorkItem `superseded`
   - remove old assignee from effective subscribers unless creator/manual subscriber
4. Update Task assignee and state.
5. Ensure current assignee is effective subscriber.
6. Create new AgentWorkItem in Agent bounded context.
7. Append system message to Task Conversation with reassignment details.
8. Sync Conversation participants.

Open design point:

- Same database implementation can do this in one transaction.
- Long-term cross-bounded-context purity may require Application Service orchestration plus outbox.

v2.7 recommendation:

- Use one Application Service orchestration with explicit transaction where current persistence allows it.
- Keep domain logic in separate packages so later outbox split is possible.

### 4.3 Agent asks for input

Flow:

1. AgentController receives model output indicating missing information or asks through MCP tool.
2. Environment records raw activity.
3. Agent AppService marks AgentWorkItem `waiting_input`.
4. ProjectManager does not automatically mark Task `blocked`.
5. Conversation receives Agent message with `context_refs.work_item_ref`.
6. User reply in same Conversation can trigger next AgentInteraction on that AgentWorkItem.

Task `blocked` only happens when the system/user explicitly calls ProjectManager BlockTask with a reason.

### 4.4 Stop Agent while work is active

Flow:

1. User calls Agent.StopAgent.
2. Agent lifecycle moves to stopped/stopping.
3. Environment receives control command and stops process/session.
4. AgentWorkItem remains active/waiting_input unless explicitly changed.
5. UI shows active work as paused/unavailable because Agent availability is unavailable.

No automatic blocked transition.

### 4.5 Upload file to Conversation

Flow:

1. Client requests Environment/FileTransfer create upload session.
2. FileTransfer returns `transfer_uri` and reserved `target_file_uri`.
3. Client uploads content.
4. FileTransfer marks upload completed and makes `target_file_uri` readable.
5. Conversation Message stores attachment metadata with `ac://files/...` URI.

Conversation does not store storage path or file_id.

## 10. Open Question Resolutions (2026-05-29)

All nine §7 open questions were closed in the design DM on 2026-05-29. These resolutions are binding inputs for the ADRs in §6.

### OQ1. Transaction boundary — introduce outbox now

- Cross-bounded-context side effects go through an **outbox**: the producing AppService writes its own BC state **and** an outbox event in one local transaction; an asynchronous, **idempotent** (dedup by event_id), replayable projector applies the cross-BC effect.
- BC-internal AR-cluster writes may stay in a single transaction (no outbox needed).
- **Three separate reliable channels**, not one table:
  1. **DB outbox** — cross-BC state projection.
  2. **Worker control flow** — Center↔Worker commands, with its own ack/idempotency (see OQ3).
  3. **Observation/SSE stream** — AgentActivityEvent fan-out.
- Operations that go through the outbox (cross-BC): CreateTask, CreateIssue, CreateProject, Add/RemoveProjectMember, Assign/ReassignTask, Subscribe/Unsubscribe Task/Issue, Block/Complete/Verify/Cancel/ReopenTask, Issue state transitions, "Agent asks for input", and "human reply wakes a waiting WorkItem".
- Operations that do **not**: CreateAgent, UpdateRuntimeConfig, plain Conversation messages (channel/dm), ReadState updates. EnqueueWorkItem / SupersedeWorkItem are **consumers** of OQ1 events, not separate producers.

### OQ2. Agent availability — 3-state derived signal

`Agent.availability` is derived, not stored, and is **not** a domain invariant (UI/dispatch only). Three values, evaluated top-down, first match wins:

| Condition (evaluated top-down) | availability |
|---|---|
| `Worker.status != online` | **unavailable** (highest priority — Worker offline overrides everything) |
| `Agent.lifecycle ∈ {stopped, stopping, resetting, error}` | **unavailable** |
| `Agent.lifecycle = running` and has an `active`/`waiting_input` WorkItem | **busy** |
| `Agent.lifecycle = running` and no active WorkItem | **available** |

The dispatcher only pushes new WorkItems to `available` Agents; `busy` does not block queuing. No `degraded`/`paused` modeling in v1.

### OQ3. Worker protocol — stream-first + offset ack + idempotency key

- Transport: **Worker-initiated long-lived bidirectional stream first**, poll as a degraded fallback. The protocol models commands as an **ordered, replayable stream** so reconnect resumes by offset and a future poll fallback is just a different transport over the same command log.
- Ack: **cumulative ack by stream offset** (Worker reports "processed through offset N"; Center resumes from N+1).
- Each command carries an **idempotency_key** so reconnect/redelivery never executes a destructive command (stop/reset) twice.

### OQ4. MCP tool surface — Agent capabilities (first version)

**Global guardrail (highest priority):** every Agent operation (read and write) is confined to the Agent's own Project/Organization domain; cross-domain access is rejected.

First-version AppService tools exposed to Agents via the Worker MCPHost (AppService only, never direct DB):

- Work/Task: `get_my_work`, `get_task` (own WorkItems only), `create_task`, `assign_task`/`reassign_task`, `subscribe`/`unsubscribe`, `post_task_message`, `request_input` (set `waiting_input`), `block_task` (requires reason), `complete_task`, `verify_task`.
- Issue: `get_issue` (read).
- Files: `upload_file`, `download_file`, `attach_file_to_message`.
- Optional: `report_activity` (most activity is captured automatically by Environment; this is a cuttable explicit supplement).

Iron rules: (1) all operations within the Agent's own Project/Org; (2) an Agent cannot verify a Task it completed itself; (3) Project/Member/Agent/Worker management stays human-only. Agent read scope is **strictly its own WorkItems** — when more context is needed, a human pastes it explicitly into the Conversation. This updates the §2.2 `verified` authorization rule.

### OQ5. User reply routing — system does not judge, the Agent decides

- The system performs **no semantic judgment** of whether a message is "an answer". When a WorkItem is `waiting_input`, the incoming message is delivered to the Agent as the response and wakes it (`waiting_input -> active`, new AgentInteraction).
- The Agent itself decides whether the input is sufficient; if not, it calls `request_input` again. No "discuss-only / don't wake" escape hatch is needed.
- (a) Input may come from **any participant except the waiting Agent itself** — humans and other Agents both count (so an orchestrator Agent can answer). The Agent's own messages never wake it (no self-loop).
- (b) Messages arriving while the WorkItem is `active` are **buffered** and delivered at the next `waiting_input`.
- (c) Multiple messages arriving while `waiting_input` are **merged into a single interaction's input**, not one wake per message.
- Waking is delivered via the OQ1 outbox (Conversation event → Agent BC), eventually consistent and idempotent.

### OQ6. ProjectMember — delivery scope + minimum write-gate, domain isolation only

- ProjectMember = project message **delivery scope** **and** the **minimum write-gate**: being a Project member is the precondition for any write in that Project. This is how the OQ4 "within own Project/Org" guardrail is enforced.
- **v1 has domain isolation, not role permissions.** No intra-Project role subdivision; all members have equal capability (except the human-only management plane). The basic read/write domain-isolation rules (an identity can only read/write inside its own Org/Project) are a hard boundary that must exist in v1.
- Being an Organization member is a prerequisite for becoming a member of a Project under that Organization.
- A complete permission model (Project admin/manager roles, creator mute, etc.) is roadmap.

### OQ7. Workspace default — `{home}/workspace`

Agent default current working directory is `{home}/workspace`, not the home root, so runtime support directories (`config/`, `logs/`, `tmp/`, `memory/`) stay physically separated from work files. (Roadmap: WorkspaceRef for `project_repo` / `managed_workspace` / `local_path` once Worker migration / cloud Agent arrives; this only sets the default when no WorkspaceRef is configured.)

### OQ8. File storage — opaque ULID identity, horizontal module

See §2.7 for the full model. Summary:

- File storage is a **horizontal, generic module** (BlobStore) decoupled from business context.
- Identity = opaque **server-generated ULID**; File URI = `ac://files/{ulid}`, returned at upload-session create.
- Physical bucketing by `hash(ulid)` to avoid timestamp hot-spotting; **no content hashing for addressing**.
- Filename/mime/size/display-name live on **reference/attachment metadata**; the same blob can be referenced from many scopes (identity ≠ placement).
- Authorization is **reachability-based** within the Org/Project domain.
- Metadata stores a **content sha256** for integrity verification and future opt-in dedup (not used for addressing). No automatic dedup in v1; recoverable later without changing URIs.

### OQ9. File retention — soft-delete references, async refcount GC

- Deleting a business object soft-deletes its **references**, not the blob.
- A blob is physically removed by an **asynchronous GC** when its live-reference count reaches zero and a grace period (default **7 days**) has elapsed.
- Cross-reference safety: a blob is not collected while any live reference (in any Org/Project) remains.
- Orphan uploads (never referenced) are collected the same way.

## 11. Non-goals

- No backward-compatible migration from v2.6 data.
- No AgentRun product concept.
- No Worker migration between machines in first version.
- No Agent degraded state in first version.
- No configurable work_dir in first version.
- No file_id business reference.
- No image/file split in Conversation attachment domain model.
- No automatic Task blocked inference from stopped Agent or Conversation text.
- No direct database tools exposed to Worker MCP.
