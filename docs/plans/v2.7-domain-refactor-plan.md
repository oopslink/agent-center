# v2.7 Domain Refactor Plan

> Branch: `design/v2.7-domain-refactor`
> Date: 2026-05-29
> Scope: ProjectManager + Agent + Environment + Conversation/File URI foundation

## 0. Position

v2.6 shipped Identity, Organization, Member, authentication, and Organization-scoped routes. v2.7 should use that foundation to correct the core domain model around project management and long-running agents.

This plan is not an implementation spec yet. It is the planning and boundary document for the next design pack. After review, the accepted parts should be split into tactical docs and ADRs before coding starts.

v2.7 does not need backward compatibility. Dropping and rebuilding tables is acceptable when it gives a cleaner domain model.

## 1. End State

AgentCenter should model four separate responsibilities:

| Responsibility | Bounded Context | Core truth |
|---|---|---|
| Work management | **ProjectManager** | Projects, project members, issues, tasks, task/issue subscription truth, task/issue state transitions |
| Agent product model | **Agent** | Agent profile, skills, runtime binding, lifecycle controls, work queue, activity stream |
| Runtime environment | **Environment** | Workers, AgentController, local process/session control, worker-initiated center connection, FileTransfer |
| Message stream | **Conversation** | Conversations, messages, participants as message-layer projection, read state, attachments by URI |

The important boundary is:

- ProjectManager decides what work exists and who should be notified.
- Agent decides which Agent is assigned what work and tracks the logical work item.
- Environment starts, monitors, and talks to local processes.
- Conversation carries human/agent communication but does not infer task or issue state.

## 2. Locked Decisions

### 2.1 ProjectManager

- The bounded context name is **ProjectManager / 项目管理域**.
- A Task must belong to exactly one Project.
- An Issue must belong to exactly one Project.
- There are no global Tasks and no cross-Project Tasks.
- A Task may be independent or derived from an Issue.
- Task and Issue each bind one stable Conversation.
- ProjectMember exists in the first version. Its first responsibility is "who is in this Project" and project message delivery scope, not fine-grained permission control.
- Subscription truth for Task/Issue lives in ProjectManager.
- Creator and current assignee are always effective subscribers.
- Manual subscribers are additive.
- Conversation participants mirror effective subscribers.
- On reassignment, the old assignee is removed from effective subscribers unless they are also creator or manual subscriber.
- Creator/current assignee cannot be removed from subscription while they still hold that role. Creator mute is future scope.

### 2.2 Task and Issue status

Issue states:

```text
open -> in_progress -> resolved -> closed
open/in_progress -> withdrawn
resolved/closed -> reopened -> open
```

`triaged` is final-shape only, not v2.7.

Task states:

```text
open -> assigned -> running -> blocked -> running
running -> completed -> verified
open/assigned/running/blocked -> canceled
completed/verified -> reopened -> open
assigned -> open  (unassign)
```

Rules:

- `blocked` is explicit business state and requires a Conversation message/reason.
- `completed` is set by assignee/agent.
- `verified` is set by creator or project manager role.
- Task/Issue state never changes by system inference from Conversation messages.

### 2.3 Agent

- Agent is an independent bounded context.
- Agent is logically long-running.
- There is no core **AgentRun** concept.
- Agent includes:
  - profile: name, description, model, CLI, environment variables
  - skill list
  - runtime config: selected Worker, runtime home, memory data
  - lifecycle controls: start, stop, restart, reset
  - observability: totals, heartbeat/status, activity events
- CreateAgent must choose a Worker.
- Agent runtime Worker binding is immutable in v2.7.
- Worker migration / cloud Agent entity is roadmap.
- Agent degraded state is out of the first version.

AgentRuntimeConfig first version:

- No configurable `work_dir`.
- Worker-local home:

```text
~/.agent-center/workers/{worker_id}/agents/{agent_id}/
```

Under home:

```text
config/
logs/
tmp/
memory/
workspace/
```

Recommended default current working directory is `{home}/workspace`, not the home root.

### 2.4 AgentWorkItem and AgentInteraction

- AgentWorkItem is a logical work item, not a process/run.
- Task assignment to an Agent creates an AgentWorkItem.
- Reassignment creates a new AgentWorkItem and marks the old one `superseded`.
- A Task keeps one stable Conversation across reassignments.
- Messages in the Task Conversation use `context_refs.work_item_ref` to distinguish attempts/segments.
- AgentWorkItem may have multiple AgentInteractions.
- A user reply in a waiting-input Task Conversation can trigger another AgentInteraction on the same AgentWorkItem.
- First version Message needs `work_item_ref`; `interaction_ref` can stay in AgentActivityEvent unless UI needs it.

AgentWorkItem status:

```text
queued -> active -> waiting_input -> active
active -> blocked
active -> done
active -> failed
queued/active/waiting_input/blocked -> canceled
queued/active/waiting_input/blocked -> superseded
```

### 2.5 Environment

- The bounded context name is **Environment**.
- Worker belongs to Environment.
- One machine may run multiple Workers.
- One Worker controls multiple Agents.
- Center to Worker control is worker-initiated long-lived stream or poll.
- Worker MCP exposes AppService tools only, not direct database access.
- AgentLauncher should be named **AgentController**.
- Worker internals:
  - ControlLoop
  - AgentController
  - MCPHost
  - FileTransferClient
  - LocalStore
- Worker.status is separate from Agent lifecycle.
- Agent availability is derived from Worker.status plus Agent lifecycle.

StopAgent:

- Operational stop must not automatically set active AgentWorkItem to blocked.
- Blocked remains explicit business state requiring message/reason.
- Stopped Agent makes active work appear paused/unavailable via Agent availability.

ResetAgent:

- Reset has scopes.
- Every reset requires second confirmation.

UpdateRuntimeConfig:

- Does not automatically restart.
- New config is applied on next restart.

### 2.6 ClaudeCode headless integration

The first Environment implementation should target headless long-lived ClaudeCode directly:

- local control channel: stream-json stdin/stdout
- business capability channel: MCP AppService tools
- Worker daemon / AgentController owns lifecycle

Per-interaction process is not the default.

AgentInteraction is a logical turn. Environment may internally track RuntimeSession or AgentProcess, but Agent bounded context exposes Agent lifecycle/status/activity, not ClaudeCodeSession as a business aggregate.

ClaudeCode headless contract has been verified manually, but v2.7 needs contract tests to protect against CLI drift.

### 2.7 File URI and FileTransfer

- Do not introduce `file_id` as business reference.
- Files are referenced by URI.
- Stable File URI format:

```text
ac://files/...
```

- Transfer URI format:

```text
ac://transfers/...
```

- Same-name uploads never overwrite; they generate collision-safe new URIs.
- `target_file_uri` can be returned when upload session is created but is unreadable until completed.
- First storage backend is Center local filesystem, resolving URI to `~/.agent-center/files/...`.
- URI is a reference, not authorization. Upload/download still go through Environment/FileTransfer AppService authorization.

### 2.8 Conversation

- Conversation remains independent bounded context.
- Conversation.kind:

```text
project_channel | issue | task | dm
```

- Conversation owner_ref uses URI:

```text
pm://projects/{project_id}
pm://issues/{issue_id}
pm://tasks/{task_id}
```

- Message attachments store one unified attachment structure with URI and metadata.
- Do not split file/image in domain structure.
- UI display type is derived from metadata.

A0 minimum attachment fields:

```text
uri: ac://files/...
filename
mime_type
size
```

Later metadata may add:

```text
media_kind
preview_uri
width
height
```

## 3. Boundary Model

### 3.1 ProjectManager aggregates

| Aggregate | Responsibility |
|---|---|
| Project | project identity, name, description, lifecycle |
| ProjectMember | members in a project; delivery scope candidate |
| Issue | project-scoped problem/discussion item and its state |
| Task | project-scoped unit of work and assignment state |
| TaskSubscriber | manual subscriber truth for Task |
| IssueSubscriber | manual subscriber truth for Issue |
| CodeRepoRef | repository reference attached to Project |

Task and Issue should not be stored inside Project as child arrays. Project is not a giant aggregate.

### 3.2 Agent aggregates and streams

| Aggregate/Stream | Responsibility |
|---|---|
| Agent | profile, skills, runtime config, lifecycle intent |
| AgentWorkItem | assignment queue item for an Agent |
| AgentActivityEvent | append-only activity/progress/status observation |

AgentWorkItem references Task by URI/ID but does not own Task state.

### 3.3 Environment aggregates and streams

| Aggregate/Stream | Responsibility |
|---|---|
| Worker | deployed always-on process on a machine |
| AgentRuntimeSession | internal/runtime record for Agent process/session, if needed |
| FileTransferSession | upload/download session and transfer state |
| WorkerControlEvent | command/ack/status stream between Center and Worker |

AgentRuntimeSession is not exposed as a product aggregate unless the user needs to operate on it directly.

### 3.4 Conversation aggregates

| Aggregate | Responsibility |
|---|---|
| Conversation | message stream metadata, kind, owner_ref |
| Message | content, sender, context_refs, attachments |
| ConversationParticipant | message-layer participation projection |
| ReadState | unread/read tracking per participant |

ConversationParticipant is not the business truth for Task/Issue subscription. It is a projection synced from ProjectManager for Task/Issue Conversations.

## 4. Cross-Context Flows

### 4.1 Create Task

Application Service: `ProjectManager.CreateTask`

Flow:

1. Validate actor is Organization member and Project member.
2. Create Task in ProjectManager.
3. Create TaskSubscriber defaults: creator, optional assignee.
4. Create Conversation with owner_ref `pm://tasks/{task_id}`.
5. Sync Conversation participants from effective subscribers.
6. Emit `task.created`.

Transaction rule:

- ProjectManager state and subscriber truth should commit atomically.
- Conversation create/participant sync should be same transaction if implementation keeps one database.
- If this crosses storage/process boundary later, use outbox with idempotent projection.

### 4.2 Assign or reassign Task to Agent

Application Service: `ProjectManager.AssignTask`

Flow:

1. Validate Task belongs to Project.
2. Validate target Agent is available for this Organization/Project.
3. If old assignee exists:
   - mark old AgentWorkItem `superseded`
   - remove old assignee from effective subscribers unless creator/manual subscriber
4. Update Task assignee and state.
5. Ensure current assignee is effective subscriber.
6. Create new AgentWorkItem in Agent bounded context.
7. Append system message to Task Conversation with reassignment details.
8. Sync Conversation participants.

Open design point:

- Same database implementation can do this in one transaction.
- Long-term cross-bounded-context purity may require Application Service orchestration plus outbox.

v2.7 recommendation:

- Use one Application Service orchestration with explicit transaction where current persistence allows it.
- Keep domain logic in separate packages so later outbox split is possible.

### 4.3 Agent asks for input

Flow:

1. AgentController receives model output indicating missing information or asks through MCP tool.
2. Environment records raw activity.
3. Agent AppService marks AgentWorkItem `waiting_input`.
4. ProjectManager does not automatically mark Task `blocked`.
5. Conversation receives Agent message with `context_refs.work_item_ref`.
6. User reply in same Conversation can trigger next AgentInteraction on that AgentWorkItem.

Task `blocked` only happens when the system/user explicitly calls ProjectManager BlockTask with a reason.

### 4.4 Stop Agent while work is active

Flow:

1. User calls Agent.StopAgent.
2. Agent lifecycle moves to stopped/stopping.
3. Environment receives control command and stops process/session.
4. AgentWorkItem remains active/waiting_input unless explicitly changed.
5. UI shows active work as paused/unavailable because Agent availability is unavailable.

No automatic blocked transition.

### 4.5 Upload file to Conversation

Flow:

1. Client requests Environment/FileTransfer create upload session.
2. FileTransfer returns `transfer_uri` and reserved `target_file_uri`.
3. Client uploads content.
4. FileTransfer marks upload completed and makes `target_file_uri` readable.
5. Conversation Message stores attachment metadata with `ac://files/...` URI.

Conversation does not store storage path or file_id.

## 5. Development Order

The accepted order is:

```text
A0 -> B -> C -> D -> E
```

### A0. Conversation and File URI foundation

Goal: introduce the minimum shared primitives needed by later ProjectManager, Agent, and Environment work.

Deliverables:

1. `owner_ref` URI on Conversation.
2. Conversation kinds: `project_channel`, `issue`, `task`, `dm`.
3. Message `context_refs` with at least `work_item_ref`, `task_ref`, `agent_ref`.
4. MessageAttachment stores URI + metadata, no file/image domain split.
5. FileURI value object.
6. FileURI resolver skeleton for `ac://files/...`.
7. Migration/drop-and-recreate plan.

Not in A0:

- upload/download sessions
- chunk transfer
- Worker file bridge
- preview generation

Acceptance:

- Existing channels/messages still work after schema rebuild.
- Task/Issue Conversation can be represented by owner_ref URI.
- Message can carry work_item_ref without UI breakage.
- Attachment model can store image/file metadata uniformly.

### B. ProjectManager bounded context

Goal: replace scattered project/issue/task behavior with ProjectManager as the work management truth.

Deliverables:

1. Project aggregate.
2. ProjectMember aggregate.
3. Issue aggregate and state machine.
4. Task aggregate and state machine.
5. TaskSubscriber and IssueSubscriber.
6. CodeRepoRef.
7. Application Services:
   - CreateProject
   - AddProjectMember
   - CreateIssue
   - CreateTask
   - AssignTask
   - ReassignTask
   - Subscribe/Unsubscribe Task/Issue
   - BlockTask
   - CompleteTask
   - VerifyTask
8. Conversation participant sync from subscriber truth.

Acceptance:

- No global Task can be created.
- Cross-Project Task/Issue mutation is rejected.
- Reassignment creates correct subscriber/participant diff.
- Creator/current assignee cannot be unsubscribed while role remains.
- Task/Issue state transitions enforce the accepted state machines.

### C. Agent bounded context

Goal: model long-running Agents and their logical work queue.

Deliverables:

1. Agent aggregate with profile, skills, runtime config, lifecycle.
2. AgentWorkItem aggregate.
3. AgentActivityEvent stream.
4. AgentInteraction logical turn model.
5. Application Services:
   - CreateAgent (requires Worker)
   - StartAgent
   - StopAgent
   - RestartAgent
   - ResetAgent
   - UpdateRuntimeConfig
   - EnqueueWorkItem
   - MarkWaitingInput
   - CompleteWorkItem
   - SupersedeWorkItem

Acceptance:

- AgentRun does not exist in product/domain API.
- StopAgent does not set AgentWorkItem blocked.
- Worker binding is immutable.
- Reset requires explicit scope and confirmation.
- WorkItem messages can be grouped by `work_item_ref`.

### D. Environment and FileTransfer

Goal: connect Workers, AgentController, headless ClaudeCode, MCP AppService tools, and file transfer mechanics.

Deliverables:

1. Worker protocol: command envelope, ack, retry, idempotency key, status report.
2. Worker ControlLoop.
3. AgentController:
   - start/stop/restart/reset process
   - stream-json stdin/stdout control
   - status and heartbeat mapping
4. MCPHost exposing AppService tools only.
5. FileTransferSession:
   - create upload/download session
   - complete/cancel/expire
   - local filesystem backend
6. ClaudeCode headless contract tests.

Acceptance:

- One Worker controls multiple Agents.
- Center command delivery survives reconnect without duplicate destructive action.
- ClaudeCode headless contract test covers start, inject task, receive output, request input, call MCP, stop/restart.
- File upload returns URI and Conversation can attach it after completion.

### E. Integration, UI, and release

Goal: provide an end-to-end user workflow across all four bounded contexts.

Deliverables:

1. Project pages:
   - Project members
   - Issues
   - Tasks
   - Code repository refs
2. Agent pages:
   - create Agent
   - lifecycle controls
   - work queue
   - activity stream
3. Environment pages:
   - Workers
   - Agents controlled by Worker
   - transfer sessions
4. Conversation UI:
   - task/issue Conversation owner banner
   - work item grouping
   - attachment display by metadata
5. Acceptance plan.
6. Migration/release notes.

Acceptance:

- User creates Project -> Task -> assign Agent -> Agent asks input -> user replies -> Agent continues -> completes -> creator verifies.
- User stops Agent during active work without accidental blocked state.
- User uploads attachment and Agent can reference it by URI.
- Reassignment creates a new work segment in the same Task Conversation.

## 7. Open Questions

These must be closed before implementation tasks are created:

1. **Transaction boundary**: Do we keep ProjectManager + AgentWorkItem + Conversation participant sync in one SQLite transaction for v2.7, accepting cross-package orchestration, or introduce an outbox now?
2. **Agent availability rules**: Exact derived state matrix from Worker.status + Agent.lifecycle + AgentWorkItem.status.
3. **Worker protocol**: stream vs poll first, and whether command ack is per command or per stream offset.
4. **MCP tool surface**: exact AppService tools exposed to Agents in first version.
5. **User reply routing**: how to decide a human message should wake a waiting AgentWorkItem versus remain ordinary discussion.
6. **ProjectMember permission**: first version has no fine-grained Project permission; confirm whether ProjectMember is delivery scope only or also a minimum write-access gate.
7. **Workspace default**: confirm Agent cwd = `{home}/workspace`, not home root.
8. **File URI namespace**: exact path structure under `ac://files/...`.
9. **File retention**: whether deleting a Conversation/Task/Project soft-deletes file references or only removes reachability.

## 8. Suggested Task Split

Create one umbrella and phase tasks only after open questions are closed.

Umbrella:

```text
v2.7 Domain refactor: ProjectManager + Agent + Environment
```

Phase tasks:

1. `v2.7-A0 Conversation owner_ref + context_refs + FileURI foundation`
2. `v2.7-B1 ProjectManager ARs and repositories`
3. `v2.7-B2 ProjectManager AppServices and subscriber sync`
4. `v2.7-B3 ProjectManager HTTP/UI migration`
5. `v2.7-C1 Agent AR + lifecycle`
6. `v2.7-C2 AgentWorkItem + AgentActivityEvent`
7. `v2.7-C3 Agent UI`
8. `v2.7-D1 Environment Worker protocol`
9. `v2.7-D2 AgentController + ClaudeCode headless contract`
10. `v2.7-D3 FileTransferSession + local backend`
11. `v2.7-E1 End-to-end UI integration`
12. `v2.7-E2 Acceptance, docs, release`

Do not run B/C/D in parallel until A0 is accepted. Within each phase, implementation tasks may be parallelized only where they do not share schema or public service contracts.

## 9. Acceptance Gate

Before ship:

- `go build ./...`
- `go test ./internal/... ./tests/integration/...`
- `go test ./tests/e2e/... -count=1`
- `make lint`
- `make build`
- `npx vitest run`
- manual PM walkthrough of E acceptance flow

The PM walkthrough must include:

1. Create Project.
2. Add ProjectMember.
3. Create Task.
4. Assign Agent.
5. AgentWorkItem created.
6. Task Conversation shows work item segment.
7. Agent asks for input.
8. User reply resumes same WorkItem.
9. Agent completes.
10. Creator verifies.
11. Reassign same Task to another Agent.
12. New WorkItem segment appears in same Conversation.
13. File attachment URI survives display/download path.

## 10. Non-goals

- No backward-compatible migration from v2.6 data.
- No AgentRun product concept.
- No Worker migration between machines in first version.
- No Agent degraded state in first version.
- No configurable work_dir in first version.
- No file_id business reference.
- No image/file split in Conversation attachment domain model.
- No automatic Task blocked inference from stopped Agent or Conversation text.
- No direct database tools exposed to Worker MCP.
