## Group A — Shipped but incomplete (real product gaps, users hit these) — **P0**

Owner pattern: **Dev backend (real fields + migration + write-path) + Dev2 FE
(render real values)**. Dev confirmed accuracy from the implementation side;
Tester confirmed none were in the v2.8.1-verified backend (= they are v2.8
planned-but-unbuilt, not regressions).

### A1. AgentDetail panel shows placeholder fake data — `#229` `#230` `#231` `#235`
v2.7.1 #228 shipped the AgentDetail UI shell with **static fallback values
presented as real data** — users click in and see fake values:
- `#229` Profile `reasoning_level/mode/provider` hard-coded `Medium/Default/Default`
  → need real values + data-source identification (claude config? worker?).
- `#230` Skills show name only, **no global/local badge, no path** → worker must
  report skill source (`~/.claude/skills/` vs project-local) + DTO carries
  indicator + path.
- `#231` WorkItem `type` is a frontend constant (`task` fallback), `priority`
  shows `—` → add real `type` (bug/task/test) + `priority` (high/...) schema
  field (pm BC or work-item BC) + migration + DTO.
- `#235` AgentDetail WorkItems has no `+New` button (no manual create endpoint)
  → add: `+New` opens Create-Task modal scoped to current project + auto-assign
  this agent.
**Owner**: Dev (agentMap/skill/work-item DTO + fields + migration) → Dev2 (render
real values + #235 +New).

### A2. Message work-item tag never renders — `#221` `#223`
The work-item tag UI shipped in v2.7.1 (#219) but the write-path was never wired,
so **real messages never link to a work-item and the tag is effectively never
shown** (only set in tests):
- `#223` wire `context_refs.work_item_ref` write path — `AddMessageCommand` has
  no `ContextRefs` field; conversation/service has no write path. Agent-sent
  messages in a work-item context must set `message.context_refs.work_item_ref`.
- `#221` add `task_title` to message `context_refs` DTO (read-time, same pattern
  as #206) so the tag shows the real task title instead of a generic "Work item"
  label.
**Owner**: Dev (write path + DTO) → FE existing.

### A3. `cli` field is inert / no multi-CLI adapter — `#180`
All agents run `claude-code` regardless of the `agent.cli` field (agent
supervisor hard-codes claude argv). Need: select adapter by `agent.cli`
(codex/opencode/claude-code), build argv + dispatch per adapter.
**Owner**: Dev backend.

## Group B — Correctness bugs (not polish) — **P0**

### B1. alpha-tint-over-token AA failure, repo-wide — Dev2 FE
`bg-{token}/{opacity}` over a hex CSS-var token renders near-transparent → same
token `text-*` fails AA contrast. **Hit 4 times in the v2.8.1 cycle**
(RoleBadge `bg-warning/10` #244 / Bot-badge `bg-accent/10` #248 / "(deleted)"
#247 / MembershipStatus `bg-success/10` #259) + remaining `Agents.tsx:199`
`hover:bg-danger/10`. Each time a code comment falsely claimed "not alpha-tint"
and human review (PD + Dev2) copied it — human eyes do not catch this. Fix:
sweep all to solid `X-100/X-800` pairs. (Source-prevention = B-guardrail B6.)

### B2. link-accent + muted-text AA sweep — Dev2 FE
`text-accent` = `#3b82f6` (light) as link text on theme backgrounds (white/subtle)
≈ 3.68 < 4.5. Repo-wide markdown-link / project-link / mention-on-theme-bg use
it. Sweep with B1. (Own `#D1E3FF` bubble link/mention already uses `text-blue-700`
= 5.15 via #239/#258 — not in scope.)
**Also: `text-text-muted` = `#94a3b8` light = 2.57 < 4.5** (the same slate-400 token
as the v2.8.1 "(deleted)" finding). Used app-wide for de-emphasized labels,
incl. the new v2.9 Plan FE (progress / metadata dl / empty states / table
headers / id columns). **Sweep must include the new Plan pages.** Convention
question for the team: is muted acceptable for decorative-only text, or should
readable labels move to `text-text-secondary` (as the (deleted) fix did)?

### B3. org-routing explicit (eliminate implicit org state) — Dev2 FE + Dev backend
Two layers (same class as the v2.8.1 agent-list orphan + `useMembers` empty +
`/api/members` implicit-org concern):
- **Backend**: org-scoped APIs currently use implicit org context
  (session / `?org_slug`). Refactor to explicit path `/api/orgs/{slug}/...`
  (convention `conventions.md §20` already established).
- **FE-nav**: `⌘1-5` / `⌘7` + nav links + post-create redirects use bare paths
  without org-slug → org-scoped fetch/nav breaks. Prefix all with org-slug.
**Verification pairs with C4 (page-sweep).**

## Group C — Quality protection (prevent the next redesign from re-breaking) — **P0**

### C4. Full-page reachability + org-routing sweep (one combined pass) — Tester2 + Tester
The #256 agent-list orphan was **two latent classes at once**: (1) nav pointed at
the legacy page (reachability), (2) the enhanced page / `useMembers` needed the
org-slug-context (org-routing). v2.8.1 only added reachability auto-assertions
for the **agents page**; workers/members/projects/issues/tasks etc. are
unverified — the same orphan/dup-route + empty-data risk may be latent there.
Do **one page-sweep** over all list/detail pages, attaching both assertions per
page (avoid two rounds over the same pages):
- **Reachability (Tester2)**: real user nav (nav/⌘/breadcrumb) lands on the
  correct canonical page (not an orphan/legacy), + retired routes redirect.
  Write assertions into `App.test.tsx`.
- **org-routing data (Tester)**: each org-scoped fetch (`/api/members`,
  `/api/tasks`, `/api/agents`, `/api/issues`, ...) returns correct data only with
  the org-slug path (no bypass, no cross-org leak).

### C5. eslint/a11y guardrails (source-level prevention) — Dev2
- **alpha-tint guardrail**: lint-ban `bg-{token}/{opacity}` + same-token `text-*`
  (B1 keeps recurring; human review proven unreliable — must enforce at source).
- **panel-actions-icon guardrail**: "panel action buttons must use an
  icon-component" (prevents the #271-class icon→text regression).

### C6. #218 raw-error class-sweep (use the new ErrorState component) — Dev2 FE
v2.9 #261 added a reusable **`ErrorState`** component (friendly `role=alert`
message + raw error hidden behind `[Details]`, per rule #218). A sub-agent
flagged `ProjectDetail` / `Channels` still render raw API errors the old way.
Sweep those (and any other raw-error sites) onto `ErrorState` — app-wide #218
compliance.
