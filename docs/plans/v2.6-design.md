## § 0. 文档约定与方法论

### § 0.1 讨论方法论：最终形态优先

任何特性、子系统、流程的讨论必须**先描述最终形态（end state）**，再决定 v2.6 在该维度的裁剪。任何"v2.6 不做 X"必须挂在最终形态的 X 上明示理由（[[feedback-end-state-first]]）。

### § 0.2 统一语言权威

本文档与 [strategic/03-bounded-contexts § 1](../design/architecture/strategic/03-bounded-contexts.md) 共同构成 v2.6 后通用语言。**任何新名词进文档前必先入 § 2**。

### § 0.3 AR / VO / Entity 选型自检

每个新引入的领域对象按以下顺序自检：

1. 它有独立生命周期吗？→ 是 → AR 或 Entity
2. 它有跨聚合一致性要求自己守门吗？→ 是 → AR
3. 它通过 ID 引用其他对象，其他对象通过它的 Root.id 引用它吗？→ 是 → AR
4. 否则它是 VO（不可变 + 值语义）或子 Entity（隶属某 AR）

### § 0.4 AppService 入口约束

所有持久化必走 BC 的 Application Service；任何 HTTP / CLI / SSE transport 不可直接调 Repository。

### § 0.5 观察 + 能力两层

本文档讨论 v2.6 的设计意图与 schema 形态；具体代码实现层（Repository SQL、HTTP handler 框架）下沉到 implementation 阶段。

### § 0.6 向后兼容

v2.6 不做向后兼容（[[no-backward-compat]]）。schema 走 drop-and-recreate；旧数据不迁移；现有用户清库重装。

## § 1. 背景与目标

### § 1.1 当前形态（v2.5 GA 后）

- Identity AR 在 **Conversation BC**（[ADR-0033](../design/decisions/0033-identity-model-refactor.md)），3 kinds：user / agent / system
- 无 Organization 概念（[ADR-0033 Context § 用户表态](../design/decisions/0033-identity-model-refactor.md)：「V2 不做多租户 也不做组织这个概念」）
- 无 Auth（cookie 都没有，全靠 hostname / localhost 信任）
- 无 Member、无 Role
- 无 UI 管理：Identity / Project / Agent / Secret 都靠 CLI，Web 上不可见
- Supervisor 是 built-in AgentInstance（[ADR-0029](../design/decisions/0029-supervisor-as-builtin-agent-instance.md)），但概念上跟 worker agent 半重叠

### § 1.2 v2.6 想达成什么

1. **Identity BC 独立**：把 Identity AR 从 Conversation BC 抽出，提升为独立 BC（BC9 Identity）
2. **Organization 概念引入**：显式多租户聚合根；多个 organization 并发存在；走 URL slug 路由
3. **Member 概念引入**：Identity ↔ Organization 的归属关系，承载 role
4. **Auth 引入**：passcode + 签名 cookie，自助注册 + 登录 + 登出
5. **Members / Organization / Agent / Project / Secret Management UI**：所有顶级实体在 web 上可管
6. **为 v2.7+ SaaS 演进铺基**：schema 一次到位（顶级实体全带 organization_id），后续不再做大规模迁移

### § 1.3 v2.6 不做（最终形态对照延后表）

| 维度 | 最终形态 | v2.6 | 延后到 |
|---|---|---|---|
| Identity 创建入口 | 自助 / 邀请 / SSO / Agent 多路径 | 自助 + Agent 走 mint-enroll | 邀请 v2.7；SSO v3+ |
| Signin | Email+Pass / Magic Link / SSO / 2FA | display_name + 6 位 passcode | 其它 v2.7+ |
| Session | JWT + refresh + remote logout + timeout | JWT httpOnly cookie，7 day fixed exp | refresh / remote logout v2.7+ |
| Credential 生命周期 | hash + reset + rotation + audit | argon2 hash | reset v2.7（要求 email） |
| API Token | PAT + scope + revoke | agent 继续走 worker-token + cert pinning | PAT v2.7+ |
| Verification | email / phone / KYC | 不做 | v2.7+ |
| Recovery | password reset / 2FA recovery | 文档说明清库 | v2.7（要求 email） |
| Permission | RBAC + ABAC + audit | 基础 RBAC（owner/admin/member）| ABAC v3+ |
| 多 Identity 浏览器 | account switcher / per-tab | /signin 切换 | switcher v2.7+ |
| 审计合规 | login log / lockout / IP | Domain Event 已有 | failed lockout v2.7+ |
| Supervisor 概念 | （目前是 built-in AgentInstance）| **砍 in v2.6**（agent 自定义即可） | — |
| 邀请流程 UI | 邀请 link + email 投递 | Invitation 表 schema 占位，UI flow 不做 | v2.7 |

## § 2. 统一语言（Ubiquitous Language）—— v2.6 增量

> 本节是 v2.6 后 [strategic/03-bounded-contexts § 1](../design/architecture/strategic/03-bounded-contexts.md#-1-通用语言ubiquitous-language) 的待合并增量。

### § 2.1 新引入术语

| 术语 | 定义 | 出处 / 落地 |
|---|---|---|
| **Identity** | 全局唯一的身份记录；表示"某个 actor"在系统中的标识。一个 Identity 不属于任何 Organization，跨 Organization 复用。 | Identity AR |
| **Identity Kind** | Identity 的种类，枚举：`user`（人类）/ `agent`（程序）。v2.6 不再有 `system` kind。 | Identity AR |
| **Organization** (Organization) | 多租户工作空间；自治的资源容器。所有顶级业务实体（Channel / Project / Worker / Agent / Secret）属于某个 Organization。 | Organization AR |
| **Organization Slug** | Organization 的 URL key，唯一、小写、`[a-z0-9-]{3,40}`，由创建者输入；用于 `/organizations/{slug}/...` 路由。 | Organization AR |
| **Member** | 某个 Identity 在某个 Organization 内的成员关系。一个 Identity 可同时是多个 Organization 的 Member；同一 Organization 内 (organization_id, identity_id) 唯一。 | Member AR |
| **Member Role** | Member 在该 Organization 内的权限级别，枚举：`owner` / `admin` / `member`。 | Member.role |
| **Member Status** | Member 在该 Organization 内的状态，枚举：`joined`（活跃）/ `disabled`（已禁用，仍保留历史）。区别于 Identity 的 `account_status`。 | Member.status |
| **Account Status** | Identity 自身的全局开关，枚举：`active` / `disabled`。disabled 时所有 Member 关系视作连带失效。 | Identity.account_status |
| **Passcode** | Identity（kind=user）的认证凭据，6 位数字；以 argon2 hash 存储。 | Identity.passcode_hash |
| **Invitation** | Identity 加入某个 Organization 的邀请记录。v2.6 仅建表，流程 v2.7。 | Invitation AR |
| **Signup** | 自助创建 Identity + 第一个 Organization + Member 的入口流程。 | Application Service |
| **Signin** | 凭 display_name + passcode 恢复会话。 | Application Service |
| **Organization-scoped** | 属于某 Organization 的实体（直接带 `organization_id` 列 或 通过 parent 间接归属）。 | Schema 约束 |

### § 2.2 修订术语

| 术语 | 旧定义（v2.5） | v2.6 后含义 |
|---|---|---|
| **Identity ID 格式** | `kind:id` 形式（`user:hayang`、`agent:01HE...`、`system`）| `user-<8hex>` / `agent-<8hex>`（worker-style）；无 `system` 单例 |
| **Identity Kind** | user \| agent \| system | user \| agent（删 system） |
| **AgentInstance.id** | ULID 全局唯一 | 仍 ULID，但与 Identity[kind=agent] 通过 `agent_instance.identity_id` 显式 FK 关联（不再用 id 派生） |

### § 2.3 退役术语

| 术语 | 处置 |
|---|---|
| `system` kind | 删除；纯系统动作 actor=NULL |
| `kind:id` ID format | 取代为 worker-style id |
| Supervisor as built-in AgentInstance | 砍 in v2.6；agent 自定义即可（详 § 11.3） |
| ChannelBinding（已 v2 删） | 不复活；vendor 重新接入时另起命名 |

## § 3. 上下文边界变化（Bounded Context Landscape）

### § 3.1 v2.5 → v2.6 BC 对比

```
v2.5 GA (7 BC + BC8):
  BC1 TaskRuntime
  BC2 Discussion
  BC3 Workforce
  BC4 Cognition (含 Supervisor)
  BC5 Observability
  BC6 Conversation (含 Identity AR)
  BC8 SecretManagement

v2.6 (7 BC + BC8 + BC9):
  BC1 TaskRuntime
  BC2 Discussion
  BC3 Workforce
  BC4 Cognition          ← Supervisor 砍（详 § 11.3）；剩 Memory 单 AR
  BC5 Observability
  BC6 Conversation       ← Identity AR 抽出；Conversation 内部不再持身份语义
  BC8 SecretManagement
  BC9 Identity           ← 新：Identity / Organization / Member / Invitation 四 AR
```

### § 3.2 Identity BC 抽出（Carve-out）

**动机**：

1. [ADR-0033](../design/decisions/0033-identity-model-refactor.md) 当时把 Identity 留在 Conversation BC 是因为 Identity 主要用作 Message.sender；v2.6 引入 Organization / Member / Auth 后，Identity 自身的语义远超 Conversation 域
2. v2.6 把 Identity 当 actor + permission 挂载点，跟 Conversation BC 强耦合违反 SRP
3. Identity BC 独立后，Auth 流程、Member 管理、Organization 切换 都聚合在一个 BC 内，跨 BC ACL 简单

**抽出后 Conversation BC 的变化**：

- Identity AR 删，迁至 BC9 Identity
- Message.sender_identity_id 从 BC 内引用变 BC 间 ID 引用
- Conversation BC 不再 INSERT / UPDATE Identity 行；只读 Identity 用作 author 显示
- 历史的 `conversation/02-identity.md` 文档归档；新 `identity/00-overview.md` 替代

### § 3.3 BC Context Map（v2.6 新增上下游）

```
                    ┌───────────────┐
                    │  Identity BC  │← 所有需要 actor / organization 上下文的 BC
                    └───────┬───────┘
                            │ (Customer-Supplier; ID 引用 + 软约束)
        ┌───────────────────┼───────────────────┐
        ▼                   ▼                   ▼
  Conversation BC      Workforce BC        TaskRuntime BC
  (Message.sender)     (Worker/Agent.organization)  (Task.actor)
        │                   │                   │
        ▼                   ▼                   ▼
                  其它 BC 同理（Discussion / Cognition / Observability / SecretManagement）
```

- **关系类型**：Identity BC 是上游（Supplier），其它 BC 是下游（Customer）
- **Anti-Corruption Layer**：暂不立独立 ACL；通过 published `identity.id` / `organization.id` / `member.id` 走 ID 引用
- **一致性窗口**：弱关联，Identity 注销不立即级联清理（参 [conventions § 0.3 弱关联]）

### § 3.4 与 v2.5 ADR 的冲突解决

| 旧 ADR | v2.6 处置 |
|---|---|
| [ADR-0033 Identity 模型重构](../design/decisions/0033-identity-model-refactor.md) | **Superseded by v2.6 ADR-0040**（新 Identity BC + 新 id 格式 + 删 system kind） |
| [ADR-0029 Supervisor as Built-in AgentInstance](../design/decisions/0029-supervisor-as-builtin-agent-instance.md) | **Superseded by ADR-0044**（Supervisor 砍 in v2.6） |
| [ADR-0013 Supervisor 并发模型](../design/decisions/0013-supervisor-invocation-concurrency.md) | **Superseded by ADR-0044** |
| [ADR-0011 Dispatch 可靠性协议](../design/decisions/0011-dispatch-reliability-protocol.md) | Re-examine in v2.6（如涉及 Supervisor） |

## § 4. Identity BC 设计

### § 4.1 BC 概述

| 属性 | 值 |
|---|---|
| BC 序号 | BC9 |
| 中文名 | 身份上下文 |
| 子域分类 | **Supporting** —— 业务支撑；不直接产生用户价值但所有 BC 依赖 |
| 上下游 | 上游所有其它 BC；下游唯有 Observability（事件订阅） |
| 聚合数 | 4（Identity / Organization / Member / Invitation） |
| 持久化 | SQLite 主库；与现有 BC 同库 |

### § 4.2 聚合根（Aggregate Roots）

#### § 4.2.1 Identity AR

```
Identity {
  id              string PK   // "user-<8hex>" | "agent-<8hex>"
  kind            enum        // user | agent
  display_name    string      // "Hayang" / "AgentCenterPD"
  description     string?     // 自由文本
  account_status  enum        // active | disabled
  passcode_hash   string?     // argon2 hash; user 必填，agent 为 NULL
  passcode_set_at timestamp?  // user 必填
  created_at      timestamp
  updated_at      timestamp
}

INDEXES:
- UNIQUE (id)
- UNIQUE (display_name) WHERE kind='user'  -- 防 signin 歧义
- INDEX (kind, account_status)
```

**Invariants（AR 内）**：

- I1：`kind=user` ⇒ `passcode_hash` 非空 + `passcode_set_at` 非空
- I2：`kind=agent` ⇒ `passcode_hash` 为 NULL
- I3：`id` 前缀必须匹配 `kind`（`user-` ⇔ user / `agent-` ⇔ agent）
- I4：`display_name` 不为空，长度 1-40
- I5：`account_status=disabled` 是终态，可被 re-enable（重置回 active）

**生命周期**：

```
created (active)  ─── disable ───→  disabled
       ↑                              │
       └─────── re-enable ────────────┘
```

#### § 4.2.2 Organization AR

```
Organization {
  id          string PK   // "organization-<8hex>"
  slug        string      // UNIQUE; [a-z0-9-]{3,40}; URL key
  name        string      // 显示名；长度 1-80
  description string?
  created_by_identity_id  string   // FK Identity.id
  created_at  timestamp
  updated_at  timestamp
  deleted_at  timestamp?  // 软删除
}

INDEXES:
- UNIQUE (id)
- UNIQUE (slug) WHERE deleted_at IS NULL  -- 软删后 slug 可被新 organization 复用
- INDEX (created_by_identity_id)
```

**Invariants**：

- O1：`slug` 满足 regex `^[a-z0-9]([a-z0-9-]{1,38}[a-z0-9])?$`
- O2：`slug` 在所有未软删 Organization 中唯一
- O3：`deleted_at` 非 NULL 时，所有该 Organization 下的 Member / Channel / Project / Worker / Agent / Secret 软删（级联，由 Domain Service 守门）
- O4：Organization 创建时必同事务创建创建者的 owner Member 行（[[I-O-M-bootstrap]]）

**生命周期**：

```
created → active ── delete (soft) ──→ deleted
```

硬删除（purge）v2.6 不出；DBA 通过 CLI / 直接 SQL 处理。

#### § 4.2.3 Member AR

```
Member {
  id                       string PK   // "mem-<8hex>"
  organization_id          string      // FK Organization.id
  identity_id              string      // FK Identity.id
  role                     enum        // owner | admin | member
  status                   enum        // joined | disabled
  joined_at                timestamp
  invited_by_identity_id   string?     // 创建该 Member 的人；signup-bootstrap 时为 NULL
  invited_at               timestamp?
  disabled_at              timestamp?
  disabled_reason          string?
}

INDEXES:
- UNIQUE (organization_id, identity_id) WHERE status='joined'   -- 同一 Identity 同一 Organization 同时只能有一个 joined 记录
- INDEX (identity_id, status)
- INDEX (organization_id, role)
```

**Invariants**：

- M1：`(organization_id, identity_id)` 在 `status='joined'` 内唯一
- M2：每个 Organization 至少有 1 个 `role=owner AND status=joined` 的 Member（[[organization-min-owner]]，由 Domain Service 守门）
- M3：role 变更不能让 Organization 失去最后一个 owner
- M4：disable 最后一个 owner Member 不允许
- M5：`disabled_at` 非 NULL ⇔ `status=disabled`
- M6：Identity.account_status=disabled 时，该 Identity 的所有 Member.status 自动视作 disabled（运行时检查；不强制写 DB）

**生命周期**：

```
joined ─── disable ───→ disabled ─── re-enable ───→ joined
```

#### § 4.2.4 Invitation AR

> **v2.6 仅建表 + 占位；不出流程 UI / 邮件投递。Application Service 不暴露 create 入口。**

```
Invitation {
  id                       string PK    // "inv-<8hex>"
  organization_id          string       // FK Organization.id
  invitee_handle           string       // 待邀请者的 display_name 或 email（v2.7 决定）
  role_to_grant            enum         // owner | admin | member
  invited_by_identity_id   string       // FK Identity.id
  status                   enum         // pending | accepted | expired | revoked
  token                    string       // 一次性 hash；UNIQUE
  created_at               timestamp
  expires_at               timestamp
  accepted_by_identity_id  string?
  accepted_at              timestamp?
}

INDEXES:
- UNIQUE (id)
- UNIQUE (token)
- INDEX (organization_id, status)
```

**Invariants（v2.7 实现）**：

- IV1：邀请方 `invited_by_identity_id` 必为同 Organization 的 owner/admin Member
- IV2：accept 时若 Identity 不存在，事务内 create Identity + Member
- IV3：`status=accepted` ⇔ `accepted_at` 非空 + `accepted_by_identity_id` 非空

### § 4.5 Domain Services

| Service | 职责 | 跨聚合关系 |
|---|---|---|
| **SignupService** | 接受表单 → 一事务里 create Identity + Organization + Member (role=owner) | Identity + Organization + Member |
| **SigninService** | 接受 display_name + passcode → verify hash → emit JWT | Identity |
| **MemberAdmissionService** | 接受 Identity + Organization → create Member 行 + 守 [[organization-min-owner]] 不变 | Member + Organization |
| **MemberRoleChangeService** | 接受 Member + new_role → 校验最后 owner 不丢 + 更新 | Member |
| **OrganizationLifecycleService** | Organization 软删 → 级联软删 Members + 触发 Workforce/TaskRuntime cleanup hook | Organization + Member + 跨 BC |
| **AgentIdentityProvisionService** | owner/admin 在 UI Add Agent → create Identity (kind=agent) + create Member + register Worker hook | Identity + Member + Workforce |

### § 4.6 Repositories

```go

type InvitationRepository interface {
```

### § 4.8 跨聚合 Invariants（三层结构）

> **方法论**：DDD 三层 invariants 责任分布：
> - **AR-internal**：聚合自身用构造器 + setter + state 守门；不依赖外部状态
> - **Cross-AR (Domain Service)**：跨多 AR 一致性；通过 Domain Service 显式封装；同事务 + 必要时悲观锁 / 乐观 CAS
> - **Cross-BC (Application Service)**：跨多 BC 协调；AppService 内串行调下游；失败时 unwind 已副作用

#### § 4.8.1 AR-internal Invariants（参 § 4.2.x）

| AR | Invariants | 守门时机 |
|---|---|---|
| Identity | I1-I5 (kind/passcode/id 前缀/display_name 长度/status 终态) | Factory + Setter + state transition |
| Organization | O1-O4 (slug 格式/slug 软删唯一性/级联软删/最小 owner) | Factory + Setter |
| Member | M1-M6 (organization+identity 唯一/最少 owner/disable 守门/account 联动) | Factory + Setter |
| Invitation | IV1-IV3 (v2.7) | v2.7 实现 |

AR 自身保证的失败 → ValueError / IllegalStateError；上层捕获后转 Domain Error。

#### § 4.8.2 跨 AR Invariants（Domain Service 守门）

##### DS-1 Signup —— Identity + Organization + Member 三件套同事务

**Trigger**：未认证用户提交 `/signup` 表单。

**Invariant**：
- 表单提交成功 ⇒ DB 中**同时**存在新 Identity 行 + 新 Organization 行 + 1 个 owner Member 行
- 任一步失败 ⇒ 整体 rollback，DB 无遗留

**Pseudocode**：
```go
func (s *SignupService) Execute(ctx, form SignupForm) (*SignupResult, error) {
    if err := form.Validate(); err != nil {
        return nil, err  // display_name length / passcode format / slug regex
    }
    return s.tx.Run(ctx, func(tx Tx) (*SignupResult, error) {
        // 1. Identity
        if _, err := s.identityRepo.GetByDisplayName(tx, form.DisplayName); err == nil {
            return nil, ErrIdentityDisplayNameTaken
        }
        identity := IdentityFactory.NewUser(form.DisplayName, form.Passcode)
        if err := s.identityRepo.Save(tx, identity); err != nil { return nil, err }
        // 2. Organization
        if _, err := s.organizationRepo.GetBySlug(tx, form.OrganizationSlug); err == nil {
            return nil, ErrOrganizationSlugTaken
        }
        organization := OrganizationFactory.New(form.OrganizationSlug, form.OrganizationName, identity.ID)
        if err := s.organizationRepo.Save(tx, organization); err != nil { return nil, err }
        // 3. owner Member
        member := MemberFactory.New(organization.ID, identity.ID, RoleOwner, nil /*invitedBy*/)
        if err := s.memberRepo.Save(tx, member); err != nil { return nil, err }
        // 4. Emit events (out-tx; or via outbox pattern)
        s.events.Emit(IdentityCreated{...}, OrganizationCreated{...}, MemberAdded{...})
        return &SignupResult{Identity: identity, Organization: organization, Member: member}, nil
    })
}
```

**Concurrency**：
- 并发 signup 同 slug → DB UNIQUE 约束捕获，其中一个返回 ErrOrganizationSlugTaken
- 并发 signup 同 display_name → 同上
- 无显式锁需要；DB 唯一约束兜底

**Failure Modes**：

| 错误 | HTTP | 说明 |
|---|---|---|
| `ErrIdentityDisplayNameTaken` | 409 | display_name 已存在 |
| `ErrOrganizationSlugTaken` | 409 | slug 已存在 |
| `ErrOrganizationSlugInvalid` | 400 | slug 格式不合法 |
| Form validation 失败 | 400 | display_name / passcode / organization_name 长度等 |

##### DS-2 [[organization-min-owner]] —— 每个 Organization 至少 1 个 active owner Member

**Trigger**：
- MemberRoleChangeService 修改 role
- MemberDisableService disable Member
- Identity self-disable（v2.7 才出 UI；v2.6 admin-disable 时仍需检查）

**Invariant**：
```
∀ organization ∈ Organization (deleted_at IS NULL):
  count(member ∈ Member WHERE member.organization_id = organization.id
        AND member.role = 'owner'
        AND member.status = 'joined') >= 1
```

**Pseudocode（MemberRoleChangeService）**：
```go
func (s *MemberRoleChangeService) Change(ctx, memberID string, newRole MemberRole, changedBy string) error {
    return s.organizationLock.WithLock(memberOrganizationID, func() error {  // in-process per-organization mutex
      return s.tx.Run(ctx, func(tx Tx) error {
        member, err := s.memberRepo.GetByID(tx, memberID)
        if err != nil { return err }
        oldRole := member.Role

        if oldRole == RoleOwner && newRole != RoleOwner {
            count, err := s.memberRepo.CountActiveOwners(tx, member.OrganizationID)
            if err != nil { return err }
            if count <= 1 { return ErrLastOwnerCannotChangeRole }
        }
        member.Role = newRole
        if err := s.memberRepo.Save(tx, member); err != nil { return err }
        s.events.Emit(MemberRoleChanged{ID: memberID, OldRole: oldRole, NewRole: newRole, ChangedBy: changedBy})
        return nil
      })
    })
}
```

**Pseudocode（MemberDisableService）**：
```go
func (s *MemberDisableService) Disable(ctx, memberID string, reason string) error {
    return s.organizationLock.WithLock(memberOrganizationID, func() error {
      return s.tx.Run(ctx, func(tx Tx) error {
        member, err := s.memberRepo.GetByID(tx, memberID)
        if err != nil { return err }
        if member.Status == StatusDisabled { return nil }  // idempotent

        if member.Role == RoleOwner {
            count, err := s.memberRepo.CountActiveOwners(tx, member.OrganizationID)
            if err != nil { return err }
            if count <= 1 { return ErrLastOwnerCannotDisable }
        }
        member.Status = StatusDisabled
        member.DisabledAt = time.Now()
        member.DisabledReason = reason
        if err := s.memberRepo.Save(tx, member); err != nil { return err }
        s.events.Emit(MemberDisabled{ID: memberID, Reason: reason})
        return nil
      })
    })
}
```

**Concurrency —— 关键 race condition**：

场景：Organization A 有 2 个 owner (M1, M2)，两个 admin 并发 disable M1 + M2。

- T1 begin tx → count_owners(A)=2 (joined)
- T2 begin tx → count_owners(A)=2 (T1 未提交，T2 看不到 T1 的 UPDATE)
- T1 UPDATE M1 status=disabled → commit
- T2 UPDATE M2 status=disabled → commit
- 结果：Organization A 失去最后 owner，**违反 DS-2**

**v2.6 解决方案**：in-process per-organization mutex（`OrganizationLockManager`）保护 owner 增减操作。
- 单机 SQLite 单 writer + Go 内存锁足够
- 不引入 DB 层 SERIALIZABLE（SQLite 不友好）
- 不引入乐观 CAS（增加 schema 复杂度）

```go
type OrganizationLockManager struct {
    mu    sync.Mutex
    locks map[string]*sync.Mutex
}
func (m *OrganizationLockManager) WithLock(organizationID string, fn func() error) error {
    m.mu.Lock()
    if m.locks[organizationID] == nil { m.locks[organizationID] = &sync.Mutex{} }
    lock := m.locks[organizationID]
    m.mu.Unlock()
    lock.Lock(); defer lock.Unlock()
    return fn()
}
```

**Failure Modes**：

| 错误 | HTTP | 说明 |
|---|---|---|
| `ErrLastOwnerCannotChangeRole` | 422 | 降级最后一个 owner 拒绝 |
| `ErrLastOwnerCannotDisable` | 422 | disable 最后一个 owner 拒绝 |
| `ErrMemberNotFound` | 404 | 找不到 |

##### DS-3 Organization 软删级联 —— Organization 软删后该 Organization 下所有 Member 软删

**Trigger**：OrganizationLifecycleService.Delete(organizationID)

**Invariant**：
```
Organization.deleted_at != NULL ⇒
  ∀ member ∈ Member WHERE member.organization_id = Organization.id:
    member.status = 'disabled' AND member.disabled_at != NULL
```

**Pseudocode**：
```go
func (s *OrganizationLifecycleService) Delete(ctx, organizationID string, deletedBy string) error {
    return s.organizationLock.WithLock(organizationID, func() error {
      return s.tx.Run(ctx, func(tx Tx) error {
        organization, err := s.organizationRepo.GetByID(tx, organizationID)
        if err != nil { return err }
        if organization.DeletedAt != nil { return nil }  // idempotent

        // 1. soft-delete all Members in this Organization
        members, _ := s.memberRepo.ListByOrganization(tx, organizationID)
        for _, m := range members {
            if m.Status == StatusJoined {
                m.Status = StatusDisabled
                m.DisabledAt = time.Now()
                m.DisabledReason = "organization-deleted"
                if err := s.memberRepo.Save(tx, m); err != nil { return err }
            }
        }
        // 2. soft-delete Organization
        organization.DeletedAt = time.Now()
        if err := s.organizationRepo.Save(tx, organization); err != nil { return err }

        // 3. emit organization-level event (downstream BCs consume)
        s.events.Emit(OrganizationDeleted{ID: organizationID, DeletedBy: deletedBy})
        return nil
      })
    })
}
```

**跨 BC 级联说明**：
- DS-3 自身只负责 Identity BC 内 Members 级联
- Channels / Projects / Workers / Agents / Secrets 的级联由各 BC AppService 订阅 `organization.deleted` event 异步处理（见 § 4.8.3 AS-3）
- v2.6 简化：异步 + 最终一致；级联完成前访问该 organization-scoped 资源 → § 6.7 中间件检测 Organization 已删 → 403

##### DS-4 Identity disabled = JWT 即时失效

**Trigger**：Identity.account_status 被设为 disabled。

**Invariant**：
```
Identity.account_status = 'disabled' ⇒
  对该 identity_id 的任何 JWT verify 后续都不通过；返回 401 / 跳 /signin
```

**实现策略**：JWT 无状态不可撤销。v2.6 采用 **DB-hit-per-request**：

```go
func (m *AuthMiddleware) Authenticate(ctx, req) (*Identity, error) {
    cookie, _ := req.Cookie("session")
    if cookie == nil { return nil, ErrUnauthenticated }
    claims, err := jwt.Verify(cookie.Value, m.signingKey)
    if err != nil { return nil, ErrUnauthenticated }
    identity, err := m.identityRepo.GetByID(ctx, claims.Subject)
    if err != nil { return nil, ErrUnauthenticated }
    if identity.AccountStatus == StatusDisabled {
        return nil, ErrUnauthenticated  // 401 → SPA 跳 /signin (OQ-2 closed)
    }
    return identity, nil
}
```

**性能考虑**：
- 每请求 1 DB 查询；SQLite 内存索引快，单机可接受
- v2.7 优化：in-memory cache + TTL；admin disable 时 invalidate
- v2.7 进阶：Identity.token_version 列 + JWT 含 version + verify 比对 version

**Failure Modes**：401 → § 6.7 中间件跳 `/signin`（per OQ-2 closed）

##### DS-5 Agent Identity Provision —— Agent Identity + Member 同事务

**Trigger**：owner/admin 在 Members 页 "Add Agent" 提交。

**Invariant**：
```
Identity[kind=agent] 创建 ⇔ 同事务存在该 Identity 对该 Organization 的 Member 行（role 由表单指定）
```

**Pseudocode**：
```go
func (s *AgentIdentityProvisionService) Provision(ctx, form AgentForm, organizationID string, provisionedBy string) (*Identity, *Member, error) {
    return s.tx.Run(ctx, func(tx Tx) (*Identity, *Member, error) {
        if err := form.Validate(); err != nil { return nil, nil, err }
        // 1. provisionedBy 必为该 Organization 的 owner/admin
        provisioner, err := s.memberRepo.GetByOrganizationAndIdentity(tx, organizationID, provisionedBy)
        if err != nil { return nil, nil, err }
        if !provisioner.Role.AtLeast(RoleAdmin) { return nil, nil, ErrForbidden }
        // 2. create Identity[kind=agent]
        identity := IdentityFactory.NewAgent(form.DisplayName, form.Description)
        if err := s.identityRepo.Save(tx, identity); err != nil { return nil, nil, err }
        // 3. create Member
        member := MemberFactory.New(organizationID, identity.ID, form.Role, &provisionedBy)
        if err := s.memberRepo.Save(tx, member); err != nil { return nil, nil, err }
        // 4. cross-BC (AS-1)：若指定 worker，调 Workforce 注册 AgentInstance.identity_id = identity.ID
        if form.WorkerID != "" {
            // Workforce.RegisterAgentInstance(workerID, identity.ID, organizationID)
        }
        s.events.Emit(IdentityCreated{...}, MemberAdded{...})
        return identity, member, nil
    })
}
```

**Concurrency**：agent display_name 不强制全局唯一（仅 user 强制）；同 organization 内不强制；UI 端可加 collision warning，spec 不强约束。

#### § 4.8.3 跨 BC Invariants（Application Service 协调）

> 跨 BC 协调遵循：
> - **同库 SQLite 同事务可行 → 同事务**
> - **同事务不可行（如远调）→ 事件驱动 + 最终一致**
> - **AppService 失败 → 已副作用 unwind**

##### AS-1 Workforce 实体 organization_id 必引用 active Organization

**触发**：Workforce AppService 创建 Worker / AgentInstance / 等。

**协调**：
```go
func (s *AgentInstanceAppService) Create(ctx, form AgentInstanceForm, organizationID string, identityID string) error {
    // AS-1: 校验 Organization active（通过 Identity BC Facade，不直读对方 Repository）
    organization, err := s.identityFacade.GetActiveOrganization(ctx, organizationID)
    if err != nil || organization.DeletedAt != nil { return ErrOrganizationNotFoundOrDeleted }
    return s.tx.Run(ctx, func(tx Tx) error {
        ai := AgentInstanceFactory.New(form, organizationID, identityID)
        return s.agentInstanceRepo.Save(tx, ai)
    })
}
```

**Failure Modes**：Organization 已软删 / 不存在 → 404；下层落库失败 → 事务 rollback。

##### AS-2 Conversation Message.sender_identity_id 必引用存在 Identity

**触发**：Conversation BC 写 Message。

**协调（弱关联）**：
- Conversation AppService 写入前 `identityFacade.Exists(identityID)`；不存在 → 拒绝
- FK NOT NULL（不允许匿名 message）
- Identity disabled 不影响历史 Message 读取（UI 投射时显示 "disabled" tag）

##### AS-3 任何 organization-scoped 实体的 organization_id 必引用未软删 Organization + 软删级联

**触发**：任何 BC AppService 创建 / 更新带 `organization_id` 的实体。

**写入前校验**：每个 AppService 调 `identityFacade.GetActiveOrganization(organizationID)`；nil 或 deleted → 拒绝。

**Organization 软删事件驱动级联**：
```
DS-3 emit organization.deleted (organizationID=X)
↓
Workforce BC 订阅 → soft-delete Worker / AgentInstance / Project where organization_id=X
Conversation BC 订阅 → soft-delete Channel where organization_id=X
SecretManagement BC 订阅 → soft-delete UserSecret where organization_id=X
↓
最终一致；级联期间 (~秒级) § 6.7 中间件拦截
```

**事件订阅故障容忍**：
- v2.6：进程内 in-memory event bus，重启重跑（soft-delete idempotent，无副作用）
- v2.7 升级：outbox pattern 持久化事件 + at-least-once 投递

## § 6. Auth & Session（v2.6 简化版）

### § 6.1 凭据：Passcode

- 6 位数字（`^\d{6}$`）
- argon2id hash（推荐参数：iterations=3, memory=64MiB, parallelism=4，与 OWASP cheat sheet 一致）
- 仅 `kind=user` Identity 有；agent 不参与 web signin

### § 6.2 Signup 流程

```
center 启动 → DB 中无任何 Identity → 任意 URL 跳 /signup
↓
表单：
  - your_display_name (1-40)
  - your_passcode (6 digits) + confirm_passcode
  - first_organization_name (1-80)
  - first_organization_slug (3-40, regex)
↓
SignupService.execute(form):
  TX:
    create Identity (kind=user, id=user-<8hex>, passcode_hash=argon2(passcode), account_status=active)
    create Organization (id=organization-<8hex>, slug, name, created_by_identity_id)
    create Member (id=mem-<8hex>, organization_id, identity_id, role=owner, status=joined)
  emit identity.created + organization.created + member.added
  return { identity_id, organization_slug, jwt }
↓
set httpOnly cookie session=jwt
↓
302 → /organizations/{slug}/
```

**注**：DB 中已有 Identity 时 `/signup` 路径不强制；后续 user 通过 `/signup` 创建额外 Identity 是 v2.7 决定项（多用户 same-machine）。

### § 6.3 Signin 流程

```
GET /signin →
  - 表单：display_name + passcode
↓
SigninService.execute(displayName, passcode):
  identity := IdentityRepository.GetByDisplayName(displayName)
  if identity == nil { return ErrPasscodeInvalid }  -- 不暴露 enumeration
  if identity.account_status == disabled { return ErrPasscodeInvalid }
  if !argon2_verify(passcode, identity.passcode_hash) { emit auth.signin_failed; return ErrPasscodeInvalid }
  jwt := mint(claims{sub=identity.id, exp=now+7d, jti=random})
  emit auth.signed_in
  return jwt
↓
set httpOnly cookie session=jwt
↓
302 → last visited organization（cookie 中带 last_organization_slug 缓存）or / (organization switcher)
```

### § 6.4 Logout

```
POST /signout →
  emit auth.signed_out
  clear cookie
  return 302 /
```

### § 6.5 JWT Cookie

| 字段 | 值 |
|---|---|
| Name | `session` |
| HttpOnly | true |
| Secure | true（生产）；false（dev） |
| SameSite | `Lax` |
| Path | `/` |
| Exp | 7 days fixed；过期需 re-signin |
| 算法 | HS256，密钥用 master_key（v2.4 已有体系） |
| Claims | `{ sub: identity_id, exp, iat, jti }` |

### § 6.6 中间件

```
HTTP request →
  1. 读 cookie session=jwt
  2. verify jwt（签名 + exp）；fail → 401 / 302 /signin（视端点要求）
  3. IdentityRepository.GetByID(jwt.sub)；nil 或 disabled → 401 / 302 /signin
  4. inject current_identity into context
  5. handler
```

### § 6.7 Per-organization Authorization

- 路由 `/organizations/{slug}/...` 在中间件层校验：
  - Slug 存在且未软删
  - current_identity 在该 Organization 有 joined Member 行
  - 否则 403 → 给出"加入此 Organization"提示（v2.6 owner 直接 Add Member 路径 / v2.7 邀请路径）

### § 6.8 Action 级 Role 检查

每个 mutating endpoint 在 AppService 内显式校验：

```go
func (s *ChannelAppService) Archive(ctx, channelID) error {
    member := s.identity.MemberForCtx(ctx)
    if !member.Role.AtLeast(MemberRoleAdmin) {
        return ErrForbidden
    }
    // ...
}
```

详见 § 8 Permission Matrix。

## § 9. Migration 策略（drop-and-recreate per [[no-backward-compat]]）

### § 9.1 总体路线

v2.6 不做向后兼容（[ADR / convention 既定]）。Migration 策略：

1. **新装**：v2.6 init 直接走 `/signup`；schema 一次到位
2. **升级**：现有 v2.5 用户清库重装；release note 明示。v2.4 install 已提供 `uninstall` + 重装路径

### § 9.2 Schema 变更清单（drop-and-recreate）

```sql
-- 1. 新表
CREATE TABLE organizations (...);
CREATE TABLE members (...);
CREATE TABLE invitations (...);

-- 2. 改 identities（迁出 + 重命名 + 加列）
DROP TABLE identities;  -- v2.5 schema
CREATE TABLE identities (
  id TEXT PRIMARY KEY,
  kind TEXT NOT NULL CHECK (kind IN ('user', 'agent')),
  display_name TEXT NOT NULL,
  description TEXT,
  account_status TEXT NOT NULL DEFAULT 'active',
  passcode_hash TEXT,
  passcode_set_at TIMESTAMP,
  created_at TIMESTAMP NOT NULL,
  updated_at TIMESTAMP NOT NULL
);

-- 3. 顶级业务实体加 organization_id
ALTER TABLE channels ADD COLUMN organization_id TEXT NOT NULL REFERENCES organizations(id);
ALTER TABLE projects ADD COLUMN organization_id TEXT NOT NULL REFERENCES organizations(id);
ALTER TABLE workers ADD COLUMN organization_id TEXT NOT NULL REFERENCES organizations(id);
ALTER TABLE agent_instances ADD COLUMN organization_id TEXT NOT NULL REFERENCES organizations(id);
ALTER TABLE agent_instances ADD COLUMN identity_id TEXT NOT NULL REFERENCES identities(id);
ALTER TABLE user_secrets ADD COLUMN organization_id TEXT NOT NULL REFERENCES organizations(id);

-- 4. （SQLite 不支持 ALTER ADD CONSTRAINT；实际操作 = rebuild table）
```

**实操**：v2.6 init 启动检测到旧 schema（无 organizations 表）→ 直接报错 "v2.6 requires fresh install. Run `agent-center uninstall` first."；不自动 migrate。

### § 9.3 v2.4 Worker enroll 协议保留

v2.4 mint-enroll + cert pinning（[[v2.4-mint-enroll]]）继续用，但 enroll 时新增：
- Worker 入参带 `organization_id`（owner/admin 在 UI Add Worker 时指定）
- Worker 关联到 Organization；不跨 Organization

agent instance create 时同事务 create Identity[kind=agent] + Member。

### § 11.3 Supervisor 砍（含在 v2.6 内）

> **2026-05-27 @oopslink 决策**：Supervisor 砍合并到 v2.6 内（原计划独立 v2.6a cycle 取消）。理由：agent 现已可由用户自定义，按项目自定义一个 agent 即可行使 supervisor 行为，无需特制；与 v2.6 的 "Member 是 owner/admin/member" 三级 RBAC 自然兼容。

#### § 11.3.1 砍除清单（DDD 视角）

| 删除项 | 所在 BC | 操作 |
|---|---|---|
| **SupervisorInvocation AR** | BC4 Cognition | 删 AR + Repository + Factory + Domain Service |
| **DecisionRecord 子从属 Entity** | BC4 Cognition | 一并删（SupervisorInvocation 的子从属）|
| **WakeScheduler / InvocationFactory / InvocationTimeoutHandler / InvocationCrashRecovery / DecisionWriter Domain Services** | BC4 Cognition | 删 |
| **AgentInstance.kind=supervisor** | BC3 Workforce | enum 枚举值删；剩 kind=agent 一种 |
| **`agent-center supervisor` CLI 子命令** | CLI surface | 删（含 `start` / `stop` / `status` / `logs` 等） |
| **派单流程中"supervisor 决策回路"** | BC1 TaskRuntime | DispatchService 中所有 supervisor-related branch 简化删 |

#### § 11.3.2 BC4 Cognition 调整后形态

```
v2.5 BC4 Cognition (2 AR):
  ├── SupervisorInvocation AR (含 DecisionRecord 子从属)
  └── Memory AR

v2.6 BC4 Cognition (1 AR):
  └── Memory AR
```

- BC4 仍存在；不缩并到其它 BC
- Memory AR + scope walk + git 仓 file-based 完全不动（per ADR-0012）
- Cognition BC 的 § X.3 Domain Service 节大幅瘦身（5 个 svc 全删，无剩余）
- 因此 Cognition BC 在 v2.6 后约等于"Memory BC"，命名是否改名 v2.6 不动（保 BC4 编号稳定）；命名调整放 v3+ 再视情况

#### § 11.3.3 行为替代路径

| 原 Supervisor 行为 | v2.6 替代 |
|---|---|
| Task 派单决策（哪个 worker 跑） | Project / Worker 静态绑定 + admin 主动指定（v2.5 已基本走这条路；supervisor 决策回路实际未在使用） |
| Task 失败兜底 / 重派 | TaskRuntime BC 的 DispatchService 自身重试机制；超出范围让用户介入 |
| 跨 Issue 决策 / 优先级仲裁 | 用户自定义 agent（如 `supervisor-mbp`）以 admin 角色加入 Org，主动监听 channel + 调用 task API 完成同类行为 |
| Agent provisioning / lifecycle | 已 v2 移到 G1 CLI Endpoint（ADR-0025）；与 supervisor 无关 |

#### § 11.3.4 跨 BC Cleanup 影响

| BC | Cleanup |
|---|---|
| BC1 TaskRuntime | DispatchService 移除 supervisor branch；ADR-0011 修订（Dispatch 可靠性不再依赖 supervisor 决策） |
| BC3 Workforce | AgentInstance.kind 枚举改单值 `agent`；built-in supervisor 注册路径删 |
| BC4 Cognition | 见 § 11.3.2 |
| BC5 Observability | event_type 的 `supervisor.*` 全部退役；audit 投射移除 supervisor 维度 |
| BC6 Conversation | v2.5 已无 supervisor kind Identity（已经合并到 agent）；只需 cleanup leftover refs |

#### § 11.3.5 Migration

per drop-and-recreate（§ 9）：
- `supervisor_invocations` 表 drop
- `decision_records` 表 drop
- `agent_instances` 表 CHECK 约束改（只允许 kind='agent'）
- event_type rows 旧 `supervisor.*` 一并清

#### § 11.3.6 ADR-0044（v2.6 内）

新 ADR 内容包含：
- Context：现有 Supervisor BC 设计（ADR-0011/0013/0029）何以 v2 时立的；为何 v2.6 不再需要
- Decision：砍除范围（§ 11.3.1）+ 替代路径（§ 11.3.3）

  - C. 引入 "Coordinator" 新概念取代 Supervisor：增加复杂度而不解决问题，否决
