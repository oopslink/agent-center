## § 0. 文档约定与方法论

### § 0.1 讨论方法论：最终形态优先

任何特性、子系统、流程的讨论必须**先描述最终形态（end state）**，再决定 v2.6 在该维度的裁剪。任何"v2.6 不做 X"必须挂在最终形态的 X 上明示理由（[[feedback-end-state-first]]）。

### § 0.2 统一语言权威

本文档与 [strategic/03-bounded-contexts § 1](../design/architecture/strategic/03-bounded-contexts.md) 共同构成 v2.6 后通用语言。**任何新名词进文档前必先入 § 2**。

### § 0.3 AR / VO / Entity 选型自检

每个新引入的领域对象按以下顺序自检：

1. 它有独立生命周期吗？→ 是 → AR 或 Entity
2. 它有跨聚合一致性要求自己守门吗？→ 是 → AR
3. 它通过 ID 引用其他对象，其他对象通过它的 Root.id 引用它吗？→ 是 → AR
4. 否则它是 VO（不可变 + 值语义）或子 Entity（隶属某 AR）

### § 0.4 AppService 入口约束

所有持久化必走 BC 的 Application Service；任何 HTTP / CLI / SSE transport 不可直接调 Repository。

### § 0.5 观察 + 能力两层

本文档讨论 v2.6 的设计意图与 schema 形态；具体代码实现层（Repository SQL、HTTP handler 框架）下沉到 implementation 阶段。

### § 0.6 向后兼容

v2.6 不做向后兼容（[[no-backward-compat]]）。schema 走 drop-and-recreate；旧数据不迁移；现有用户清库重装。

## § 1. 背景与目标

### § 1.1 当前形态（v2.5 GA 后）

- Identity AR 在 **Conversation BC**（[ADR-0033](../design/decisions/0033-identity-model-refactor.md)），3 kinds：user / agent / system
- 无 Organization 概念（[ADR-0033 Context § 用户表态](../design/decisions/0033-identity-model-refactor.md)：「V2 不做多租户 也不做组织这个概念」）
- 无 Auth（cookie 都没有，全靠 hostname / localhost 信任）
- 无 Member、无 Role
- 无 UI 管理：Identity / Project / Agent / Secret 都靠 CLI，Web 上不可见
- Supervisor 是 built-in AgentInstance（[ADR-0029](../design/decisions/0029-supervisor-as-builtin-agent-instance.md)），但概念上跟 worker agent 半重叠

### § 1.2 v2.6 想达成什么

1. **Identity BC 独立**：把 Identity AR 从 Conversation BC 抽出，提升为独立 BC（BC9 Identity）
2. **Organization 概念引入**：显式多租户聚合根；多个 org 并发存在；走 URL slug 路由
3. **Member 概念引入**：Identity ↔ Organization 的归属关系，承载 role
4. **Auth 引入**：passcode + 签名 cookie，自助注册 + 登录 + 登出
5. **Members / Org / Agent / Project / Secret Management UI**：所有顶级实体在 web 上可管
6. **为 v2.7+ SaaS 演进铺基**：schema 一次到位（顶级实体全带 org_id），后续不再做大规模迁移

### § 1.3 v2.6 不做（最终形态对照延后表）

| 维度 | 最终形态 | v2.6 | 延后到 |
|---|---|---|---|
| Identity 创建入口 | 自助 / 邀请 / SSO / Agent 多路径 | 自助 + Agent 走 mint-enroll | 邀请 v2.7；SSO v3+ |
| Signin | Email+Pass / Magic Link / SSO / 2FA | display_name + 6 位 passcode | 其它 v2.7+ |
| Session | JWT + refresh + remote logout + timeout | JWT httpOnly cookie，7 day fixed exp | refresh / remote logout v2.7+ |
| Credential 生命周期 | hash + reset + rotation + audit | argon2 hash | reset v2.7（要求 email） |
| API Token | PAT + scope + revoke | agent 继续走 worker-token + cert pinning | PAT v2.7+ |
| Verification | email / phone / KYC | 不做 | v2.7+ |
| Recovery | password reset / 2FA recovery | 文档说明清库 | v2.7（要求 email） |
| Permission | RBAC + ABAC + audit | 基础 RBAC（owner/admin/member）| ABAC v3+ |
| 多 Identity 浏览器 | account switcher / per-tab | /signin 切换 | switcher v2.7+ |
| 审计合规 | login log / lockout / IP | Domain Event 已有 | failed lockout v2.7+ |
| Supervisor 概念 | （目前是 built-in AgentInstance）| **砍**（agent 自定义即可，per [[v2.6a-supervisor-cut]]） | — |
| 邀请流程 UI | 邀请 link + email 投递 | Invitation 表 schema 占位，UI flow 不做 | v2.7 |

## § 2. 统一语言（Ubiquitous Language）—— v2.6 增量

> 本节是 v2.6 后 [strategic/03-bounded-contexts § 1](../design/architecture/strategic/03-bounded-contexts.md#-1-通用语言ubiquitous-language) 的待合并增量。

### § 2.1 新引入术语

| 术语 | 定义 | 出处 / 落地 |
|---|---|---|
| **Identity** | 全局唯一的身份记录；表示"某个 actor"在系统中的标识。一个 Identity 不属于任何 Organization，跨 Org 复用。 | Identity AR |
| **Identity Kind** | Identity 的种类，枚举：`user`（人类）/ `agent`（程序）。v2.6 不再有 `system` kind。 | Identity AR |
| **Organization** (Org) | 多租户工作空间；自治的资源容器。所有顶级业务实体（Channel / Project / Worker / Agent / Secret）属于某个 Organization。 | Organization AR |
| **Org Slug** | Organization 的 URL key，唯一、小写、`[a-z0-9-]{3,40}`，由创建者输入；用于 `/orgs/{slug}/...` 路由。 | Organization AR |
| **Member** | 某个 Identity 在某个 Organization 内的成员关系。一个 Identity 可同时是多个 Org 的 Member；同一 Org 内 (org_id, identity_id) 唯一。 | Member AR |
| **Member Role** | Member 在该 Org 内的权限级别，枚举：`owner` / `admin` / `member`。 | Member.role |
| **Member Status** | Member 在该 Org 内的状态，枚举：`joined`（活跃）/ `disabled`（已禁用，仍保留历史）。区别于 Identity 的 `account_status`。 | Member.status |
| **Account Status** | Identity 自身的全局开关，枚举：`active` / `disabled`。disabled 时所有 Member 关系视作连带失效。 | Identity.account_status |
| **Passcode** | Identity（kind=user）的认证凭据，6 位数字；以 argon2 hash 存储。 | Identity.passcode_hash |
| **Invitation** | Identity 加入某个 Organization 的邀请记录。v2.6 仅建表，流程 v2.7。 | Invitation AR |
| **Signup** | 自助创建 Identity + 第一个 Organization + Member 的入口流程。 | Application Service |
| **Signin** | 凭 display_name + passcode 恢复会话。 | Application Service |
| **Org-scoped** | 属于某 Org 的实体（直接带 `org_id` 列 或 通过 parent 间接归属）。 | Schema 约束 |

### § 2.2 修订术语

| 术语 | 旧定义（v2.5） | v2.6 后含义 |
|---|---|---|
| **Identity ID 格式** | `kind:id` 形式（`user:hayang`、`agent:01HE...`、`system`）| `user-<8hex>` / `agent-<8hex>`（worker-style）；无 `system` 单例 |
| **Identity Kind** | user \| agent \| system | user \| agent（删 system） |
| **AgentInstance.id** | ULID 全局唯一 | 仍 ULID，但与 Identity[kind=agent] 通过 `agent_instance.identity_id` 显式 FK 关联（不再用 id 派生） |

### § 2.3 退役术语

| 术语 | 处置 |
|---|---|
| `system` kind | 删除；纯系统动作 actor=NULL |
| `kind:id` ID format | 取代为 worker-style id |
| Supervisor as built-in AgentInstance | 砍（v2.6a 独立 cycle）；agent 自定义即可（per [[v2.6a-supervisor-cut]]） |
| ChannelBinding（已 v2 删） | 不复活；vendor 重新接入时另起命名 |

## § 3. 上下文边界变化（Bounded Context Landscape）

### § 3.1 v2.5 → v2.6 BC 对比

```
v2.5 GA (7 BC + BC8):
  BC1 TaskRuntime
  BC2 Discussion
  BC3 Workforce
  BC4 Cognition (含 Supervisor)
  BC5 Observability
  BC6 Conversation (含 Identity AR)
  BC8 SecretManagement

v2.6 (7 BC + BC8 + BC9):
  BC1 TaskRuntime
  BC2 Discussion
  BC3 Workforce
  BC4 Cognition          ← Supervisor 砍（v2.6a）；其它聚合不变
  BC5 Observability
  BC6 Conversation       ← Identity AR 抽出；Conversation 内部不再持身份语义
  BC8 SecretManagement
  BC9 Identity           ← 新：Identity / Organization / Member / Invitation 四 AR
```

### § 3.2 Identity BC 抽出（Carve-out）

**动机**：

1. [ADR-0033](../design/decisions/0033-identity-model-refactor.md) 当时把 Identity 留在 Conversation BC 是因为 Identity 主要用作 Message.sender；v2.6 引入 Org / Member / Auth 后，Identity 自身的语义远超 Conversation 域
2. v2.6 把 Identity 当 actor + permission 挂载点，跟 Conversation BC 强耦合违反 SRP
3. Identity BC 独立后，Auth 流程、Member 管理、Org 切换 都聚合在一个 BC 内，跨 BC ACL 简单

**抽出后 Conversation BC 的变化**：

- Identity AR 删，迁至 BC9 Identity
- Message.sender_identity_id 从 BC 内引用变 BC 间 ID 引用
- Conversation BC 不再 INSERT / UPDATE Identity 行；只读 Identity 用作 author 显示
- 历史的 `conversation/02-identity.md` 文档归档；新 `identity/00-overview.md` 替代

### § 3.3 BC Context Map（v2.6 新增上下游）

```
                    ┌───────────────┐
                    │  Identity BC  │← 所有需要 actor / org 上下文的 BC
                    └───────┬───────┘
                            │ (Customer-Supplier; ID 引用 + 软约束)
        ┌───────────────────┼───────────────────┐
        ▼                   ▼                   ▼
  Conversation BC      Workforce BC        TaskRuntime BC
  (Message.sender)     (Worker/Agent.org)  (Task.actor)
        │                   │                   │
        ▼                   ▼                   ▼
                  其它 BC 同理（Discussion / Cognition / Observability / SecretManagement）
```

- **关系类型**：Identity BC 是上游（Supplier），其它 BC 是下游（Customer）
- **Anti-Corruption Layer**：暂不立独立 ACL；通过 published `identity.id` / `organization.id` / `member.id` 走 ID 引用
- **一致性窗口**：弱关联，Identity 注销不立即级联清理（参 [conventions § 0.3 弱关联]）

### § 3.4 与 v2.5 ADR 的冲突解决

| 旧 ADR | v2.6 处置 |
|---|---|
| [ADR-0033 Identity 模型重构](../design/decisions/0033-identity-model-refactor.md) | **Superseded by v2.6 ADR-0040**（新 Identity BC + 新 id 格式 + 删 system kind） |
| [ADR-0029 Supervisor as Built-in AgentInstance](../design/decisions/0029-supervisor-as-builtin-agent-instance.md) | **Superseded by v2.6a ADR-0044**（Supervisor 砍） |
| [ADR-0013 Supervisor 并发模型](../design/decisions/0013-supervisor-invocation-concurrency.md) | **Superseded by v2.6a ADR-0044** |
| [ADR-0011 Dispatch 可靠性协议](../design/decisions/0011-dispatch-reliability-protocol.md) | Re-examine in v2.6a（如涉及 Supervisor） |

## § 4. Identity BC 设计

### § 4.1 BC 概述

| 属性 | 值 |
|---|---|
| BC 序号 | BC9 |
| 中文名 | 身份上下文 |
| 子域分类 | **Supporting** —— 业务支撑；不直接产生用户价值但所有 BC 依赖 |
| 上下游 | 上游所有其它 BC；下游唯有 Observability（事件订阅） |
| 聚合数 | 4（Identity / Organization / Member / Invitation） |
| 持久化 | SQLite 主库；与现有 BC 同库 |

### § 4.2 聚合根（Aggregate Roots）

#### § 4.2.1 Identity AR

```
Identity {
  id              string PK   // "user-<8hex>" | "agent-<8hex>"
  kind            enum        // user | agent
  display_name    string      // "Hayang" / "AgentCenterPD"
  description     string?     // 自由文本
  account_status  enum        // active | disabled
  passcode_hash   string?     // argon2 hash; user 必填，agent 为 NULL
  passcode_set_at timestamp?  // user 必填
  created_at      timestamp
  updated_at      timestamp
}

INDEXES:
- UNIQUE (id)
- UNIQUE (display_name) WHERE kind='user'  -- 防 signin 歧义
- INDEX (kind, account_status)
```

**Invariants（AR 内）**：

- I1：`kind=user` ⇒ `passcode_hash` 非空 + `passcode_set_at` 非空
- I2：`kind=agent` ⇒ `passcode_hash` 为 NULL
- I3：`id` 前缀必须匹配 `kind`（`user-` ⇔ user / `agent-` ⇔ agent）
- I4：`display_name` 不为空，长度 1-40
- I5：`account_status=disabled` 是终态，可被 re-enable（重置回 active）

**生命周期**：

```
created (active)  ─── disable ───→  disabled
       ↑                              │
       └─────── re-enable ────────────┘
```

#### § 4.2.2 Organization AR

```
Organization {
  id          string PK   // "org-<8hex>"
  slug        string      // UNIQUE; [a-z0-9-]{3,40}; URL key
  name        string      // 显示名；长度 1-80
  description string?
  created_by_identity_id  string   // FK Identity.id
  created_at  timestamp
  updated_at  timestamp
  deleted_at  timestamp?  // 软删除
}

INDEXES:
- UNIQUE (id)
- UNIQUE (slug) WHERE deleted_at IS NULL  -- 软删后 slug 可被新 org 复用
- INDEX (created_by_identity_id)
```

**Invariants**：

- O1：`slug` 满足 regex `^[a-z0-9]([a-z0-9-]{1,38}[a-z0-9])?$`
- O2：`slug` 在所有未软删 Org 中唯一
- O3：`deleted_at` 非 NULL 时，所有该 Org 下的 Member / Channel / Project / Worker / Agent / Secret 软删（级联，由 Domain Service 守门）
- O4：Org 创建时必同事务创建创建者的 owner Member 行（[[I-O-M-bootstrap]]）

**生命周期**：

```
created → active ── delete (soft) ──→ deleted
```

硬删除（purge）v2.6 不出；DBA 通过 CLI / 直接 SQL 处理。

#### § 4.2.3 Member AR

```
Member {
  id                       string PK   // "mem-<8hex>"
  organization_id          string      // FK Organization.id
  identity_id              string      // FK Identity.id
  role                     enum        // owner | admin | member
  status                   enum        // joined | disabled
  joined_at                timestamp
  invited_by_identity_id   string?     // 创建该 Member 的人；signup-bootstrap 时为 NULL
  invited_at               timestamp?
  disabled_at              timestamp?
  disabled_reason          string?
}

INDEXES:
- UNIQUE (organization_id, identity_id) WHERE status='joined'   -- 同一 Identity 同一 Org 同时只能有一个 joined 记录
- INDEX (identity_id, status)
- INDEX (organization_id, role)
```

**Invariants**：

- M1：`(organization_id, identity_id)` 在 `status='joined'` 内唯一
- M2：每个 Org 至少有 1 个 `role=owner AND status=joined` 的 Member（[[org-min-owner]]，由 Domain Service 守门）
- M3：role 变更不能让 Org 失去最后一个 owner
- M4：disable 最后一个 owner Member 不允许
- M5：`disabled_at` 非 NULL ⇔ `status=disabled`
- M6：Identity.account_status=disabled 时，该 Identity 的所有 Member.status 自动视作 disabled（运行时检查；不强制写 DB）

**生命周期**：

```
joined ─── disable ───→ disabled ─── re-enable ───→ joined
```

#### § 4.2.4 Invitation AR

> **v2.6 仅建表 + 占位；不出流程 UI / 邮件投递。Application Service 不暴露 create 入口。**

```
Invitation {
  id                       string PK    // "inv-<8hex>"
  organization_id          string       // FK Organization.id
  invitee_handle           string       // 待邀请者的 display_name 或 email（v2.7 决定）
  role_to_grant            enum         // owner | admin | member
  invited_by_identity_id   string       // FK Identity.id
  status                   enum         // pending | accepted | expired | revoked
  token                    string       // 一次性 hash；UNIQUE
  created_at               timestamp
  expires_at               timestamp
  accepted_by_identity_id  string?
  accepted_at              timestamp?
}

INDEXES:
- UNIQUE (id)
- UNIQUE (token)
- INDEX (organization_id, status)
```

**Invariants（v2.7 实现）**：

- IV1：邀请方 `invited_by_identity_id` 必为同 Org 的 owner/admin Member
- IV2：accept 时若 Identity 不存在，事务内 create Identity + Member
- IV3：`status=accepted` ⇔ `accepted_at` 非空 + `accepted_by_identity_id` 非空

### § 4.3 Value Objects

| VO | 说明 |
|---|---|
| **IdentityKind** | `user | agent`；ID 前缀守门 |

### § 4.5 Domain Services

| Service | 职责 | 跨聚合关系 |
|---|---|---|
| **SignupService** | 接受表单 → 一事务里 create Identity + Organization + Member (role=owner) | Identity + Organization + Member |
| **SigninService** | 接受 display_name + passcode → verify hash → emit JWT | Identity |
| **MemberAdmissionService** | 接受 Identity + Org → create Member 行 + 守 [[org-min-owner]] 不变 | Member + Organization |
| **MemberRoleChangeService** | 接受 Member + new_role → 校验最后 owner 不丢 + 更新 | Member |
| **OrganizationLifecycleService** | Org 软删 → 级联软删 Members + 触发 Workforce/TaskRuntime cleanup hook | Organization + Member + 跨 BC |
| **AgentIdentityProvisionService** | owner/admin 在 UI Add Agent → create Identity (kind=agent) + create Member + register Worker hook | Identity + Member + Workforce |

```go
    Save(ctx context.Context, inv *Invitation) error

    GetByToken(ctx context.Context, token string) (*Invitation, error)
    ListByOrg(ctx context.Context, orgID string) ([]*Invitation, error)
}

// Domain errors (sentinel pattern per conventions § 0.3)
var (
    ErrIdentityNotFound       = errors.New("identity not found")

```

## § 6. Auth & Session（v2.6 简化版）

### § 6.1 凭据：Passcode

- 6 位数字（`^\d{6}$`）
- argon2id hash（推荐参数：iterations=3, memory=64MiB, parallelism=4，与 OWASP cheat sheet 一致）
- 仅 `kind=user` Identity 有；agent 不参与 web signin

### § 6.2 Signup 流程

```
center 启动 → DB 中无任何 Identity → 任意 URL 跳 /signup
↓
表单：
  - your_display_name (1-40)
  - your_passcode (6 digits) + confirm_passcode
  - first_org_name (1-80)
  - first_org_slug (3-40, regex)
↓
SignupService.execute(form):
  TX:
    create Identity (kind=user, id=user-<8hex>, passcode_hash=argon2(passcode), account_status=active)
    create Organization (id=org-<8hex>, slug, name, created_by_identity_id)
    create Member (id=mem-<8hex>, org_id, identity_id, role=owner, status=joined)
  emit identity.created + organization.created + member.added
  return { identity_id, org_slug, jwt }
↓
set httpOnly cookie session=jwt
↓
302 → /orgs/{slug}/
```

**注**：DB 中已有 Identity 时 `/signup` 路径不强制；后续 user 通过 `/signup` 创建额外 Identity 是 v2.7 决定项（多用户 same-machine）。

### § 6.3 Signin 流程

```
GET /signin →
  - 表单：display_name + passcode
↓
SigninService.execute(displayName, passcode):
  identity := IdentityRepository.GetByDisplayName(displayName)
  if identity == nil { return ErrPasscodeInvalid }  -- 不暴露 enumeration
  if identity.account_status == disabled { return ErrPasscodeInvalid }
  if !argon2_verify(passcode, identity.passcode_hash) { emit auth.signin_failed; return ErrPasscodeInvalid }
  jwt := mint(claims{sub=identity.id, exp=now+7d, jti=random})
  emit auth.signed_in
  return jwt
↓
set httpOnly cookie session=jwt
↓
302 → last visited org（cookie 中带 last_org_slug 缓存）or / (org switcher)
```

### § 6.4 Logout

```
POST /signout →
  emit auth.signed_out
  clear cookie
  return 302 /
```

v2.6 不做"远程 logout / 其它设备登出"。

### § 6.5 JWT Cookie

| 字段 | 值 |
|---|---|
| Name | `session` |
| HttpOnly | true |
| Secure | true（生产）；false（dev） |
| SameSite | `Lax` |
| Path | `/` |
| Exp | 7 days fixed；过期需 re-signin |
| 算法 | HS256，密钥用 master_key（v2.4 已有体系） |
| Claims | `{ sub: identity_id, exp, iat, jti }` |

### § 6.6 中间件

```
HTTP request →
  1. 读 cookie session=jwt
  2. verify jwt（签名 + exp）；fail → 401 / 302 /signin（视端点要求）
  3. IdentityRepository.GetByID(jwt.sub)；nil 或 disabled → 401 / 302 /signin
  4. inject current_identity into context
  5. handler
```

### § 6.7 Per-org Authorization

- 路由 `/orgs/{slug}/...` 在中间件层校验：
  - Slug 存在且未软删
  - current_identity 在该 Org 有 joined Member 行
  - 否则 403 → 给出"加入此 Org"提示（v2.6 owner 直接 Add Member 路径 / v2.7 邀请路径）

### § 6.8 Action 级 Role 检查

每个 mutating endpoint 在 AppService 内显式校验：

```go
func (s *ChannelAppService) Archive(ctx, channelID) error {
    member := s.identity.MemberForCtx(ctx)
    if !member.Role.AtLeast(MemberRoleAdmin) {
        return ErrForbidden
    }
    // ...
}
```

详见 § 8 Permission Matrix。

## § 9. Migration 策略（drop-and-recreate per [[no-backward-compat]]）

### § 9.1 总体路线

v2.6 不做向后兼容（[ADR / convention 既定]）。Migration 策略：

1. **新装**：v2.6 init 直接走 `/signup`；schema 一次到位
2. **升级**：现有 v2.5 用户清库重装；release note 明示。v2.4 install 已提供 `uninstall` + 重装路径

### § 9.2 Schema 变更清单（drop-and-recreate）

```sql
-- 1. 新表
CREATE TABLE organizations (...);
CREATE TABLE members (...);
CREATE TABLE invitations (...);

-- 2. 改 identities（迁出 + 重命名 + 加列）
DROP TABLE identities;  -- v2.5 schema
CREATE TABLE identities (
  id TEXT PRIMARY KEY,
  kind TEXT NOT NULL CHECK (kind IN ('user', 'agent')),
  display_name TEXT NOT NULL,
  description TEXT,
  account_status TEXT NOT NULL DEFAULT 'active',
  passcode_hash TEXT,
  passcode_set_at TIMESTAMP,
  created_at TIMESTAMP NOT NULL,
  updated_at TIMESTAMP NOT NULL
);

-- 3. 顶级业务实体加 org_id
ALTER TABLE channels ADD COLUMN org_id TEXT NOT NULL REFERENCES organizations(id);
ALTER TABLE projects ADD COLUMN org_id TEXT NOT NULL REFERENCES organizations(id);
ALTER TABLE workers ADD COLUMN org_id TEXT NOT NULL REFERENCES organizations(id);
ALTER TABLE agent_instances ADD COLUMN org_id TEXT NOT NULL REFERENCES organizations(id);
ALTER TABLE agent_instances ADD COLUMN identity_id TEXT NOT NULL REFERENCES identities(id);
ALTER TABLE user_secrets ADD COLUMN org_id TEXT NOT NULL REFERENCES organizations(id);

-- 4. （SQLite 不支持 ALTER ADD CONSTRAINT；实际操作 = rebuild table）
```

**实操**：v2.6 init 启动检测到旧 schema（无 organizations 表）→ 直接报错 "v2.6 requires fresh install. Run `agent-center uninstall` first."；不自动 migrate。

### § 9.3 v2.4 Worker enroll 协议保留

v2.4 mint-enroll + cert pinning（[[v2.4-mint-enroll]]）继续用，但 enroll 时新增：
- Worker 入参带 `organization_id`（owner/admin 在 UI Add Worker 时指定）
- Worker 关联到 Org；不跨 Org

agent instance create 时同事务 create Identity[kind=agent] + Member。
