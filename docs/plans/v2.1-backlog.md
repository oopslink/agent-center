## Unread message tracking

**User story**: channel / DM list rows show a badge with the number of
new messages since the user last viewed that conversation.

**Surfaces where deferred** (currently noop):
- `web/src/pages/DMs.tsx` — `<DMs />` row's right column is blank;
  comment in source references this backlog
- `web/src/pages/Channels.tsx` — same (no badge column at all today)
- Sidebar nav badges — Zustand `navBadges.inputRequests` exists but
  there is no per-channel / per-DM counter

**What's missing to ship it**:
1. **Domain model**: a `user_conversation_read_state(user_id,
   conversation_id, last_seen_message_id, updated_at)` table + repo
2. **Service**: `MarkSeenService.MarkSeen(user_id, conv_id, msg_id)`
   + read-state queries that join on the latest message id per conv
3. **HTTP endpoints**:
   - `POST /api/conversations/{id}/seen {message_id}` — bump
   - `GET /api/conversations?include=unread_count` — augment list
4. **SSE event type**: `conversation.read_state_changed` so other tabs
   refresh the badge without polling
5. **Frontend**:
   - mutation hook `useMarkConversationSeen`
   - ChannelDetail / DMDetail call it on mount + when new messages
     arrive while the page is focused
   - List rows render `unread_count` when > 0

**Estimate**: ~1 day backend + ~0.5 day frontend (~6h total)

**Priority**: post-v2 GA — single-user loopback workflow can tolerate
"unread = re-scroll the list" for now. Becomes worth doing when multi-
device usage starts (then read-state sync also covers desktop ↔ mobile).

**Why deferred**:
- Out of P11 plan § 1.6 page scope (page list doesn't include unread
  badge)
- Not blocking v2 GA functional surface (everything else works without
  this) — `/inputrequests` page already covers the urgent-attention
  case via the IR badge

**Tracking**:
- noop-render sites tagged with `v2.1-backlog "Unread message
  tracking"` comments so a grep surfaces them
- Re-evaluate at P12 release-gate review
