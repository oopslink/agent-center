| 维度 | 现状 | v2.6 目标 |

- **多用户 SaaS / 跨实例 sync** — Identity BC 提升不引入跨实例同步；本机单 Identity 仓库

- **Identity profile picture / avatar** — 不做
- **Audit log UI** — Identity lifecycle events 已 emit，但专门的 audit log UI 推迟

## § 3 Identity BC 提升

### § 3.1 Identity AR 模型

```
Identity {
  id              : "user:hayang" | "agent:AgentCenterPD" | "system:bootstrap"
                    // kind:id prefix per ADR-0033, immutable
  kind            : human | agent | system
  display_name    : string (mutable, e.g. "OopsLinK", "AgentCenter PD")
  description     : string optional
  role            : owner | admin | member | guest
                    // v2.6 stored only, not enforced on endpoints (see § 2.2)
  status          : active | invited | disabled
  created_at      : timestamp
  updated_at      : timestamp
  invited_by      : Identity.id optional (who created this identity)
  disabled_at     : timestamp optional
  disabled_reason : string optional
}
```

**Lifecycle events** (Observability BC):
- `identity.created` — payload: id / kind / display_name / invited_by
- `identity.updated` — payload: id / changed_fields
- `identity.role_changed` — payload: id / old_role / new_role / by
- `identity.disabled` — payload: id / by / reason
- `identity.re_enabled` — payload: id / by

### § 3.2 Carve-out from Conversation BC

**当前**：
- `internal/conversation/identity.go`（per ADR-0033）有 identity-related types + `ChannelBinding`
- `conversations` 表无 identity 表；identity 是 kind:id 字符串嵌在 messages / participants

**v2.6 之后**：
- 新建 `internal/identity/` 包 + `identities` 表（migration 0032）
- ChannelBinding 留在 Conversation BC（它是 conversation 内的概念）但其 owner 字段引用 Identity.id
- Workforce / Cognition / Observability 引用 Identity.id 不变（已经是 kind:id 字符串）
- Conversation participants / message authors 引用 Identity.id 不变

**Migration 策略**：
- 扫现有所有 BC 数据，提取所有 distinct kind:id → 一次性创建 Identity 行（kind from prefix, display_name = id 后缀作 default, role = "member"）
- 系统 identity（`system:bootstrap` 等）自动 created with role = "system"
- 老 ChannelBinding 等表的 identity 字段不动；只是新加 Identity 表作为 catalog

### § 3.3 Cross-BC interactions（map）

```
Identity BC          ←——————— principals catalog
   ↑
   |── refed by FK (string kind:id)
   |
Conversation BC      — message.author_id / participants
Workforce BC         — Worker.owner / AgentInstance.identity_id
Cognition BC         — Supervisor identity (built-in)
Observability BC     — event.actor_id
```

新 Identity BC 是其它 BC 的 **upstream** —— 它们引用 Identity，Identity 不知道它们。

## § 6 待 @oopslink 决策点

### D1. Identity.role enum

提议：`owner / admin / member / guest`。

- **owner**：所有权限（user:hayang 一开始就是 owner）
- **admin**：可管理其它 members、resources，不能删除 owner
- **member**：标准成员（创建 task / 看 fleet / 改自己的 worker / 写消息）
- **guest**：只读权限（看 fleet / 看 conversation，不能写）

可选简化：去掉 guest，只用 owner / admin / member 三档。

**你拍**：3 档 / 4 档 / 别的方案？

### D2. Invite human 机制

提议：v2.6 走 "generate invite link"（用户手动发链接），不集成 email。

- 链接长这样：`http://127.0.0.1:7100/accept-invite?token=<one-time-invite-token>`
- 受邀人打开 → 填 display name → 创建 user identity → 加入 Members

**你拍**：generate invite link / 其它机制 / 暂不做 invite human（v2.6 只支持 disable existing humans + set role）？

### D3. Agent disable 行为

提议：disable agent 时：
- Revoke 所有 admin tokens owned by 这个 identity
- AgentInstance 标记 `status=disabled`（新字段）
- 新 dispatch 拒绝走这个 agent
- Active executions 继续到完成 / fail / kill

**你拍**：上面方案 OK？要不要 "force kill running executions"？

### D4. Project Delete cascade 行为

提议：
- 默认 reject if active issues / tasks exist (return 409)
- `--force-delete` flag (CLI) or "Force delete" button (UI) → cascade abandon all tasks + close all issues

**你拍**：reject-by-default OK？还是默认就 cascade？

### D5. Secret Revoke cascade 行为

提议：
- Revoke 改 Secret.status = revoked
- 已经 reference 的 agent 不立刻挂（仍能 read 缓存的 plaintext from agent 进程），但 supervisor 收到 alert
- 重启 agent 进程时 secret resolve 失败 → agent enroll 失败 → 状态 disabled

**你拍**：alert-based 软处理 OK？要不要立刻 disable references？

### D6. AgentInstance vs Identity 映射

**关键 design 问**：一个 agent identity 对应**一个 AgentInstance** 还是**多个**？

- **方案 A**：1:1 —— Identity = AgentInstance 同义；每个 agent 进程一个 identity
- **方案 B**：1:N —— 一个 Identity 可以有多个 AgentInstance（同一身份的 agent 可以同时跑在多 worker 上）

v2 现状是模糊的（AgentInstance 一等公民，但跨 conversation 引用用 kind:id 字符串）。Members 视图选 (a) 会让 list 干净；选 (b) 反映现实灵活。

**你拍**：1:1 / 1:N / 不知道（等 v2.6 实施时再说）？

### D7. Global project switcher 默认 project

新用户登录默认 project：
- (a) "All projects" (不 filter)
- (b) 最近创建的 project
- (c) 用户上次选的（localStorage 记住）
- (d) 用户的 "default project"（Identity 字段新加）

提议：(c)，第一次登录回退 (a)。

**你拍**：(c) OK / 其它？

### Receiver 端：访问 `/accept-invite?token=...`

```
┌─ Accept invitation ───────────────────────────────┐
│                                                    │
│  You've been invited to join                       │
│  this AgentCenter as **member**.                   │
│                                                    │
│  Your display name (preset by inviter):           │
│  ┌──────────────────────┐                         │
│  │ Alice                 │ ← editable             │
│  └──────────────────────┘                         │
│                                                    │
│  Your identity will be saved as:                   │
│  ┌────────────────────┐                            │
│  │ user:alice_<8hex>  │ (immutable)                │
│  └────────────────────┘                            │
│                                                    │
│  [ Accept invitation ]                             │
└────────────────────────────────────────────────────┘
```

→ POST `/api/accept-invite {token, final_display_name}` → 验证 token + identity status: invited → 转 status: active + emit `identity.accepted` → 重定向到 `/`。

**Error states for invite acceptance**:
- token expired → "This invite has expired. Ask the inviter to generate a new one."
- token revoked (inviter removed member) → "This invite was revoked."
- token already used → "This invite was already accepted. Please log in or ask for a new invite."

**v2.6 不做 authentication**：Accept invite UX 不 require 密码 / 2FA / SSO —— 单 user 实例 + 受 deployment trust 边界保护（local-only WC 默认）。multi-user with auth 留 v3+ SaaS。

## § 6.6 Identity migration 0032 SQL draft

```sql
-- migration 0032: identity BC standalone

CREATE TABLE identities (
  id              TEXT PRIMARY KEY,           -- "user:hayang" | "agent:AgentCenterPD" | "system:bootstrap"
  kind            TEXT NOT NULL CHECK(kind IN ('human','agent','system')),
  display_name    TEXT NOT NULL,
  description     TEXT NOT NULL DEFAULT '',
  role            TEXT NOT NULL DEFAULT 'member' CHECK(role IN ('owner','admin','member','guest','system')),
  status          TEXT NOT NULL DEFAULT 'active' CHECK(status IN ('active','invited','disabled')),
  invited_by      TEXT,                       -- FK → identities.id (nullable)
  disabled_at     INTEGER,                    -- unix ms (nullable)
  disabled_reason TEXT,
  created_at      INTEGER NOT NULL,           -- unix ms
  updated_at      INTEGER NOT NULL,
  version         INTEGER NOT NULL DEFAULT 1, -- CAS
  FOREIGN KEY (invited_by) REFERENCES identities(id)
);

CREATE INDEX idx_identities_kind_status ON identities(kind, status);
CREATE INDEX idx_identities_role ON identities(role);

-- Backfill: extract distinct kind:id from existing data
-- (executed as Go code in migration; this SQL is the schema only)

-- Source tables to scan for kind:id strings:
-- - conversation participants (conversations.participants_json)
-- - message author_id
-- - workers.owner
-- - agent_instances.identity_id
-- - events.actor_id
-- - admin_tokens.owner (already kind:id formatted)
-- - supervisor_invocations.actor_id

-- Backfill logic (Go pseudo):
-- 1. scan all source tables, collect distinct kind:id strings
-- 2. for each kind:id:
--    parse kind from prefix
--    display_name = id-after-prefix (capitalize first letter)
--    role = system if kind=system else owner if id=hayang else member
--    status = active
--    created_at = MIN(created_at) from where identity was first seen
-- 3. INSERT INTO identities (... ON CONFLICT DO NOTHING)
```

**Migration safety**：
- Backfill is **idempotent** (ON CONFLICT DO NOTHING)
- All existing kind:id references in other tables stay as-is (still strings, FK semantic is now formal)
- Rollback (down migration) just DROPs identities table — existing data unaffected since other tables don't reference it via SQL FK yet

## § 6.8 ADR-0042 outline: "Identity as standalone BC"

```
# ADR-0042. Identity as standalone Bounded Context

| Field | Value |
|---|---|
| Status | Proposed (v2.6) |
| Date | 2026-05-26 |
| Amends | ADR-0033 (partial — identity-in-Conversation portion superseded) |
| Related | ADR-0037 § W1.4 (amended by ADR-0043) |

## Context

ADR-0033 把 4-kind identity model 落进 Conversation BC（ChannelBinding 那一层）。
随着 v2.6 Members 管理需求出现 + identity 被跨 Workforce/Cognition/Observability
BC 引用变频繁，identity 留在 Conversation BC 里造成两类问题：

1. **BC 边界 leak**：Workforce / Cognition 需要 identity 元数据（display_name,
   role, status）就得 reach into Conversation BC，违反 § 9.z 不共享物理表。
2. **Identity 缺乏自己的 lifecycle**：v2.6 Members 要求 invite / disable /
   role_change，需要一个有状态、有 lifecycle events 的 AR；Conversation BC 内
   持不住这层逻辑。

## Decision

Identity 提升为**独立 BC** (`internal/identity/`)。
Conversation BC 保留 ChannelBinding（vendor-specific 身份绑定），但 ChannelBinding.identity_id
引用新 Identity BC 的 AR。

Identity AR fields: id (kind:id prefix immutable) / kind / display_name / role /
status / lifecycle metadata (see v2.6-candidates § 3.1).

Cross-BC reference: 所有其它 BC 用 string kind:id 引用 Identity（per ADR-0033 prefix model 不变），
Identity BC 是 upstream catalog。

## Consequences

+ BC 边界清晰；Identity lifecycle 有专门 AR
+ Members 管理有正规 BC 落点
- 1 个新 BC 增加复杂度
- migration 0032 需要 backfill
- Conversation BC carve-out 工作
```
