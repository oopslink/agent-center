# v2.2 — Mac single-host deployment guide

> Target audience: anyone (operator or developer) standing up
> agent-center on a single Mac. v2.2 ships the
> `server + worker daemon + web console` deployment that v2.0 GA
> declared but never actually wired (per `docs/plans/v2.2-audits/v22-closeout-audit.md`).
>
> Multi-host TCP transport is filed for v2.3 (ADR-0040 reserves the
> design; v2.2 ships unix socket only).

## § 1. Prerequisites

- macOS (Linux works too; instructions identical except socket path)
- Go 1.22+ (only needed if building from source)
- Node.js + pnpm (only needed for Playwright e2e or rebuilding the
  Web Console SPA)
- 100MB free disk (sqlite + binaries + logs)

## § 2. Build

```bash
cd /path/to/agent-center
make build         # produces bin/agent-center + bin/agent-center-worker-daemon + bin/fakeagent
```

Three binaries land in `./bin/`:

| Binary | Role |
|--------|------|
| `agent-center` | Server, CLI, and supervisor subprocess (single binary, multi-mode) |
| `agent-center-worker-daemon` | Worker process that drains dispatched envelopes and runs agent CLIs |
| `fakeagent` | Scripted agent stub for LLM-independent tests; can substitute for claude-code / codex / opencode |

## § 3. Config

Minimum v2.2 config (`/tmp/v22-deploy.yaml`):

```yaml
server:
  listen_addr: ":7000"
  sqlite_path: "/tmp/v22-deploy.db"
  admin_socket_path: "/tmp/v22-admin.sock"

identity:
  default_user: hayang
```

Web Console is enabled by default in v2.2 (binds `127.0.0.1:7100`
on loopback per ADR-0037). To opt out (headless server), add:

```yaml
web_console:
  enabled: false
```

To bind a different port:

```yaml
web_console:
  listen_addr: "127.0.0.1:7200"   # still loopback per ADR-0037
```

Field guide:
- `admin_socket_path` — unix socket the CLI and worker daemon use
  to reach the AppService. Required for the v2.2 architecture (per
  `conventions § 0.4`).
- `web_console.listen_addr` — loopback only per ADR-0037; SSH-tunnel
  for remote browser access.
- For production: pick a stable directory under your user's runtime
  dir (e.g. `~/Library/Application Support/agent-center/` on macOS)
  rather than `/tmp/`.

## § 4. Start the three processes

Three terminals (or `tmux` panes / `launchd` units):

### Terminal 1 — server

```bash
./bin/agent-center server --config=/tmp/v22-deploy.yaml
```

Expected banner:
```
agent-center server: db=/tmp/v22-deploy.db listen=:7000 web=127.0.0.1:7100 admin=/tmp/v22-admin.sock (escalator running)
```

### Terminal 2 — worker daemon

```bash
./bin/agent-center-worker-daemon \
    --config=/tmp/v22-deploy.yaml \
    --worker-id=mac-w-1 \
    --capabilities=fakeagent \
    --fake-agent=./bin/fakeagent
```

Expected (two lines on startup, then silent main loop):
```
[worker] starting: worker_id=mac-w-1 socket=/tmp/v22-admin.sock poll=1s overrides=1
[worker] enrolled as worker_id=mac-w-1
```

After enroll the main poll loop runs quietly — every 1s the daemon
drains the dispatch/kill queues and only logs on actual events
(`dispatch <execID> task=<taskID> agent=<cli>`, `report-progress
milestone=…`, `conclude execution=…`, etc.). Silence means healthy.

### Terminal 3 — Web Console

Open in browser:
```
http://127.0.0.1:7100
```

You should see the agent-center SPA: channels list (empty),
agent list (empty), fleet (worker `mac-w-1` shown as offline until
first heartbeat — heartbeat current = 1s poll, so it flips quickly).

## § 5. Drive a task end-to-end

### 5.1 Via CLI

```bash
# Create a project + task (Web Console can do this too)
./bin/agent-center project add \
    --config=/tmp/v22-deploy.yaml \
    --id=demo --name=Demo --base-path=/tmp/demo-project

./bin/agent-center task new \
    --config=/tmp/v22-deploy.yaml \
    --project-id=demo \
    --title="hello world"

# Dispatch (assign to a worker)
./bin/agent-center dispatch <task-id> \
    --config=/tmp/v22-deploy.yaml \
    --worker=mac-w-1 \
    --agent-cli=fakeagent

# Observe progression
./bin/agent-center ps --config=/tmp/v22-deploy.yaml
```

Within ~2 seconds the task should hit status `done`. Worker terminal
will log `dispatch <execID> task=<taskID> agent=fakeagent` followed
by progression events.

### 5.2 Via Web Console

In browser at `127.0.0.1:7100`:
1. Navigate to **Channels** → create one
2. Send a message
3. Click message → "Derive Task" (top-right) → select project →
   confirm
4. New task appears in **Tasks**; status flips through
   `submitted → working → completed` as the worker picks it up.
5. **Fleet** page shows the worker + execution counts.

## § 6. Verify the install

A one-shot self-test:

```bash
make smoke
```

Output should end with `smoke pass: NN seconds`. This:
- Builds all 3 binaries
- Runs the Phase D Playwright spec (`tests/e2e/v2/tests/v22-deployed-pipeline.spec.ts`)
- Exercises the full pipeline (server + worker + fakeagent) end-to-end
- Asserts task lands at `done` status

Failure means something is wrong with the install or the system
state — investigate before declaring deploy ready.

## § 7. Shutdown

Each process accepts SIGTERM (Ctrl-C). Order doesn't matter:
- Server cleans the admin socket file on exit
- Worker daemon drains in-flight executions for `ShutdownGrace`
  (default 30s) then escalates SIGKILL

For a real agent (claude-code / codex / opencode), the worker waits
for the agent subprocess to exit before reporting `failed` —
make sure your real agent honors SIGTERM.

## § 8. Production hardening (not in v2.2 scope)

- Multi-host: filed for v2.3 (ADR-0040; admin endpoint will gain TCP
  binding with bootstrap-token auth)
- launchd / systemd unit files: see contrib/ (TODO)
- Log rotation: server writes to stderr; pipe to your preferred
  log shipper
- Secret rotation: see `docs/operations/master-key.md`
- Backup: `agent-center.db` is the only stateful file; standard
  sqlite backup tools work

## § 9. Troubleshooting

| Symptom | Likely cause | Fix |
|---|---|---|
| `connection refused: /tmp/v22-admin.sock` | Server not started, or `admin_socket_path` mismatch | Start server first; check config |
| Worker stays `offline` in Fleet | Worker daemon not running, or wrong `--worker-id` vs `worker enroll` | Restart daemon; verify worker_id matches |
| Task stays `submitted` | Worker daemon isn't draining (check terminal 2 log) OR `--agent-cli=X` doesn't match the worker's capabilities | Check worker `--capabilities=...`; verify agent CLI binary exists |
| `socket already in use` | Stale socket from crashed prior run | `rm <socket-path>`; server cleans on next start |
| `task_execution.failed reason=agent_exit_nonzero` | Agent subprocess exited non-zero | Check worker terminal for agent stderr; for fakeagent verify your script JSONL ends with `{"type":"done","exit_code":0}` |
