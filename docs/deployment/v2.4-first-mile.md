# v2.4 — First-Mile Deployment Guide

> Audience: anyone bringing up an agent-center server + one or more
> worker daemons for the first time. Pairs with
> [v2.3 multi-host guide](v2.3-multi-host.md), which is now mostly a
> reference for the cross-host TCP+TLS internals — operationally,
> prefer this guide.
>
> Background: v2.4-D (first-mile deployment) lands the `install`
> subcommand, the atomic-symlink upgrade with auto-rollback, and the
> Web Console **Add Worker** Modal that mints a one-time enroll token
> and shows a copyable install command. See
> [v2.4-deployment-first-mile.md](../plans/v2.4-deployment-first-mile.md)
> for the product context.

## § 1. Prereqs

You have an `agent-center-v2.4.0-<os>-<arch>/` directory extracted
locally. The tarball layout:

```
agent-center-v2.4.0-darwin-arm64/
├── bin/
│   ├── agent-center
│   └── agent-center-worker-daemon
├── assets/
│   └── default-config.yaml
├── LICENSE
├── README.md
└── install            -> bin/agent-center        (symlink)
```

`./install center` and `./install worker` are the only two commands
you need to know.

Mac is fully verified for v2.4. Linux installs the systemd unit and
should work, but Mac was the acceptance target — Linux validation is
deferred to v2.5.

## § 2. Journey A — install the center

`cd` into the extracted tarball directory, then:

```bash
$ ./install center
agent-center install center:
  prefix:        ~/Library/Application Support/agent-center
  user-mode:     true
  web port:      7100
  state:         fresh
✓ AgentCenter v2.4.0 installed
✓ launchd service loaded (auto-start enabled)
✓ Database initialized at ~/Library/Application Support/agent-center/data/agent-center.db
✓ Bootstrap admin token saved to ~/Library/Application Support/agent-center/data/bootstrap_token
✓ Web Console: http://127.0.0.1:7100

Open the URL above to continue setup.
```

End-to-end: ≤ 30 seconds on a warm machine.

### 2.1 Flags

| Flag | Default | Meaning |
|---|---|---|
| `--prefix=<path>` | OS-dependent (see below) | Install prefix |
| `--user-mode` | `true` on Mac, `false` on Linux | User-mode install (no sudo, no systemd unit at /etc) |
| `--port=<n>` | `7100` | Web Console listen port (loopback only) |
| `--tcp-listen=<host:port>` | empty | Open admin TCP listener for cross-host workers |
| `--dry-run` | `false` | Print planned actions without mutating state |

Defaults for `--prefix`:
- **Mac, user mode**: `~/Library/Application Support/agent-center`
- **Linux, user mode**: `~/.local/share/agent-center`
- **Linux, system mode** (`--user-mode=false`): `/opt/agent-center`

### 2.2 Single-host vs multi-host

If you only need workers on this same Mac, the default (`--tcp-listen=""`)
is enough — workers will use the local unix socket.

For workers on **other machines**, pass `--tcp-listen=0.0.0.0:7300`
so the admin endpoint accepts TCP+TLS. The Web Console Modal will
include the right `--bootstrap=tcp://...` URL in the install command
it generates.

## § 3. Journey B — add a worker

### 3.1 From the Web Console

1. Open `http://127.0.0.1:7100/fleet`.
2. Click **+ Add Worker** (top-right).
3. The Modal shows a one-line install command with a 30-minute,
   one-time-use enroll token already baked in. Click **Copy**.
4. On the worker machine, `cd` into that machine's extracted
   `agent-center-v2.4.0-<os>-<arch>/` directory and paste:

   ```bash
   $ ./install worker \
       --bootstrap=tcp://abc1...:7300 \
       --token=acat_xxxxxxxxx
   ```

5. The Modal flips to **Worker connected** with the new worker's id +
   capabilities; the Fleet table row briefly highlights green. If
   the Modal was closed, a bottom-right toast appears instead.

The minted token is shown **once**. The server stores only its
sha256 hash (per
[ADR-0037](../decisions/0037-loopback-only-web-console.md) / v2.4-A3).
The token burns on first use; if someone uses it before your worker
does, the Modal transitions to the **Token used** state.

### 3.2 From the CLI directly

If you'd rather not click around the Web Console, mint a token from
the host that's running the center:

```bash
$ agent-center admintoken mint-enroll --ttl=30m
{
  "id": "...",
  "plaintext": "acat_xxxxxxxxx",
  "expires_at": "2026-05-26T13:00:00Z"
}
```

Then run `./install worker --bootstrap=tcp://<fp>@host:7300 --token=...`
on the worker box.

### 3.3 Worker `install worker` flags

| Flag | Required | Meaning |
|---|---|---|
| `--bootstrap=<url>` | yes | Admin endpoint to dial (`tcp://<fp>@host:7300` or `unix:/path`) |
| `--token=<token>` | yes | One-time enroll token (from the Modal or `mint-enroll`) |
| `--worker-id=<id>` | no | Defaults to OS hostname |
| `--capabilities=<csv>` | no | e.g. `claudecode,fakeagent` (auto-detected if omitted) |
| `--prefix=<path>` | no | Same default as `install center` |
| `--user-mode` | no | Same default as `install center` |
| `--dry-run` | no | Print planned actions without mutating state |

## § 4. Journey C — upgrade in place

Same command, new tarball.

1. Extract `agent-center-v2.4.1-<os>-<arch>/` over a different path.
2. `cd` into it, then:

   ```bash
   $ ./install center
   agent-center install center:
     state:         upgrade (current=v2.4.0, this=v2.4.1)
   ✓ migrated DB to schema version 30
   ✓ swapped current → v2.4.1
   ✓ launchd service restarted
   ✓ health probe passed
   ```

The upgrade is **atomic**: the new version is laid down at
`<prefix>/versions/v2.4.1/`, the schema migration runs, and then
`<prefix>/current` is flipped via a POSIX `rename(2)` to the new
versioned directory. The service is restarted; the installer probes
the health endpoint, and if the probe fails it **automatically rolls
back** the symlink to the previous version and restarts.

Worker upgrades work the same way (`./install worker --bootstrap=... --token=...`
against the same existing worker — re-uses the same prefix, same
worker-id; the token can be a fresh enroll-token from a new Modal
session or an existing long-term token).

### 4.1 Rollback behavior

| Step that fails | Auto-rollback? | Operator does |
|---|---|---|
| Copy binaries into `<prefix>/versions/<new>/bin/` | n/a — symlink not flipped yet | Re-run `./install center` after fixing disk/perm |
| DB migration | n/a — symlink not flipped yet | Re-run; migration is idempotent. If it's stuck on a specific migration step, see § 6 troubleshooting |
| Atomic symlink swap | n/a — POSIX `rename(2)` is atomic | n/a |
| Service restart | yes — symlink flipped back to old version, service restarted | Check logs at `<prefix>/data/server.log` and the underlying error printed by the installer |
| Health probe (5s timeout) | yes — same as above | Same |

### 4.2 Versions kept on disk

Old versions are retained at `<prefix>/versions/<old>/` so the
auto-rollback has something to point at. The installer doesn't yet
garbage-collect; cleanup is a manual `rm -rf <prefix>/versions/<old>/`
when you're confident you don't need to roll back further.

## § 5. Idempotency & dry-run

- Running `./install center` twice with the **same version** is a
  no-op: `state: same_version` → `✓ AgentCenter v2.4.0 already
  installed, no changes`.
- `--dry-run` prints the detected state + planned actions without
  changing anything. Use it to preview a tarball before committing.

## § 6. Failure modes

The installer maps common syscall errors to friendly messages with
a hint and the underlying error. The pattern:

```
Disk full: cannot write to /var/lib/agent-center

What to try:
  Free up space on the target filesystem, or pass --prefix=<path>
  to install somewhere with more room.

Underlying: no space left on device
```

Coverage:

| Symptom | Friendly message | Hint |
|---|---|---|
| Port 7100 already in use | `Port 127.0.0.1:7100 is already in use` | `Use --port=<n> to pick a different port` |
| Disk full | `Disk full: cannot write to <prefix>` | `Free up space on the target filesystem, or pass --prefix=<path>` |
| Permission denied writing systemd unit | `Permission denied writing /etc/systemd/system/agent-center.service` | `Run with sudo, or use --user-mode for a per-user install` |
| Permission denied writing binary | `Permission denied writing <prefix>/versions/.../bin/` | `Use --prefix=<path-you-own>` |
| Enroll token already used | (Modal: **Token used**) | Mint a new token from the Modal — the old one only works once |
| Enroll token expired (30 min) | (Modal: **Token expired**) | Re-mint; the worker side will report `401 auth_expired` if you race |
| Worker can't reach center | `connection refused` / `connection timed out` | Check firewall; verify `--tcp-listen` is set on the center; verify `--bootstrap` matches |
| Worker fingerprint mismatch | `fingerprint mismatch dialing <addr>` | Cert rotated, or MITM. Re-fetch fingerprint from the center via a trusted channel; if it really changed, paste the new value into the Modal-generated command |
| Upgrade health probe failed | `health probe failed: <reason>` | Installer auto-rolled back. Inspect `<prefix>/data/server.log` for the new-version crash |
| DB migration failed | `migration 0029 failed: <reason>` | Old binary is still on disk; investigate, re-run when fixed. Migrations are idempotent |
| Both `--user-mode` and `--prefix=/opt/...` | "permission denied" downstream | Pick one mode; system-mode prefixes need `sudo` |

## § 7. What's still NOT in v2.4

Deferred to v3 deployment-as-product (or earlier minor releases):

- Tarball distribution (where the user gets the tarball from) — v2.5+
- CA-signed cert / Let's Encrypt — v3
- mTLS — v3
- Per-IP allowlist (CIDR) — v3
- Hot token rotation (no worker restart) — v3
- Container deploy (Docker / k8s) — v3
- `agent-center diagnose` self-check tool — v3
- New SSE events `worker.enroll_attempt_failed` + `admintoken.expired`
  — deferred (audit:
