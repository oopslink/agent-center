# Agent Center v2.7 Release Notes

**Release date:** 2026-06-03
**Upgrade path:** v2.6.x → v2.7.0 (fresh install required; see migration guide)

---

## Highlights

### Domain refactor — a clean execution model

v2.7 lands a full domain refactor: the legacy task-runtime / discussion /
old-tasks-issues-projects schema is retired and replaced with a clean set of
bounded contexts — ProjectManager (Project / Issue / Task), Agent (Agent +
lifecycle), Environment (Worker / Agent supervisor / file transfer sessions),
Conversation (one URI scheme across channels / DMs / task / issue conversations),
and Files. Real-execution agents now flow through a dedicated AgentWorkItem
control stream end-to-end. ~26,000 lines of legacy code were removed in the
carve-out.

### Agents that survive worker-daemon restarts

Each agent is now hosted by a per-agent supervisor process that detaches from
the worker daemon. A worker daemon upgrade or restart no longer interrupts a
running agent — the daemon re-attaches to the live supervisor across the gap,
output is durably buffered, and ack-offset state survives daemon bounce.
A genuine claude-process death falls back to Mode-B: reap-then-relaunch with
session resume (no work loss, no silent failure). See the survive-reattach
behavior spec at
[docs/design/requirements/survive-reattach-product-behavior.md](../design/requirements/survive-reattach-product-behavior.md).

### Real-time control: stream-first, poll fallback

Worker control commands (start / stop / reset / work) are now delivered over a
long-lived SSE control stream feeding the same monotonic
`WorkerControlEvent` log. On disconnect or heartbeat-timeout, the daemon falls
back to poll over the same offset cursor — no duplicates, no gaps. See ADR-0050.

### Worker capability auto-discovery

Every worker probes the locally-installed agent CLIs (`claude-code`, `codex`,
`opencode`) on each `online` event and reports the result to the center
(version, detected/enabled flags, MCP/skills/session support). The Environment
worker card surfaces what's installed. The `--capabilities` flag is gone.

**v2.7 execution caveat:** only `claude-code` is dispatched at runtime — the
`codex` / `opencode` adapters are discovery-only (`ErrNotImplemented` execution
stubs); the per-CLI dispatch layer is the v2.8 feature tracked as #180. Agent
creation validates `cli ∈ {"claude-code"}` and the UI shows only `claude-code`,
so an operator can't accidentally create a non-runnable agent.

### Conversation attachments — uploaded, downloaded, safely

Conversation messages can now carry file attachments end-to-end: upload via the
files BC, attach to a `sendMessage` (atomically validated), and download via a
new per-message route. Cross-org and non-existent URIs return byte-identical
`403`s on both attach and download (no existence oracle); a participant
removed from the channel loses download on the same edge as their
participation. `scope=uploader` cannot be set by the client (server-derived
only).

### Web Console redesign

- Sidebar Information Architecture: ORGANIZATION group renamed `Members`
  (Humans + Agents — one entry); SYSTEM group is now `{Environment, Settings}`.
- The legacy Fleet page is merged into Environment (worker cards + operational
  data + Add/Install modals in one place); `/fleet` 302s to `/environment`.
- Adding an Agent to your organization (Members → Add Agent) creates the
  execution Agent + organization Member in one atomic step; clicking the
  resulting Member row opens the Agent detail / lifecycle page.
- Channel participant invite is a search + multi-select modal (Human/Agent
  badges); previously-removed participants reappear as candidates and can be
  re-invited.
- All copy spelled out (`Organization` not `Org`); the entire UI is English.
- All confirmation dialogs are in-app modals; native
  `window.confirm/alert/prompt` are banned (enforced via ESLint in `make lint`).
- Connection status flips to `live` on first frame instead of after the first
  30s heartbeat.

### Identity flows that actually work

- Fresh install web console first-screen routes correctly:
  uninitialized → `/signup`, initialized-but-not-logged-in → `/signin`
  (via a new public `GET /api/auth/bootstrap` endpoint), instead of the
  prior `401 JSON` everywhere.
- The web console reads the real session identity per-request — channel
  ownership / Invite buttons / sender display / read-state all stamp the
  authenticated user, not a configured default.
- Message senders and participants render the real display name in the UI,
  not raw `user:<id>` / `agent:<id>` refs.

## Upgrade notes

### Fresh install required

v2.7 is a clean-schema release: a v2.6 → v2.7 in-place upgrade is not
supported. Back up first, then:

```bash
# stop and remove the previous install
agent-center uninstall center
launchctl bootout gui/$(id -u) ~/Library/LaunchAgents/com.agent-center.center.plist 2>/dev/null || true
for p in ~/Library/LaunchAgents/com.agent-center.worker.*.plist; do
  [ -e "$p" ] || continue
  launchctl bootout gui/$(id -u) "$p" 2>/dev/null || true
  rm -f "$p"
done
rm -rf ~/.agent-center            # data is dropped — back up first

# install v2.7
agent-center install center
```

On first browser visit you will see `/signup` (the bootstrap endpoint returns
`{"initialized":false}`, the SPA routes you to signup). Complete signup to
create the first Identity and Organization.

### macOS 26 (Darwin 25) install fix

`install` / `upgrade` activate the launchd unit via `launchctl
bootout`/`bootstrap` (the modern API). On macOS 26 the legacy `launchctl load`
was removed, which made the v2.6 / v2.5.x `install` silently fail to start the
service. v2.7 installs cleanly on macOS 26.

### Default `server.listen_addr` is `:7050`

Was `:7000` (vestigial / never bound, but the install banner printed it). The
new default avoids confusion with macOS AirPlay Receiver. Custom installs that
overrode the value are unaffected. The field itself is removed in v2.8 (#174).

### `install worker --worker-id` is now required

`install worker` no longer falls back to `hostname` for `--worker-id`. Use the
Web Console "Add Worker" flow (which generates a unique id) and copy the full
command — that is the supported path for multi-worker-per-machine. The
`list-local-workers` discovery command is deferred to v2.8 (#170).

### CLI commands removed

`agent-center agent / secret / channel / conversation / project / message /
inspect / query / ps / stats / logs / peek-trace` are removed. Use the Web
Console or REST API instead. The operational CLI subset is preserved:
`server`, `version`, `worker` (+ `run` / `mcp-host` / `agent-supervisor` /
`shim`), `install`, `uninstall`, `upgrade`, `migrate`, `bootstrap`, `admin`,
`help`.

### Agent claude credentials — subscription works out of the box

If you've signed into the desktop Claude Code with `/login` on the worker
machine, agent-managed claude processes use the same subscription
automatically — no additional env required. The supervisor's claude argv is
`--setting-sources user,project`: the **user** source carries your keychain
`/login` credential, the **project** source loads
`<agent-home>/workspace/.claude` (created empty per agent — drop your own
`settings.json` there to customize a specific agent).

**Operator settings inheritance.** Loading the user source also loads the
operator's own `~/.claude` settings — concretely, **hooks**, **plugins**, and
non-secret **env vars** are inherited by every agent's claude (which runs in
`bypassPermissions` mode). MCP servers are NOT inherited (the supervisor pins
MCP via `--strict-mcp-config` + a per-agent generated `--mcp-config`).
Operators with sensitive user-level `SessionStart` hooks should be aware that
those hooks will execute under `bypassPermissions` inside agents. Full
operator-isolation via a headless OAuth token
(`CLAUDE_CODE_OAUTH_TOKEN` from `claude setup-token`) is the v2.8 path.

### Worker model is one model (workforce)

The Environment page's worker list reads `workforce.Worker` (the same model
used by the Fleet projections), and `environment.Worker` no longer carries a
duplicate `organization_id`. The control-channel ack-offset state still lives
on `environment.Worker` (load-bearing for resume / survive-reattach); nothing
changed there.

## What's not in v2.7

The following items are deferred to v2.8 with explicit triggers / tracking
tasks:

- **Per-CLI execution dispatch (#180)** — `codex` and `opencode` discovery
  works today, but the runtime supervisor argv is `claude-code`-shaped only.
  Until #180 lands, only `claude-code` agents actually execute.
- **Single-machine multi-worker tooling (#170)** — `install worker` already
  isolates per-`worker-id` (data dir, launchd label, log dir), but
  `list-local-workers` and `uninstall worker --name` / `--id` discovery
  commands are pending. Center installs on the same machine still share a
  single global launchd label — for v2.7, run only one center per machine.
- **Operator-isolated agent auth (`CLAUDE_CODE_OAUTH_TOKEN` allowlist)** —
  see Upgrade notes; the env-allowlist carve-out for the headless token is
  the v2.8 path to a clean operator-settings boundary.
- **Vestigial `server.listen_addr` field removal (#174)** — the field is
  still validated-required (with the new `:7050` default) but is genuinely
  unused; the struct/CLI/YAML/install-template cleanup is mechanical and
  parked for v2.8.
- **Per-agent-home `workers/<worker-id>` double-nesting (#179)** — already
  fixed in v2.7. (Listed here because the task #179 framing predated the
  v2.7 fix.)
- **Agent terminal-fail → queued WorkItem reassignment** — see Notes /
  Compatibility in CHANGELOG.
- **SQLite 517 retry beyond `RunInTx`** — addressed in v2.7 inside
  `RunInTx` (#149); broader read-modify-write call sites that don't use
  `RunInTx` will be audited later if needed.

See CHANGELOG.md `[Unreleased]` → **Notes / Compatibility** for the full
deferred-with-trigger list (Mode-B WorkItem-id rebind, agent-supervisor
RPC protocol, stderr capture, fork-generation cleanup, etc.).
