# v2.9 Acceptance Plan

This plan is written from the **product-feature perspective** — it describes the product capabilities v2.9 delivers and the observable acceptance criteria for each, validated **black-box from product intent**, independent of implementation or development progress. A capability is accepted when its criteria are demonstrably true on a real instance, regardless of how or when it was built.

## Acceptance Methodology

Each capability is validated through independent faces, with a verify-not-trust posture (observe the real behavior, never trust that "it should work"):

- **Run-real (UI/UX + end-to-end):** exercise the capability on a real instance with a real agent; measure computed truth (actual rendered text, actual network calls, actual DB/event state). Headline behaviors are accepted only when demonstrated end-to-end run-real.
- **Data/API (deterministic):** black-box from the product spec — assert the API/domain behavior and its inverse (the rejection paths), without consulting the implementation.
- **Isolation & negative cases:** cross-org / unauthorized / archived / precondition-violation paths are explicitly exercised and must be denied.
- **Regression nets:** behaviors that span layers (event → projector → wake; tool registration; route registration) are guarded so a future change can't silently break the whole class.
- **Both-mode visual:** contrast (AA) verified in both light and dark; every page reachable via real navigation.

## 1. Plan Orchestration

**Capability — author a plan as a task DAG.** A user can create a plan, add backlog tasks as nodes, and wire dependencies between them while the plan is in draft.
- Acceptance: create plan → add tasks → add/remove dependency edges; dependency editing is allowed only in draft; a dependency that would form a cycle (or a self-edge) is rejected; the DAG renders with synthetic start/end nodes and a correct per-node derived status.

**Capability — the system auto-executes a started plan.** When a plan is started, the system advances the DAG and dispatches each ready task to its assigned agent automatically, with no manual stepping, until the plan completes.
- Acceptance (headline, run-real): start a plan → ready nodes are dispatched to assigned agents → agents do the work → the plan advances node-by-node to DONE with zero manual advance.
- Acceptance: stopping a running plan returns it to draft for editing; a failed node blocks its subtree and the plan's creator is notified/woken to handle it.

**Capability — an agent can build and run a plan programmatically.** A PM-style agent can, through its own tools, create a plan, add tasks, wire dependencies, and start it — and the system then auto-executes as above.
- Acceptance (headline, run-real): an agent discovers the plan tools, calls them to assemble a plan, starts it, and the plan runs to completion — i.e. the agent's tools are actually reachable and functional, not merely defined server-side.

## 2. Plan Lifecycle — Delete & Archive

**Capability — delete a plan.** Deleting a plan unloads its tasks back to the Backlog, removes the plan's bound conversation, and deletes the plan.
- Acceptance: after delete, the plan's tasks appear unplanned in the Backlog; the plan's conversation is gone; the plan no longer exists. Deleting a plan does not delete the underlying tasks.

**Capability — archive a plan (irreversible).** Archiving freezes a plan and all its tasks into a terminal archived state.
- Acceptance: archive is rejected while the plan is running (must be stopped/finished first); archive is rejected while any task in the plan is still running (finish/stop the task first); a successful archive marks the plan and all its tasks archived (orthogonal to task status); an archived plan is read-only and cannot be re-archived; an archived plan is still viewable.

## 3. Agent Wake & Collaboration

**Capability — humans reach agents through conversation.** @mentioning or directly messaging an agent wakes it to respond.
- Acceptance (run-real): a human DM to an agent wakes it and it replies; @mentioning an agent in a plan / issue / task conversation wakes it and it replies — and this works for any agent in the project, not only those already participating in that conversation.
- Acceptance (negative): system- or agent-authored messages never spuriously wake an agent (only human-authored messages trigger conversational wake) — auto-dispatch is a separate work-delivery path, not a chat wake.

## 4. Projects & Archival

**Capability — manage projects and their members/work.** Create, list, and edit projects; manage members, issues, and tasks within a project.
- Acceptance: standard CRUD on projects and their children behaves as expected within the owning org.

**Capability — an archived project is read-only.** Once a project is archived, it cannot be modified.
- Acceptance: every mutation on an archived project or its children — create/edit/delete task or issue, assign/transition task, add/remove member or change role, create/edit/start a plan in it — is rejected; reads (viewing the project and its contents) still work. (A plan that was already running when its project was archived may wind down — stop/advance — but no new work can start and nothing can be edited.)

**Capability — archived projects don't clutter the default views.** Archived projects are excluded from the default project list and surfaced separately.
- Acceptance: the default project list and the sidebar show only active projects; an explicit "archived" view returns archived projects; a direct project read still works for an archived project.

## 5. Work Board & Plan UI

**Capability — operate plans visually.** The Work Board and plan detail give a complete, legible operating surface.
- Acceptance: DAG visualization with start/end nodes and 6-state node status; edit dependency edges and remove tasks from a plan (in draft); drag a task freely across plans and back to the Backlog; open a task's detail in a new tab; resize the DAG↔chat split; the plan delete/archive entries appear only when applicable and a confirm dialog spells out the (irreversible) consequences; an archived task shows an "Archived" badge wherever it appears (board card, DAG node, task list).

**Capability — people are shown by name.** Assignees and identity references are shown as readable names, not raw ids.
- Acceptance: wherever an assignee/identity appears in visible content, the agent/user name is shown (raw ids remain only in functional positions like links/keys/payloads).

**Capability — the agent list stays current.** Creating an agent reflects immediately in the list.
- Acceptance: after adding an agent, the agent list shows it without a manual refresh.

## 6. Org Isolation & Explicit Context

**Capability — strict org isolation.** Org-scoped resources are accessible only within their owning org.
- Acceptance (negative): accessing a resource of one org under a different org's scope is denied without leaking the resource's existence; switching org shows only the current org's data; cross-org access never returns another org's data.

**Capability — org context is explicit, never implicit.** Which org a request targets is explicit in the resource path, so the same path can never return different data depending on hidden state.
- Acceptance: org-scoped endpoints are addressed under an explicit org path; there is no implicit "current org" inferred from session/query that could make a list silently empty or wrong.

## 7. Access & Security

**Capability — privileged state-changing actions require authorization.** Revoking a worker enrollment token requires an authenticated caller who belongs to the token's org.
- Acceptance (negative): an unauthenticated revoke is rejected; a revoke by a non-member of the token's org is rejected; a legitimate org-member's revoke succeeds.

## 8. Visual & Navigational Quality

- Acceptance: text/badge contrast meets AA in both light and dark modes; every list and detail page is reachable through real navigation (no orphan routes, no page that fails to populate).

---

## Acceptance Gate (release-ready)

v2.9 is acceptance-ready when:
1. Every capability above is demonstrated true on a real instance (the headline plan-orchestration, agent-builds-plan, and agent-wake behaviors via run-real).
2. All negative/isolation/security criteria are demonstrated denied (cross-org, archived, unauthorized, precondition-violation).
3. Both-mode AA and full page reachability hold.
4. No observable P0/P1 product defect in any capability.

(Implementation status, PR tracking, and per-change review are maintained separately as development tracking — they are not part of this product-acceptance plan.)
