技术方案要点（详见 #并发讨论2 / 技术方案）

  • 采集(worker):监工 engine 的 executor pool 可即时快照——Pool.Handles()(μs 级锁)+ orphans map 合并;每 executor 出 {executor_id, task_id, cli, model, started_at, state, pid, last_progress_at}。需小改:Handle 加 `startedAt` 字段(5 行);task/model 从 input.json 读或随 Handle 缓存。
  • 传输 = 快照随心跳 + 自适应节奏(已定):心跳体扩展可选字段 `agent_concurrency_snapshots`(向后兼容)。自适应心跳:有活跃 executor 时心跳加快到 ≈3–5s,空闲回落 30s(仍是"快照随心跳",只在并发真正发生时高频,流量有界)。
  • 展示约定:UI 一律显示 task ref(T123),不显示 raw task-id;快照/接口仍传 `task_id`,Center/FE 解析为 ref 展示。
  • Center 存储:内存 ephemeral cache(map[agent_id]Snapshot + received_at,TTL=2×心跳),抽 LiveStateStore 接口(in-memory 默认实现,后续可换落库/时序)。心跳 handler 写入。
  • cap/queued = Center 侧:cap 在 enforceConcurrencyCap(assign_flow)派发侧限流;queued = 该 agent open/assigned 未跑未 blocked 的 task 数。无需 worker 提供 queue。
  • 读取/视图:新 GET /api/agents/{id}/concurrency 读 cache + 拼 cap/queued;web `useConcurrencySnapshot()` 轮询 3s;agent 详情新增「Concurrency」标签。stale/单活/orphan/离线四态降级。
  • 范围:单 agent。org/fleet 总览 → issue-ea949e06(后续 Insight 模块)。
