设计要点

  • 顶部 Conversations 分段控件（Unread | Channels | DMs）常驻页内，不需要点 ☰ 才能切换来源——高频操作优先于严格遵循导航框架的抽屉范式。
  • Unread 行用 来源标签（Channel/DM/Task 三色）+ 加粗发送人名 + 单行预览，@ 提及用红色 @ 角标区分于普通未读数字角标（紫色）。
  • ChannelDetail 消息气泡：他人消息浅色卡片靠左、自己消息品牌紫靠右，符合常见 IM 视觉习惯；顶部 Chat/Threads/Files/People 子分段对应 PC 端消息+线程+文件+参与者的信息密度。
  • 头像统一 38px（列表）/ 26px（消息气泡）/ 24px（顶栏），色板取品牌色系轮换（紫/绿/棕）区分不同人，不用随机色避免视觉噪音。
  • 消息流按天插入日期分隔线（今天/昨天/具体日期），跨天滚动时始终能一眼定位是哪天的消息，不需要逐条看时间戳。
  • 消息操作（复制/引用/开 Thread）走长按触发的悬浮操作条——贴靠在被长按消息旁，避免像桌面 hover 那样占用寸土寸金的移动端屏幕空间；被操作的消息加高亮描边表明当前操作对象。
  • 有讨论串的消息下方挂一个"N 条回复" thread pill，点击进入该消息的 Thread 详情（复用 Threads 子分段的展示逻辑，不是新开一套 UI）。Thread 详情页根消息钉在顶部浅底卡片区分于回复列表，顶栏显示所属频道。
  • DMs 列表和 DMDetail 顶栏头像都挂在线状态点：绿=在线、灰=离线、紫=Agent（Agent 视为常驻在线，颜色与人类的绿区分）。
  • DMs 内部有三类会话（对应 PC 端 dm_type：默认 / agent_agent_dm / system_dm），移动端用 Channels/DMs 分段下的第二层 subtabs（我的 / Agent 互聊 / 系统）承接，不合并成一个列表——三类的操作权限不同（Agent 互聊对人类只读旁观）。
  • Unread 页补齐 All/Mentions/Unread 三个筛选 chip（各带实时计数）+ 全局"全部已读"按钮——PC 端本来就没有单条标记已读，移动端不额外发明。
  • Channels/DMs 列表的 Archive/Delete 是独立于整行点击的操作入口（右侧小图标），不是把它们并入"点整行"；已归档频道单独收进折叠分组，懒加载、只读、无操作按钮。
  • 详情页头部补齐 Follow/Unfollow（星标图标）；DMDetail 额外有"更多"溢出菜单（当前仅一项：复制链接）。
  • 消息流顶部有"加载更早的消息"（游标分页），滚动到底部之外时右下角悬浮"跳到最新"圆按钮，新消息到达时额外弹出"↓ N 条新消息"胶囊——两者互斥出现，对应 PC 端两种不同触发条件。
  • Composer 补齐附件入口（相册/文件/拍照，对应 PC 端拖拽/粘贴/多选上传合并为一个入口）与 @/# 提及提示（含 @all 广播）。

功能覆盖清单（本批 mockup 未逐屏画出、但已在 PC 端确认存在，写入 spec 时需逐项定移动端方案）

  • Participants 面板（ChannelDetail 的 People 子分段）：owner 专属的邀请成员 / 移除成员操作，普通成员不可见——移动端需要一个"仅 owner 可见"的入口态设计，本批未展开。
  • 系统消息折叠：agent 执行失败通知、系统提醒消息默认折叠为一行 + [详情] 展开，消息流里穿插出现，需要单独的紧凑折叠态样式。
  • Maximize/restore 切换：现有实现里移动端有一个"全屏/还原"聊天框切换（Esc 还原、锁 body 滚动）；如果新导航框架下详情页本身已经是全屏，这个开关可能不再需要——需要显式决策，不能默默丢弃也不能默默保留。
  • 头像堆叠三处实现（ChannelDetail 头部 / Participants 面板 / Channels 列表行）目前 PC 端是三套独立实现，移动端重新设计时建议统一成一个组件，避免重复三次维护成本。
