设计要点

  • System 顶层 tabstrip：Environment / Secrets / Reminders / Settings 四项——Reminders 本身有自己的底部 Tab 入口（Remind），但在 System 内部也留一个平级 tab，因为 PC 端两者本来就是同一 col① rail 下的并列项，不强行只留一个入口。
  • Force Delete 保留了输入名称二次确认，没有为了"移动端操作更快"而降级成普通 Yes/No——这是全模块最强的安全设计，触屏误触概率比鼠标更高，反而更需要这层保护，不能因为是移动端就削弱。
  • Secrets 列表把现有移动卡片丢掉的 kind/state/创建时间字段全部找回来；创建流程严格保留"提交后立即清空明文、永不提供查看入口"的 ADR-0026 约束，mockup 里特意写了一句说明防止实现时手滑加个"眼睛图标"。
  • Reminder 的三种触发类型（Cron/单次/事件）用顶部三段切换承接，Cron 保留预设 chip + 人类可读预览（对应桌面版的"这是什么时候触发"直觉展示）；"跳过重叠"和"结束条件"明确标注"仅创建时可设"，防止实现时误做成可编辑。
  • Reminder 详情页的内容区保留了任务/计划/issue 引用的富文本可点击链接（图中 #T-501 是可点的），不能退化成纯文本——PC 端复用的是聊天消息同一套 markdown+linkify 管线，移动端也要复用第二批已定案的消息渲染规则，不要另起一套。

功能覆盖清单（未逐屏画出，写 spec 时逐项定案）

  • Worker 安装口令的复制/分享方式：手机上复制一段命令去粘贴到目标机器不现实，需要专门设计（二维码 / 系统分享面板 / 发送到桌面端），本批只标出了入口，具体方案留后续。
  • Reminder Cancel 目前在 PC 端没有二次确认（与 Delete 不一致）：移动端触屏误触率更高，是否要给 Cancel 补一个轻量确认/撤销条，需要显式决策，不能照搬 PC 端的"无确认"直接用在移动端。
  • Reminder Range 筛选和搜索（PC 端由页面外的侧边栏驱动，桌面折叠导航在移动端本来就不可达）：本批用筛选 chip 里的"范围"入口代表，具体搜索框交互留后续。
  • Worker Agents tab 的逐行 Restart 操作、Environment 页的 Activity 四 tab 汇总流（All/Work Items/Issues/Transfers）：本批未逐屏画出，入口需要在实现阶段补齐，Transfers 已有的"移动端卡片/桌面表格"双布局是可复用的既有模式。
  • AddWorkerModal（新建 Worker 表单）：入口已标（顶栏 + 号），表单细节留后续。
