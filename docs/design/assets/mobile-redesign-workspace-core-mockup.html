设计要点

  • Projects 列表卡片：常规态展示 4 个高频快捷操作（Edit/Board/Tasks/更多），卡片信息密度高时收进"⋯"菜单——直接复用 PC 端已有的"≥md 内联图标行、<md 收进 kebab 菜单"这个既有响应式断点策略，而不是重新发明。
  • ProjectDetail 用统计磁贴（4 个数字块）替代桌面的"点击跳 tab"语义——磁贴本身可点，效果一致；下方 tabstrip 横向可滑动承接 Issues/Tasks/Plans/Repos/Members 五个 tab，不裁剪任何一个。
  • Issue/TaskDetail 默认"聊天优先"：状态横幅 + 聊天区占满屏幕，信息折叠进 ⓘ 触发的底部抽屉——这是移动端对 PC 桌面版"默认展开描述 + 常驻侧栏"的独立重设计，但抽屉里的信息颗粒度（描述/标签/项目链接/派生任务/审计时间线）跟桌面侧栏一一对应，一条不少。
  • TaskDetail 独有 Stuck 横幅（agent 卡在某个决策点等待人工介入）——这是 agent 编排的核心语义，用琥珀色高优先级横幅常驻在状态栏下方，不折叠进抽屉，因为它要求用户立即注意。
  • Assignee 用不同颜色圆点区分 agent（紫）/human（青）+ 头像内嵌类型图标，与 Conversations 模块的头像色板风格统一但语义不同（这里颜色代表"谁在负责"，不是"在线状态"）。
  • Composer 占位文案明确写"会唤醒对应 Agent"——提醒这不是普通评论区，发消息是真实的 agent 控制动作，这是本产品的核心差异化语义，移动端必须保留这个提示强度。
  • Work Board 独立重设计：PC 端是横向多列 kanban + 鼠标拖拽，桌面版对移动端的既有妥协是"横向滚动 + scroll-snap + 长按拖拽 + 竖屏旋转提示"（本质是把桌面横向结构硬塞进小屏）。这批改为单列竖排 + 顶部列选择 chip 横滑切换：一次只看一列，卡片竖排更好读更好点；移动卡片不用长按拖拽跨列（小屏精确拖拽体验差、容易误触），改成卡片上的"移动"图标 → 底部弹层选目的列——这个弹层本质是独立重设计版的 PC 端"Add to plan"键盘可达下拉菜单，两端殊途同归，都不依赖鼠标/触控的精确拖拽。
  • 锁定的结构化 Plan 列（运行中/已完成/已归档）用横幅 + 卡片降低透明度表示只读，移动弹层里对应选项直接标"已锁定"且置灰不可点，而不是点了之后再报错——错误提前到操作前，而不是操作后。
  • Claimable/Starved 是纯 agent 编排概念（人类 PM 工具没有对应物），用色块 pill 强区分：绿色 Claimable（已派发等待认领）、红色 Starved（没有满足能力要求的在线 agent）——移动端沿用桌面的颜色语义，不重新定义。

功能覆盖清单（未逐屏画出，写 spec 时逐项定案）

  • 已归档 Projects 分组：懒加载折叠列表，已在①用 disclosure 占位，具体行内容留实现阶段。
  • Project 编辑弹层（含 auto-assign 主开关）、创建 Project/Issue/Task 弹层：本批未画表单细节，只在①③④标出入口。
  • Plans tab（只读表格，不能在这里建 Plan——建 Plan 只能在看板做）：入口已留，需要在实现阶段明确"只读"这条约束不能丢。
  • PlanDetail 的 DAG 编辑（依赖边增删，draft 计划专属）：本批只画了只读的 stepper 展示态，编辑交互（加/删依赖边）留给 Task 列表 tab 内的表单式编辑，具体表单本批未展开。
  • WorkItemFilterBar（状态多选/assignee 单选/创建-更新时间范围）：② 顶部"筛选"入口已标注计数徽标，具体弹层内容留后续。
  • Follow/unfollow、Maximize、Threads/Files 子面板：与 Conversations 模块共享 WorkItemConversation，复用已经在上一批定案的规则，不重新设计。
  • Work Board 的"新建 Plan"表单（名称/描述/目标日期）、"新建 Task"目的地选择弹层（Backlog/Pool/某 draft Plan）：本批只标了入口（＋新建 Plan 列、FAB），表单内容留后续。
  • 长按拖拽是否作为"移动到"弹层之外的可选增强：PC 端已有 useBoardTouchDrag.ts 的长按拖拽实现，本批的独立设计默认不复用它（弹层是主交互），但不排斥实现阶段把长按拖拽作为同列内排序等场景的锦上添花，需要显式决策。
