设计要点

  • Settings.tsx 是系统级配置（界面语言 + 唤醒护栏五项数值参数），不是用户账号设置——命名容易让人以为是"我的账号"，mockup 标题里特意点明，避免实现时塞错内容。
  • OrgSettings 的 Humans section 是独立于第四批 MembersHumans 的另一套更简化的实现（纯表格，无行内菜单/popover），需要单独设计，不能假设第四批的移动端工作已经覆盖它；加了一句说明性 banner 帮用户理解"这里是组织级角色管理，日常查看去 Team 标签"，避免两个相似页面造成认知混淆。
  • Agents section 是原样复用 Agents.tsx，mockup 里用一句说明代替重复画面，不重复设计。
  • Invitations 的 Cancel（无确认）和 Delete（有确认）延续 PC 端的既有不对称——Cancel 只是撤回一个待处理邀请（影响小），Delete 会清除记录，风险不同，保留这个区分是合理的，不强行拉平。
  • 删除组织的确认强度做了主动加强：PC 端目前只有单击 ConfirmModal，本次独立设计把它提到与 Worker Force Delete 同等（甚至因为影响范围更大而更值得强调）的"输入组织名称"二次确认。这是本批唯一一处刻意不照搬 PC 端现状、而是主动补强安全设计的地方——因为审计发现这是全系统影响面最大的操作，PC 端现有的单击确认强度本身就不够，独立设计不代表照抄现状的不足。
  • 审计日志卡片保留"即将推出"的占位态而非隐藏——让用户知道这个能力存在但未上线，而不是完全消失显得像被砍掉的功能。

功能覆盖清单（未逐屏画出，写 spec 时逐项定案）

  • 禁用/启用组织的确认弹层（本批画了入口按钮，未画确认弹层本身）：强度维持 PC 端的标准 ConfirmModal 级别即可（可逆操作，不需要像删除组织那样加强）。
  • Danger section 的路由级权限门禁：PC 端组件本身没有显式角色判断（推测由路由/API 层控制），移动端需要在实现阶段确认非 owner 用户是否应该在客户端就隐藏整个 Danger tab，而不是让他们进来了才被 API 拒绝。
  • Org Profile 的 Logo 上传：PC 端本身就是未实现的占位桩，移动端保持占位，不要在移动端"顺手"把 PC 端还没做的功能做全。
