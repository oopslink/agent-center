设计要点

  • Workspace 顶层 tabstrip 从"Projects"扩展到"Projects / Issues / Tasks / Plans / Repos / Templates"六项，横向可滑动——这是 PC 端 col① rail 的 pathPrefixes（projects/issues/tasks/plans/repos/templates）在移动端的统一入口，跟第三批 ProjectDetail 内部的 tabstrip 是同一视觉语言但不同语义层级（这里切的是"跨项目视图"，那里切的是"单项目内的维度"）。
  • 项目筛选器（"全部项目"下拉）只在这批跨项目视图里出现——这是与第三批 ProjectDetail 表格视图唯一的结构性差异（项目内视图项目维度已固定，不需要这个筛选器），mockup 里特意标出来避免复用第三批组件时误加或误删。
  • OrgWorkItems/OrgPlans 点行是只读摘要弹层 + "打开完整详情"跳转按钮，不能就地编辑状态/负责人——这是刻意保留的 PC 端设计（列表本身是只读投影，编辑要进项目内详情页），移动端不要"顺手"加一个编辑按钮。
  • OrgRepos 明确是工作区仓库注册表（增删改仓库元数据 + 只读查看远端 commits/branches），不是 GitHub OAuth 连接流程，也不是"项目引用哪些仓库"（那是第三批 ProjectDetail 的 Repos tab）——两个"仓库"页面职责不同，mockup 标题和文案里都点出这个区分，避免实现时混淆两个概念。
  • OrgTemplates 是纯文本 prompt 模板 CRUD，没有"应用/实例化"动作——跟另一个不相关的 Team Templates（团队组成模板，可导入/导出/实例化）完全是两回事，本批的模板卡片没有"应用"按钮是有意为之，不是漏画。
  • 内置模板（builtin）隐藏编辑/删除入口，只保留查看——与自定义模板视觉区分只用一个"内置"灰色徽章，不用禁用态图标制造额外视觉噪音。

功能覆盖清单（未逐屏画出，写 spec 时逐项定案）

  • OrgWorkItems 的完整筛选弹层（状态多选/项目单选/负责人单选/创建-更新时间范围）：入口已标（筛选 chip + 计数），弹层内容留后续，与第三批 WorkItemFilterBar 的"Deferred"结论一致，此处额外多一个项目筛选项要补进弹层。
  • OrgWorkItems 跨项目创建（先选项目，再委托给项目内的 Create 弹层）：入口已标（顶栏 + 号），二级"选项目"步骤留后续。
  • OrgRepos 的 Add/Edit Repo 表单（label/provider/url/默认分支/凭证，凭证字段留空=保留原值）：入口已标，表单细节留后续。
  • OrgRepos 删除仓库的差异化确认文案（有项目引用 vs 无引用时警示不同）：需要在实现阶段体现"有引用"警示，不能所有删除都用同一句文案。
  • OrgTemplates 的 Add/Edit 表单（name/description/content 多行文本）：入口已标，表单细节留后续；编辑时要求 content 异步加载完成才能保存（避免覆盖为空），这条约束要写进 spec。
