# v2.8 P1 — unread / badge / follow consumer contract (frontend ↔ backend)

> **Status:** day-0 contract (mock=contract, per Dev request, to avoid the #240/#258 mock-drift).
> **Pair:** #264 frontend (Dev2, consumer) ↔ #268 backend (Dev, producer). Anchors to R3
> `(conversation_id, user_id, last_read_seq)` tuple + #263 §2.2 Tester gates + Q-T1 human-only lock.
> **This is the frontend's expected DTO/API shape.** Dev builds backend to match; Dev2 then re-runs
> against the real endpoint (run-real-usage). Changing a field here = cross-pair break, re-flag.

The frontend consumes three things: **(1)** per-conversation unread/mention **counts** (sidebar
badges), **(2)** a **read-position advance** call (mark-as-read), **(3)** **thread follow/unfollow**
state + toggle. All **human-only** (Q-T1: agents are directed-wake, never accumulate unread/badges).

## 1. Message gains `seq` (monotonic per conversation)

Read-position is by `last_read_seq`, so the message DTO must expose the per-conversation sequence:

```ts
export interface Message {
  // ...existing (id, conversation_id, sender_identity_id, content_kind, content,
  //    direction, posted_at, context_refs?, attachments?)...
  seq: number;   // NEW — monotonic per conversation, gap-free or strictly increasing;
                 // the unit last_read_seq compares against. Reuses the existing backend
                 // message seq (P1 task "复用 trunk: message seq").
}
```

`GET /conversations/:id/messages` returns messages ordered by `seq` ascending (unchanged ordering;
just exposes the field).

## 2. Conversation DTO gains unread/mention/followed (per current user)

Badge source = the existing `GET /conversations` list (sidebar `useConversations({kind})`) and
`GET /conversations/:id`. Add three fields, **computed for the requesting human user**:

```ts
export interface Conversation {
  // ...existing (id, kind, name, status, participants?, peer_*, owner_ref?, ...)...
  unread_count: number;   // NEW — messages with seq > my last_read_seq; 0 when caught up.
  mention_count: number;  // NEW — subset of unread that @-mention me (directed); 0 when none.
  followed: boolean;      // NEW — am I following this conversation/thread (see §4).
}
```

- **human-only (Q-T1):** counts are computed for human identities only. An agent session MUST get
  `unread_count: 0 / mention_count: 0` (agents are woken by mention → execute, never "browse
  unread"). Backend keys the tuple on human user ids; no unread row is created for agents.
- `mention_count ≤ unread_count` always.
- For `kind: 'channel' | 'dm'` and the work-item thread conversations (`owner_ref` set), all carry
  these fields uniformly (surface-agnostic — sidebar #258 Issues/Tasks nav + Channels/DMs all badge
  off the same fields).

## 3. Read-position advance (mark-as-read)

```
POST /api/conversations/:id/read?org_slug=<slug>
body: { "last_read_seq": <number> }     # the seq of the latest message the user has seen
→ 200 { "unread_count": 0, "mention_count": 0 }   (the recomputed counts after advance)
```

Semantics (R3 + #263 §2.2):
- **monotonic / never regresses:** stored `last_read_seq = max(stored, body.last_read_seq)`. A lower
  value is a no-op (idempotent), not a regression. (Layer-2 never-drift analog: read-position only
  advances.)
- **idempotent:** repeating the same `last_read_seq` is a no-op, returns the same counts.
- frontend calls this when the user views/scrolls a conversation to bottom (debounced); the sidebar
  badge then clears via query invalidation.

## 4. Thread follow / unfollow

```
POST   /api/conversations/:id/follow?org_slug=<slug>     → 200 { "followed": true }
DELETE /api/conversations/:id/follow?org_slug=<slug>     → 200 { "followed": false }
```

- state is **per-(human user, conversation/thread)**; reflected in `Conversation.followed` (§2).
- **auto-follow** on participate (send a message) or being @-mentioned (backend sets `followed=true`).
- unfollow → stops delivery/counting for that thread (mirrors the CLI `slock thread unfollow`).
- threads are conversations with `parent_conversation_id` set; follow applies at that conversation id.

## 5. Frontend render (informational — for Dev's context, not a backend obligation)

- Sidebar nav item (Channels / DMs / #258 Issues / Tasks): `mention_count` → red badge with the
  **precise number** (not a dot); `unread_count>0` (no mention) → bold + neutral dot. **Not
  color-only** (number + bold + icon — a11y, §A red-line).
- A conversation's own header may show a follow toggle (visualizes `followed`).
- The frontend will mock these fields in tests until the real endpoint lands, then re-run real
  (run-real-usage) — mock matches THIS contract exactly.
