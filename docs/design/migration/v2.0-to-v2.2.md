# v2.0 → v2.2 Upgrade Guide

> Target audience: operators running a v2.0 (or v2.1) agent-center who
> are upgrading to v2.2. v2.2 introduces the transport layer that v2.0
> GA shipped without — CLI + worker daemon now talk to the server via
> the admin unix socket instead of touching sqlite directly. For a
> fresh install, follow [v2.2-mac-single-host](../deployment/v2.2-mac-single-host.md)
> instead.

> v2.2 is a **minor version** with one breaking config default change
> (web console now on by default). Read [§ 2 Behavior changes](#-2-behavior-changes)
> before upgrading any non-toy deployment.

---

## § 0. Why upgrade

- **Actual worker daemon process** — `cmd/worker-daemon` binary
  ships, fixing the v2.0 GA "worker process 装配从未交付" defect.
  Real dispatch → real subprocess → real task completion.
- **CLI / worker now go through AppService transport** — conventions
  § 0.4 ("AppService is the only entry to domain state") finally
  enforced. v2.0 GA had CLI directly opening the same sqlite file as
  the server; v2.2 routes through admin unix socket.
- **`make smoke` deploy-binary smoke gate** — every release now ships
  with the full server + worker + agent pipeline verified by a real
  Playwright e2e against the deployed binaries (no mocks).
- **Process gates** — mock-as-default lint, doc-impl drift check,
  arch test enforcing CLI handler whitelist (per conventions § 0.4
  Enforce mechanisms).

If you're happy with v2.0/v2.1 not actually being able to dispatch
tasks to a worker, this upgrade is optional. Otherwise it's
recommended.

## § 1. Pre-flight checklist

```bash
# 1. Stop v2.0/v2.1 server
sudo systemctl stop agent-center

# 2. Backup sqlite (MANDATORY)
sudo cp /var/lib/agent-center/agent-center.db \
        /var/lib/agent-center/agent-center.db.pre-v2.2

# 3. Backup config (v2.2 introduces no breaking config-schema changes,
#    only a default change — see § 2.1)
sudo tar czf /root/agent-center-v2.0-config.tar.gz /etc/agent-center/

# 4. Verify admin_socket_path is set in your config; if not, add it.
#    v2.2's CLI + worker daemon require it.
grep -A 4 '^server:' /etc/agent-center/config.yaml | grep admin_socket_path \
  || echo 'MISSING — add `server.admin_socket_path: "/run/agent-center/admin.sock"`'
```

## § 2. Behavior changes

### § 2.1 Web Console default flipped to ON (BREAKING DEFAULT)

**v2.0/v2.1**: Web Console was opt-in. A config that omitted
`web_console.enabled` did NOT start the Web Console HTTP server.

**v2.2**: Web Console is enabled by default. A config that omits
`web_console.enabled` starts the SPA at `127.0.0.1:7100`. The
loopback bind constraint (ADR-0037) still applies — no remote
exposure unless you SSH-tunnel.

**Migration**:
- If you WANT Web Console: do nothing. v2.2 will start it
  automatically on `127.0.0.1:7100`.
- If you DON'T want Web Console (truly headless server): add the
  explicit opt-out:
  ```yaml
  web_console:
    enabled: false
  ```
- If you want a different port: keep your existing
  `web_console.listen_addr` setting; v2.2 only changes the boolean
  default, not the addr default.

**Reason**: Per @oopslink (2026-05-25, during v2.2 release-gate
manual verify): "默认 config 不开 web console 这个不对 默认打开". The
primary v2 user entry is the SPA, so default-off was the wrong
shape.

### § 2.2 Worker daemon is a separate process

**v2.0/v2.1**: `agent-center worker run` was a placeholder ("Worker
daemon (TaskRuntime) is implemented in Phase 2; this mode is
reserved."). No real worker process existed; dispatched tasks sat
forever in `submitted` state.

**v2.2**: New binary `agent-center-worker-daemon` (built by
`make build`). It connects to the server via the admin unix socket,
enrolls itself, polls the dispatch + kill queues, spawns the agent
CLI subprocess, and reports back through admin endpoints.

**Migration**:
- Build the new binary: `make build` (or download from release
  artifacts).
- Add a systemd unit / launchd plist / supervisor entry that starts
  it after the server is up.
- See [v2.2-mac-single-host § 4](../deployment/v2.2-mac-single-host.md#-4-start-the-three-processes)
  for the typical command shape.

### § 2.3 CLI no longer touches sqlite directly

**v2.0/v2.1**: Every CLI subcommand opened the same sqlite file the
server was using. Worked because of SQLite WAL multi-process access,
but violated DDD (`conventions § 0.4`) — invariants enforced by the
server's Application Services were bypassed.

**v2.2**: All 36 CLI subcommands talk to the server via admin
endpoint (HTTP/JSON over unix socket). The only exceptions —
hardcoded in arch test — are:
- `agent-center migrate up` / `migrate v1-to-v2` (schema migration
  tool, must touch DB directly to apply DDL)
- `agent-center server` (the server itself — it OWNS the DB)

**Migration impact**: usually invisible to operators — CLI behavior
preserved. Only visible difference: a CLI command run while the
server is down now returns a friendly error ("server not running at
<socket>; start it with: agent-center server") instead of opening a
stale DB handle.

### § 2.4 Real SupervisorSpawner

**v2.0/v2.1**: `app.SupervisorSpawner = nil`. Any supervisor
invocation (cron, manual retrigger, etc.) silently no-op'd.

**v2.2**: Server boot wires `scheduler.NewSpawner(...)` with
`ExecProcessRunner`. Supervisor subprocess invocations actually
fork+exec.

**Migration**: no action required. If you had no supervisor flows
firing in v2.0/v2.1 (because they were no-ops), you'll suddenly see
them in v2.2. Validate your supervisor config (cron schedule, scope
filters) before letting it run hot.

### § 2.5 dispatch.NoopSender / kill.NoopKillSender removed from production wiring

**v2.0/v2.1**: `app.go` wired `dispatch.NoopSender{}` (envelopes →
/dev/null) and `kill.NoopKillSender{}` (kill requests → /dev/null).

**v2.2**: Replaced with `dispatchq.DispatchSender` / `KillSender`
backed by an in-process queue that the worker daemon drains.

**Migration**: no action required. Visible effect: a v2.0 install
that *appeared* to dispatch tasks (events emitted, state advanced
to `submitted`) now actually delivers envelopes to the worker.

## § 3. Schema migrations

**No new schema migrations in v2.2.** The `migrate v1-to-v2` tool
still applies the v1→v2 schema if you upgrade from a v1 install.
v2.0 → v2.2 is purely process / transport / wiring.

Run after install for safety:
```bash
agent-center migrate up --config=/etc/agent-center/config.yaml
```

(should report `already at v2; no action` if v2.0+).

## § 4. Install v2.2 + smoke

```bash
# 1. Install the v2.2 binaries (pre-built or from source)
make build
sudo install -m 0755 ./bin/agent-center               /usr/local/bin/
sudo install -m 0755 ./bin/agent-center-worker-daemon /usr/local/bin/
sudo install -m 0755 ./bin/fakeagent                  /usr/local/bin/  # for smoke + LLM-free tests

# 2. Replace systemd unit (v2.2 adds the worker-daemon target +
#    starts admin socket on server bring-up). See contrib/.

# 3. Start
sudo systemctl start agent-center
sudo systemctl start agent-center-worker-daemon

# 4. Verify
sudo -u agent-center make smoke
# expect: smoke pass: NN seconds
```

For a quick manual smoke without the smoke gate:
- `curl --unix-socket /run/agent-center/admin.sock http://localhost/admin/health`
  → `{"ok":true,"transport":"unix","endpoint":"admin"}`
- `curl http://127.0.0.1:7100/api/health` → `{"status":"ok",...}`
- `agent-center worker list` → enrolled worker appears
- `agent-center ps` → fleet snapshot via admin endpoint

## § 5. Rollback

Same shape as v1→v2 rollback. v2.0 binary can read the v2.2 DB (no
schema changes), so revert is straightforward:

```bash
sudo systemctl stop agent-center agent-center-worker-daemon
sudo cp /var/lib/agent-center/agent-center.db.pre-v2.2 \
        /var/lib/agent-center/agent-center.db
sudo install -m 0755 /path/to/agent-center-v2.0-<sha> /usr/local/bin/agent-center
sudo systemctl start agent-center
# worker-daemon won't run on v2.0 binary; that's expected (v2.0 had no daemon).
```

## § 6. FAQ

**Q. I see Web Console listening on 127.0.0.1:7100 after upgrade and
I didn't want that.**
A. § 2.1 default flip. Add `web_console: {enabled: false}` to your
config + restart.

**Q. CLI returns "server not running at /run/agent-center/admin.sock".**
A. v2.2 CLI requires the admin endpoint. Start the server first; check
`server.admin_socket_path` is set in config.

**Q. `agent-center worker run` says it's a placeholder.**
A. That command stayed as a stub pointer to the new
`agent-center-worker-daemon` binary. Use the binary directly.

**Q. Old v2.0 supervisor cron job is suddenly emitting events.**
A. § 2.4 — supervisor was a nil no-op in v2.0/v2.1, now real. Verify
your cron schedule is what you intended before letting it run hot.
