# v2.4 — First-Mile Deployment Guide

> Audience: anyone bringing up an agent-center server + one or more
> worker daemons for the first time. Pairs with
> [v2.3 multi-host guide](v2.3-multi-host.md), which is now mostly a
> reference for the cross-host TCP+TLS internals — operationally,
> prefer this guide.
>
> Background: v2.4-D (first-mile deployment) lands the `install`
> subcommand, the atomic-symlink upgrade with auto-rollback, and the
> Web Console **Add Worker** Modal that mints a one-time enroll token
> and shows a copyable install command. See
> [v2.4-deployment-first-mile.md](../plans/v2.4-deployment-first-mile.md)
> for the product context.

## § 1. Prereqs

You have an `agent-center-v2.4.0-<os>-<arch>/` directory extracted
locally. The tarball layout:

```
agent-center-v2.4.0-darwin-arm64/
├── bin/
│   ├── agent-center
│   └── agent-center-worker-daemon
├── assets/
│   └── default-config.yaml
├── LICENSE
├── README.md
└── install            -> bin/agent-center        (symlink)
```

`./install center` and `./install worker` are the only two commands
you need to know.

Mac is fully verified for v2.4. Linux installs the systemd unit and
should work, but Mac was the acceptance target — Linux validation is
deferred to v2.5.

## § 2. Journey A — install the center

`cd` into the extracted tarball directory, then:

```bash
$ ./install center
agent-center install center:
  prefix:        ~/.agent-center
  user-mode:     true
  web port:      7100
  state:         fresh
✓ AgentCenter v2.4.1 installed
✓ launchd service loaded (auto-start enabled)
✓ Database initialized at ~/.agent-center/var/agent-center.db
✓ Bootstrap admin token saved to ~/.agent-center/var/bootstrap_token
✓ Web Console: http://127.0.0.1:7100

Open the URL above to continue setup.
```

End-to-end: ≤ 30 seconds on a warm machine.

### 2.1 Flags

| Flag | Default | Meaning |
|---|---|---|
| `--prefix=<path>` | OS-dependent (see below) | Install prefix |
| `--user-mode` | `true` on Mac, `false` on Linux | User-mode install (no sudo, no systemd unit at /etc) |
| `--port=<n>` | `7100` | Web Console listen port (loopback only) |
| `--tcp-listen=<host:port>` | empty | Open admin TCP listener for cross-host workers |
| `--dry-run` | `false` | Print planned actions without mutating state |

Defaults for `--prefix` (v2.4.1 unified to `~/.agent-center/` for user-mode):
- **Mac, user mode**: `~/.agent-center`
- **Linux, user mode**: `~/.agent-center`
- **Linux, system mode** (`--user-mode=false`): `/opt/agent-center`

### 2.2 Single-host vs multi-host

If you only need workers on this same Mac, the default (`--tcp-listen=""`)
is enough — workers will use the local unix socket.

For workers on **other machines**, pass `--tcp-listen=0.0.0.0:7300`
so the admin endpoint accepts TCP+TLS. The Web Console Modal will
include the right `--bootstrap=tcp://...` URL in the install command
it generates.

## § 3. Journey B — add a worker

### 3.1 v2.5 model: add ≠ install

v2.5 splits the two halves of "bring a new worker online" into
distinct steps:

- **Adding** a worker is a *logical* action — clicking + Add Worker
  in the Web Console creates the Worker record immediately
  (status=offline) and mints a 30-minute install token bound to that
  worker_id. The Modal closes right away; the new row appears in the
  Fleet table.
- **Installing** the worker is a *physical* action — on the worker
  machine, the operator runs `./install worker --token=...`. The
  daemon enrolls against the existing record, which flips its status
  to online.

The two steps don't have to happen back-to-back. You can mint twenty
workers in one Fleet view, then walk over to each machine and run
the install command at your leisure. The Fleet shows you what's
still offline.

### 3.2 From the Web Console

1. Open `http://127.0.0.1:7100/fleet`.
2. Click **+ Add Worker** (top-right).
3. Modal opens with a **Worker name** input. Type a friendly name
   (e.g. `test-1`, `tenant-foo`) — this is the label you'll see in
   the Fleet table. Use a unique name per worker on the same machine
   so you can tell them apart. You can rename later from the Fleet
   row.
4. Click **Add worker**. The server creates the Worker record
   (status=offline) + mints an enroll token bound to its
   server-generated immutable id (`worker-<8-hex>`). The Modal
   closes; the new row appears in Fleet (status=offline, no install
   command yet — see step 5).
5. In the Fleet row, click **Show install command** to display the
   copyable command:

   ```bash
   $ ./install worker \
       --bootstrap=tcp://host:7300 \
       --server-fingerprint=sha256:... \
       --worker-id=worker-f3c04c1d \
       --worker-name='test-1' \
       --token=acat_xxxxxxxxx
   ```

   The plaintext bearer is encrypted at rest with the server's
   `master_key` and can be re-shown until the token is consumed
   (daemon enrolls) or its 30-minute TTL elapses.

6. On the worker machine, `cd` into that machine's extracted
   `agent-center-v2.5.0-<os>-<arch>/` directory and paste the
   command.
7. The Fleet row flips to **online** and briefly highlights green;
   a bottom-right toast appears too.

The **worker id is immutable** (server-generated at mint time). The
**worker name** is editable — see § 3.5.

### 3.2.1 If the install command expired or was lost

Two recovery affordances on each offline Fleet row:

- **Show install command** — if the token is still alive (within
  the 30-min TTL and not yet consumed), the original command is
  re-displayed verbatim. If the token has expired or was burned,
  the Modal offers a one-click **Re-mint install command**.
- **Re-mint install command** — revokes any active install token
  for this worker_id and issues a fresh one. The Worker record
  stays put (same id). Refused with a 409 if the worker is already
  online — in that case the operator should **Remove** the record
  and add a new one if they really need to reinstall.

### 3.2.2 Remove worker

Every Fleet row has a **Remove** button (danger styled). Confirm
the dialog and the server will revoke all admin tokens bound to
that worker (long-term + any active install token) and drop the
Worker record. The row disappears from Fleet via SSE without an
explicit page refresh. If the daemon was still running, its next
admin call will 401 and the operator should kill the process on
the worker machine.

### 3.3 From the CLI directly

If you'd rather not click around the Web Console, mint a token from
the host that's running the center:

```bash
$ agent-center admintoken mint-enroll --ttl=30m
{
  "id": "...",
  "plaintext": "acat_xxxxxxxxx",
  "expires_at": "2026-05-26T13:00:00Z"
}
```

Then run `./install worker --bootstrap=tcp://<fp>@host:7300 --token=...`
on the worker box.

### 3.4 Worker `install worker` flags

| Flag | Required | Meaning |
|---|---|---|
| `--bootstrap=<url>` | yes | Admin endpoint to dial (`tcp://<fp>@host:7300` or `unix:/path`) |
| `--server-fingerprint=<sha256>` | yes for tcp:// | SSH-style cert fingerprint; minted in same Modal response |
| `--token=<token>` | yes | One-time enroll token (from the Modal or `mint-enroll`) |
| `--worker-id=<id>` | yes (Modal pre-fills) | Server-generated `worker-<8hex>`; immutable; comes from mint response |
| `--worker-name=<name>` | no | Display name set in the Modal; daemon forwards to enroll; renameable later via WC |
| `--capabilities=<csv>` | no | e.g. `claudecode,fakeagent` (auto-detected if omitted) |
| `--prefix=<path>` | no | Defaults to `<install-default>/worker-<worker-id>/`, isolating multi-worker installs on the same machine |
| `--user-mode` | no | Same default as `install center` |
| `--dry-run` | no | Print planned actions without mutating state |

### 3.5 Multi-worker on the same machine

Each `install worker` registers a launchd service labelled
`com.agent-center.worker.<worker-id>` and lays down state under a
prefix that includes the id, so several workers can coexist on one
host without colliding:

```
~/.agent-center/                         # center install
├── current → versions/v2.4.1/           # center binary symlink
├── versions/v2.4.1/{bin/, ...}
├── etc/config.yaml
├── var/{agent-center.db, admin.sock, bootstrap_token, admin-tls.{crt,key,fingerprint}}
├── logs/{stdout.log, stderr.log}        # center logs (no longer in /tmp)
└── workers/                             # per-worker subtree (v2.4.1)
    ├── worker-f3c04c1d/                 # worker named "test-1"
    │   ├── current → versions/v2.4.1/
    │   ├── var/{worker.db, worker-token}
    │   └── logs/{stdout.log, stderr.log}
    └── worker-de6b92eb/                 # worker named "test-bob"
        ├── current → versions/v2.4.1/
        ├── var/{worker.db, worker-token}
        └── logs/{stdout.log, stderr.log}
```

The Fleet table shows each worker's friendly name (large) with its
immutable id (mono, small) underneath. Click the name in a Fleet row
to rename it inline — the id never changes.

Use multi-worker for: load testing, multi-tenant simulation, or
exercising agent isolation. Each worker keeps its own long-term
admin token; revoke + re-enroll one worker without affecting the
others.

## § 4. Journey C — upgrade in place

Same command, new tarball.

1. Extract `agent-center-v2.4.1-<os>-<arch>/` over a different path.
2. `cd` into it, then:

   ```bash
   $ ./install center
   agent-center install center:
     state:         upgrade (current=v2.4.0, this=v2.4.1)
   ✓ migrated DB to schema version 30
   ✓ swapped current → v2.4.1
   ✓ launchd service restarted
   ✓ health probe passed
   ```

The upgrade is **atomic**: the new version is laid down at
`<prefix>/versions/v2.4.1/`, the schema migration runs, and then
`<prefix>/current` is flipped via a POSIX `rename(2)` to the new
versioned directory. The service is restarted; the installer probes
the health endpoint, and if the probe fails it **automatically rolls
back** the symlink to the previous version and restarts.

Worker upgrades work the same way (`./install worker --bootstrap=... --token=...`
against the same existing worker — re-uses the same prefix, same
worker-id; the token can be a fresh enroll-token from a new Modal
session or an existing long-term token).

### 4.1 Rollback behavior

| Step that fails | Auto-rollback? | Operator does |
|---|---|---|
| Copy binaries into `<prefix>/versions/<new>/bin/` | n/a — symlink not flipped yet | Re-run `./install center` after fixing disk/perm |
| DB migration | n/a — symlink not flipped yet | Re-run; migration is idempotent. If it's stuck on a specific migration step, see § 6 troubleshooting |
| Atomic symlink swap | n/a — POSIX `rename(2)` is atomic | n/a |
| Service restart | yes — symlink flipped back to old version, service restarted | Check logs at `<prefix>/data/server.log` and the underlying error printed by the installer |
| Health probe (5s timeout) | yes — same as above | Same |

### 4.2 Versions kept on disk

Old versions are retained at `<prefix>/versions/<old>/` so the
auto-rollback has something to point at. The installer doesn't yet
garbage-collect; cleanup is a manual `rm -rf <prefix>/versions/<old>/`
when you're confident you don't need to roll back further.

## § 4.5 Journey D — uninstall

### 4.5.1 Center

```bash
$ agent-center uninstall center [--prefix=...] [--purge] [--yes] [--dry-run]
```

Default behaviour (no `--purge`):

- Stops + unloads the launchd / systemd service.
- Removes the service unit file (LaunchAgents plist or systemd unit).
- Removes the install artefacts: `<prefix>/versions/`,
  `<prefix>/current`, `<prefix>/etc/`.
- **Preserves** `<prefix>/var/` (sqlite database, master_key,
  bootstrap_token, etc.) and `<prefix>/logs/`.

Output names exactly what survived + how to come back:

```
✓ AgentCenter center uninstalled
  Data preserved at:
    ~/.agent-center/var/   (sqlite + master.key + tokens)
    ~/.agent-center/logs/  (service stdout/stderr)
  Reinstall with `./install center` at the same prefix will reuse them.
  To remove all data: rerun with --purge
```

A subsequent `./install center --prefix=<same path>` walks the
fresh-install path but reuses the existing `master.key`,
`agent-center.db`, etc. — no data loss, no token rotation.

### 4.5.2 Worker

```bash
$ agent-center uninstall worker --worker-id=<id> [--prefix=...] [--purge] [--yes]
```

`--worker-id` is required. Scoped to the matching worker subtree
under `<prefix>/workers/<id>/` (or whatever `--prefix` overrides);
sibling workers + the center install are untouched.

### 4.5.3 `--purge` (destructive)

`--purge` extends the cleanup to wipe `var/`, `logs/`, and the
prefix itself. Without `--yes` the command prompts:

```
  --purge will DELETE ~/.agent-center and everything under it.
  This includes the sqlite database, master.key, and worker tokens.
  This is NOT REVERSIBLE.

  Type "yes" to continue, anything else to abort:
```

`--yes` skips the prompt (for scripted teardown).

### 4.5.4 `--dry-run`

Prints the full plan (every shell command + every `rm -rf` target)
without mutating anything. Use this when you're unsure what
uninstall will touch.

## § 5. Idempotency & dry-run

- Running `./install center` twice with the **same version** is a
  no-op: `state: same_version` → `✓ AgentCenter v2.4.0 already
  installed, no changes`.
- `--dry-run` prints the detected state + planned actions without
  changing anything. Use it to preview a tarball before committing.

## § 6. Failure modes

The installer maps common syscall errors to friendly messages with
a hint and the underlying error. The pattern:

```
Disk full: cannot write to /var/lib/agent-center

What to try:
  Free up space on the target filesystem, or pass --prefix=<path>
  to install somewhere with more room.

Underlying: no space left on device
```

Coverage:

| Symptom | Friendly message | Hint |
|---|---|---|
| Port 7100 already in use | `Port 127.0.0.1:7100 is already in use` | `Use --port=<n> to pick a different port` |
| Disk full | `Disk full: cannot write to <prefix>` | `Free up space on the target filesystem, or pass --prefix=<path>` |
| Permission denied writing systemd unit | `Permission denied writing /etc/systemd/system/agent-center.service` | `Run with sudo, or use --user-mode for a per-user install` |
| Permission denied writing binary | `Permission denied writing <prefix>/versions/.../bin/` | `Use --prefix=<path-you-own>` |
| Enroll token already used | (Modal: **Token used**) | Mint a new token from the Modal — the old one only works once |
| Enroll token expired (30 min) | (Modal: **Token expired**) | Re-mint; the worker side will report `401 auth_expired` if you race |
| Worker can't reach center | `connection refused` / `connection timed out` | Check firewall; verify `--tcp-listen` is set on the center; verify `--bootstrap` matches |
| Worker fingerprint mismatch | `fingerprint mismatch dialing <addr>` | Cert rotated, or MITM. Re-fetch fingerprint from the center via a trusted channel; if it really changed, paste the new value into the Modal-generated command |
| Upgrade health probe failed | `health probe failed: <reason>` | Installer auto-rolled back. Inspect `<prefix>/data/server.log` for the new-version crash |
| DB migration failed | `migration 0029 failed: <reason>` | Old binary is still on disk; investigate, re-run when fixed. Migrations are idempotent |
| Both `--user-mode` and `--prefix=/opt/...` | "permission denied" downstream | Pick one mode; system-mode prefixes need `sudo` |
| Worker stuck in `401 auth_revoked` restart loop (operator revoked the worker via WC but daemon keeps reusing its persisted long-term token) | `[worker] heartbeat: ... 401 auth_revoked` repeatedly in the worker log | See § 6.1 below |

### 6.1 Decommissioning a worker (revoke recovery)

If you revoke a worker from the Web Console but the worker daemon is
still running on its host, the daemon will keep retrying with its
persisted long-term token and hit `401 auth_revoked` every cycle.
That's by design (v2.4 does not auto-clean a revoked worker
client-side), but it means you need to clean up the worker host:

```bash
# 1. Stop the launchd service AND deregister it from macOS Background
#    Items (Service Management). v2.5.17 (#72): `launchctl unload`
#    alone leaves a stale ON toggle in System Settings → Login Items
#    & Extensions → Allow in Background on macOS Ventura+. Use the
#    modern bootout instead — kills the daemon AND clears the
#    SMAppService registration.
launchctl bootout gui/$(id -u) ~/Library/LaunchAgents/com.agent-center.worker.<worker-id>.plist

# 2. Remove the persisted long-term token so a future re-enroll starts fresh
rm -f ~/.agent-center/workers/<worker-id>/var/worker-token

# 3. If you want the worker gone entirely, also remove the plist + state dir
rm -f ~/Library/LaunchAgents/com.agent-center.worker.<worker-id>.plist
rm -rf ~/.agent-center/workers/<worker-id>

# 4. To re-add: mint a fresh token in the Web Console (it will allocate
#    a new worker-id) and run ./install worker as usual.
```

A polished `agent-center worker remove <id>` CLI that does steps 1–3
in one shot is on the v3 deployment-as-product list.
