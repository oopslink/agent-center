# v2.3 — Multi-host deployment guide

> Audience: anyone running an agent-center server on one machine and
> one or more worker daemons on different machines (cloud VM, second
> Mac, Linux box). Pairs with [v2.2 single-host guide](v2.2-mac-single-host.md);
> v2.2 still applies if you only need the local case.
>
> Background: v2.3-7 (task #27) added TCP+TLS to the admin endpoint
> with SSH-style fingerprint pinning + per-token rate limiting. This
> guide is the end-to-end operator walkthrough.

## § 1. Architecture

```
+------------------+              +-----------------------+
| host A: SERVER   |              | host B: WORKER DAEMON |
|                  |              |                       |
| agent-center     |   TCP + TLS  | agent-center-worker-  |
|   server         |<------+      |   daemon              |
|                  |       |      |                       |
| admin endpoint:  |       |      | dials                 |
|   unix sock      |       +------| tcp://A:7300          |
|   tcp :7300 TLS  |              | + server_fingerprint  |
|                  |              | + bearer token        |
| Web Console:     |              |                       |
|   loopback :7100 |              | local worker binaries:|
|                  |              |   ./agent-center      |
+------------------+              +-----------------------+
       |                                    |
       | SSH tunnel for Web Console         | local CLI uses
       v                                    | unix socket OR
   browser                                  | same TCP target
```

Key invariants:
- Server's **admin endpoint** is reachable both on unix socket (local
  tools) AND TCP+TLS (cross-host worker / CLI). Both listeners share
  the same auth + rate limit middleware.
- Trust model: client (worker / CLI) pins the server's self-signed
  cert SHA256 fingerprint. No CA, no mTLS — operator manages
  fingerprint just like SSH host keys.
- Web Console stays loopback-only; SSH-tunnel for remote browser
  access (unchanged from v2.2).

## § 2. Server side (host A)

### 2.1 Config

`/etc/agent-center/config.yaml`:

```yaml
server:
  listen_addr: ":7000"
  sqlite_path: "/var/lib/agent-center/agent-center.db"

  # v2.2 unix socket — for local CLI on host A
  admin_socket_path: "/run/agent-center/admin.sock"

  # v2.3-7a: TCP+TLS listener — for cross-host worker / CLI
  admin_tcp_listen: "0.0.0.0:7300"

  # Optional: explicit TLS cert/key paths. Defaults under
  # filepath.Dir(sqlite_path), i.e. /var/lib/agent-center/
  # admin-tls.{crt,key,fingerprint}. Auto-generated on first boot.
  # admin_tls_cert_path: /etc/agent-center/admin-tls.crt
  # admin_tls_key_path:  /etc/agent-center/admin-tls.key

identity:
  default_user: hayang

web_console:
  enabled: true
  listen_addr: "127.0.0.1:7100"
```

`tcp_listen: ""` (default) = unix socket only (v2.2 behavior).
`socket_path: ""` + `tcp_listen` set = TCP-only mode (no local CLI
on host A).
Both empty = boot fails with error.

### 2.2 Start + read the fingerprint

```bash
$ ./bin/agent-center server --config=/etc/agent-center/config.yaml
agent-center server: db=/var/lib/agent-center/agent-center.db listen=:7000 web=127.0.0.1:7100 admin=/run/agent-center/admin.sock (escalator running)
  admin tcp:  0.0.0.0:7300 (TLS, auto-generated)
              cert valid until 2027-05-26 (364 days)
              fingerprint: sha256:7F:4A:8C:9B:...:6F:2C
```

The fingerprint also lands at `/var/lib/agent-center/admin-tls.fingerprint`
(operator-readable). Copy this string — you'll paste it into the
worker config.

The first boot also writes a **bootstrap admin token** at
`/var/lib/agent-center/bootstrap_token` (operator-only mode 0600).
This is the bearer worker daemons will use. For long-term operation,
mint per-worker tokens via `agent-center admintoken create` and put
the bootstrap token away.

### 2.3 Firewall

Open TCP **7300** (the `admin_tcp_listen` port) from worker host(s)
to host A. Other ports unchanged:
- `7000` is the legacy worker enroll port (kept for backward compat);
  v2.3 worker daemons go through `7300` admin endpoint
- `7100` web console **must stay closed** to public — loopback only

Recommend explicit allowlist of worker host IPs at the firewall
level, on top of bearer token + rate limit. v2.3 ships no IP CIDR
gate inside agent-center (deferred to v3 deployment-as-product).

## § 3. Worker side (host B)

### 3.1 Get the worker binary

Today: copy `./bin/agent-center-worker-daemon` from host A's build
to host B. (Future: download tarball + extract per v2.4-deployment-
first-mile plan.) Place at e.g. `~/.local/bin/agent-center-worker-
daemon`.

### 3.2 Hand off fingerprint + token

The operator copies two strings from host A to host B (manual; via
SSH / paste / password manager — fingerprint is public info, token
is bearer secret so handle accordingly):

```bash
# On host A:
FP=$(cat /var/lib/agent-center/admin-tls.fingerprint)
TOKEN=$(sudo cat /var/lib/agent-center/bootstrap_token)
echo "FP=$FP"
echo "TOKEN=$TOKEN"
```

### 3.3 Run the worker

```bash
$ ./bin/agent-center-worker-daemon \
    --worker-id=W-vps-1 \
    --admin-target=tcp://host-a.example.com:7300 \
    --server-fingerprint="sha256:7F:4A:8C:9B:...:6F:2C" \
    --admin-token="acat_aRrr_..." \
    --capabilities=claudecode,fakeagent \
    --config=/path/to/local/config.yaml
[worker] starting: worker_id=W-vps-1 target=tcp://host-a.example.com:7300 poll=1s overrides=0
[worker] enrolled as worker_id=W-vps-1
```

Quiet main loop after enroll. Activity shows up on dispatch /
execution events.

### 3.4 Verify from host A

```bash
# On host A:
$ agent-center worker list
WORKER_ID     STATUS      CAPABILITIES
W-vps-1       online      claudecode, fakeagent
```

Or in Web Console at http://127.0.0.1:7100 → `/fleet`, the worker
shows up.

## § 4. Token + fingerprint hygiene

### 4.1 Mint per-worker tokens

The bootstrap token has `*` scope (admin all-everything). For prod
deployments mint a scoped token per worker:

```bash
$ agent-center admintoken create --owner=worker:W-vps-1 --scopes='task:*,secret:resolve'
{"id":"...","plaintext":"acat_...","owner":"worker:W-vps-1","scopes":["task:*","secret:resolve"]}
```

Show the plaintext to the operator ONCE (it's never persisted),
then deploy the plaintext to host B (encrypted channel / wrapper
script env-var).

### 4.2 Rotate token

```bash
# Mint a new token first
$ agent-center admintoken create --owner=worker:W-vps-1 --scopes='task:*,secret:resolve'

# Deploy new plaintext to host B; restart worker

# Verify worker reconnected with new token by re-listing
$ agent-center worker list

# Revoke old token
$ agent-center admintoken revoke --id=<old-token-id> --reason="rotation"
```

v2.3 doesn't support hot rotation (worker daemon won't pick up a new
token without restart). v3 deployment-as-product theme will add
graceful renew.

### 4.3 Rotate cert / fingerprint (destructive)

Cert rotation in v2.3 is **destructive**:
1. Stop server
2. Delete `/var/lib/agent-center/admin-tls.{crt,key,fingerprint}`
3. Restart server — new cert auto-generated; new fingerprint
4. **All workers immediately break with "fingerprint mismatch"**
5. Update every worker's `--server-fingerprint` flag to the new
   value, restart each worker

The 30-day cert expiry warning fires `admin.tcp_cert_expiring`
observability event well in advance; query with `agent-center
events query --type=admin.tcp_cert_expiring`.

## § 5. Rate limit

Per-token rate limit defaults: burst=200, refill=10/sec sustained.
A normal worker daemon (1 poll/sec + ~3 calls per task progression)
sits well below this; a stolen-token attacker hits the ceiling fast
and the operator sees `admin.rate_limit_hit` events.

```bash
# Operator dashboard query:
$ agent-center query events --type=admin.rate_limit_hit --limit=20
```

Each event payload contains `token_id`, `client_ip`, `method`, `path` —
operator can `agent-center admintoken revoke --id=<token-id>` on
suspicious patterns.

## § 6. Troubleshooting

| Symptom | Likely cause | Fix |
|---|---|---|
| Worker `fatal: clienttransport: fingerprint mismatch dialing ...` | Cert rotated, or **someone is MITM'ing**. Don't proceed until verified. | Re-fetch fingerprint from host A via secure channel; compare; if it really changed (rotation), update worker. If you can't explain change, treat as compromise. |
| Worker `connect: connection refused` | TCP listener not running OR firewall blocking | Check `admin tcp:` line in server boot log; `curl -k https://host-a:7300/admin/health` from host B |
| Worker `401 unauthorized` | Bearer token wrong / revoked | Re-mint via `agent-center admintoken create`; redeploy |
| Worker `429 rate_limited` | Rate limit exceeded (suspicious or test-burst) | Check `Retry-After` header; if legit, slow caller; if not, revoke token |
| Server boot fails `admin api: at least one of socket_path or tcp_listen required` | Both `admin_socket_path` and `admin_tcp_listen` empty in config | Set at least one |
| Server boot fails `cert exists at X but key missing at Y` | Half of cert/key files missing | Delete both `admin-tls.crt` and `admin-tls.key`; restart for auto-regenerate |

## § 7. What's NOT in v2.3 (filed for v3 / v2.4)

- **One-command install + WC-shown enroll command** — v2.4 deployment
  first-mile theme; lets operator paste a single CLI line on the
  worker box instead of constructing the flag set manually
- **CA-signed cert / Let's Encrypt** — v3 deployment-as-product
- **mTLS** — v3
- **Per-IP allowlist (CIDR)** — v3
- **Hot token rotation (no restart)** — v3
- **Automated cert expiry renewal** — v3
- **Worker daemon graceful 401 → re-auth** — v3
- **Container deploy (Docker / k8s)** — v3 deployment-as-product
