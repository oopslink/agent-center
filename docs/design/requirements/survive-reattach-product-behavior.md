# Survive & Re-attach — 产品行为说明（e2e 验收源）

> **用途**：本文是「worker 重启/部署不打断在飞 agent 工作」这一功能的**产品行为说明**，是 **e2e 验收的唯一权威依据**。
>
> - **e2e 测试只从本文 + 产品运维操作推导**；**dev 全程禁入 e2e**（不参与、不提供操作 facts、不给"怎么跑"）。
> - **机制级验证**（进程内部，如 pid 是否不变 / re-attach 而非 kill-relaunch）**不在本文**——它归**集成测试层**（见文末「不在本说明内」）。
> - 关联：[功能需求 F30](01-functional.md)、[v2.7 plan §5 D](../../plans/v2.7-domain-refactor-plan.md)。

## 1. 产品承诺

运维**重启或部署一个 worker** 时，该 worker 上**在飞的 agent 任务继续、不中断**：

- agent **不被杀+重起**，它的工作与会话**延续**；
- **无产出丢失、无缺口**；
- **不重复执行**（执行/副作用恰一次，见 §3.3 的精确口径）。

## 2. 运维操作（e2e 触发重启用——产品方式，非内部 kill）

e2e 通过**产品文档的运维方式重启 worker**来触发，**不使用任何内部 kill 技巧**：

> **停止该 worker 守护进程 → 用启动 worker 的同一产品命令再次起它**（`agent-center worker …`，经运维的服务管理器或重跑启动命令）。

这是运维层动作（停 + 起 worker）。e2e 据此触发，**不碰 dev 的内部进程组 kill（killpg）**——后者是机制细节、归集成层。

## 3. 用户可观测结果（e2e 断言源）

运维重启该 worker 后：

### 3.1 任务继续推进
在飞任务**仍为进行中、继续推进**；**不**表现为失败 / 重置 / 卡死。

### 3.2 不丢、无缺口
任务的产出 / 事件时间线**连续、无缺口、无丢失**——重启前已做的工作保留，重启后从断点继续。

### 3.3 执行 / 副作用**恰一次**（"无重复"的精确口径）
重启**不**导致工作被重复执行、副作用被重复产生：**任务消息发一次（非两次）、工具副作用一次、整段活不重跑**。

> ⚠️ **observability 活动事件流 = at-least-once（与已锁语义一致）**：活动 / 观测**事件流**在重连时**可能合法地重报某条事件**（at-least-once）——这是**设计内行为、判 PASS 不判 fail**。
>
> 本节"无重复"指的是**执行 / 副作用层的恰一次**，**不是**活动事件流逐条字节唯一。e2e **不得**把 at-least-once 的活动重报判为违规（那是 false-FAIL）。

### 3.4 agent 持续可用
在 fleet view / agent status 中，agent 跨重启**持续可用**——不是「掉线→再上线」。

## 4. 边界：真死 = 短暂恢复（非零中断）

若 agent 进程**真的没了**——**整机重启**，或 **agent 自身崩溃**（**不是**单纯的 worker 守护进程重启）——系统**恢复**它：重新拉起 agent + 续上会话。此情形**有一次短暂恢复中断**（agent 确实没了），但**恢复后工作续完、不丢**。

即 **Mode-B 用户结果 = 恢复（不是零中断）**：恢复后活续完、产出不丢、不重复执行。

## 5. 不在本说明内（归集成测试层）

**机制级保证**——即 **同一进程没死（pid 不变 / 从未退出）、worker 是 re-attach 而非 kill-relaunch**——是**进程内部属性**，本产品说明**不可观测、不写**。

它由**集成测试层**用内部观察（pidfile / instance 标识等）验证，是**真正区分"真存活" vs "够快的 kill-then-resume 假通过"**的那条断言。

- **e2e（本文）** 验**用户可观测结果**（§3）。
- **集成层** 验**机制存活**（同一进程没死）。
- **两层都绿 + PD 设计复审，才 ship**——单 e2e 会被够快的 kill-then-resume 假通过放过，集成层的"同一进程没死"才真堵假通过。
