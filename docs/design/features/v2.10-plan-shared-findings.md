# v2.10 Plan Shared Findings — design / scope

> **v2.10 headline.** 给 Plan 编排加一条 **DeLM 式「共享的、经校验的上下文」**：agent
> 把发现 / 失败 / 约束作为紧凑 **Finding（gist）** record 回 Plan；派单下游节点时把已
> 记录的 findings **注入到 @mention 内容**，让并行 / 下游 agent 在彼此的已验证进展上
> 接力，而不是失忆重来。决策见 [ADR-0053](decisions/0053-plan-shared-findings.md)。
> 来源：arXiv:2606.10662（DeLM, Stanford 2026-06）。

## 1. Why

Plan 编排（v2.9）已经能把工作拆成 DAG 并自动推进，但节点被派单时 agent 只拿到自己
任务的 `title + description`（`plan_lifecycle.go` `dispatchReadyNodes` 的 `content`）。
上游产出、兄弟分支的发现、踩过的坑，**都不跨 agent 共享**；任务结果只是 status 翻转。
DeLM 证明：把这些中间产物变成**紧凑、经校验、可复用**的共享状态，能显著提升 test-time
scaling 的质量与成本（SWE-bench +10.5pp，成本约减半）。

## 2. Model（confirmed，详 ADR-0053）

```
Project
 └── Plan
      └── 0..N PlanFinding        # plan 作用域共享上下文（pm BC 内新 AR）
           { id, plan_id, task_id(来源), project_id, author_ref, kind, content, created_at, version=1 }
```

- **PlanFinding** = 紧凑 gist。`kind ∈ {fact, failure, constraint, patch_summary}`（DeLM tag）。
- **content** 构造期强制 ≤ 4000 字符（compact；留在 DB TEXT，不入 BlobStore）。
- **不可变**：无 Update；改 = retract + 重记。
- **入库校验 v1**：author 必须 == 来源 Task 的 assignee（证据归属），且 project member、project 未 archived。

## 3. Lifecycle（end-to-end）

1. agent 执行某 Plan 节点 Task 时，调 `record_finding(plan_id, task_id, kind, content)`
   把发现写回 Plan 共享上下文（admission：它就是该 Task 的 assignee）。
2. 下游节点 upstream 全 done → orchestrator 派单：`dispatchReadyNodes` 把该 Plan 已记录
   的 findings 拼成「Shared context so far」段，**追加到 @mention 派单内容**，下游 agent
   收到任务时即带着已知进展。
3. built-in pull pool 的 agent（无 @mention）通过 `list_findings(plan_id)` 主动读。
4. 记录错了 → `RetractFinding`（作者或 project owner）。
5. Plan 被 delete → cascade 删其 findings。

## 4. Key reuse（keeps scope bounded）

- **AR / repo / 事件 / tx 范式**：完全 mirror Plan（`plan.go` / `sqlite/plan_repo.go` /
  `service/service.go` 的 emit + RunInTx）。
- **注入点**：`dispatchReadyNodes`（`plan_lifecycle.go`）一处，findings repo 在 Service 上
  OPTIONAL（nil-safe，mirror `plans` / `agentDir`）。
- **Agent tool 面**：mirror `agent_tools_plans.go`（`requireAgentOnWorker` → `agentActor`
  → `PMService.*` → `mapXxxToolError` → `writeJSON`）+ `mcphost` 注册 + parity test。
- **派单注入对结构化 Plan 生效**；pull pool 走 `list_findings`（pull 语义一致）。

## 5. New surfaces

### Backend（`internal/projectmanager`）
- `PlanFinding` AR（`plan_finding.go`）+ `PlanFindingKind` 枚举 + domain errors。
- `PlanFindingRepository` 接口 + `sqlite/plan_finding_repo.go` 实现。
- migration `0060_v210_plan_findings`（up/down）+ 5 处 `schema_version` 断言 59→60。
- AppService：`RecordFinding` / `ListPlanFindings` / `RetractFinding`；cascade 进 `DeletePlan`。
- 事件：`pm.plan_finding.recorded` / `pm.plan_finding.retracted`。
- 注入：`dispatchReadyNodes` 读 findings 拼接进派单 content（newest-first，cap 20，标注截断）。

### Agent MCP tools
- `record_finding(plan_id, task_id, kind, content)` → `{finding_id}`
- `list_findings(plan_id)` → `{findings:[...]}`
- admin route `POST /admin/agent-tools/record_finding` / `list_findings` + parity（`agent_facing_set.go`）。

### Skill 文档
- `assets/skills/supervisor.md`（或 worker-agent skill）补两条 tool 说明：何时记 finding、kind 语义。

### Frontend
- **推迟**（§6 节奏决策）：Web Console 查看/管理 findings 的 UI 入口 v2.10 不做；仅后端 + agent tool。

## 6. Phasing

- **P1（本次）**：domain + persistence + AppService + 事件 + 派单注入 + 2 个 MCP tool + skill 文档 + 全测试。
- **P2（roadmap）**：LLM-verifier admission gate（DeLM 完整 verified-before-admission）。
- **P3（roadmap）**：分层 unfold（gist→summary→raw，复用 ADR-0035 carry-over）；FE 查看 UI；task 失败自动落 FAIL gist；built-in pool 派单注入。

## 7. Acceptance（high level）

- **Backend unit**：PlanFinding 构造/枚举/长度校验；repo round-trip；RecordFinding admission
  矩阵（非 assignee→拒、跨 plan task→拒、archived project→拒、超长→拒）；事件发射；retract +
  cascade；派单注入（fake dispatcher 断言 content 含 findings、空时不注入、cap 截断标注）。
- **Integration**：admin handler `record_finding` / `list_findings`（鉴权 403 / 校验 400 /
  持久化 / 错误映射）。
- **Deployed-binary smoke（≥1）**：真 binary，agent 经 admin endpoint record_finding → 下游
  节点派单 @mention 内容含该 finding（端到端证明「共享上下文」价值），DB 可查。
- 行覆盖率（整体 + diff）≥ 90%；测试计划 + 报告分层（unit / integration-mocked / deployed-smoke）。

## 8. Out of scope / 推迟（§6）

- LLM-verifier、分层 unfold、FE UI、失败自动落 gist、built-in pool 注入 → 见 §6 P2/P3，登记 roadmap。
- Finding 可变 / 编辑 → 否决（不可变 + retract，ADR-0053 alt D）。
