# v2.9 Plan Orchestration — design / scope

> **v2.9 headline feature.** Turns agent-center from manual task dispatch into a
> multi-agent **workflow orchestration engine**: a PM agent (or human) groups
> pre-assigned tasks into a **Plan**, lays out a **DAG** of dependencies, and the
> system auto-advances the DAG — when a node's upstream is done it @mentions the
> node's assignee (human or agent) in the Plan's conversation to pull them in.
>
> Status: design confirmed with @oopslink 2026-06-10 (DM thread). Ready to break
> into P1 tasks. Owner: PD (design) → Dev (backend) + Dev2 (frontend) + Tester/Tester2.

## 1. Why

The PM domain today (`internal/projectmanager`: Project, Issue, Task,
ProjectMember, Subscriber, CodeRepoRef) has **no concept for grouping work into
parallel, ordered development plans** — no Milestone / Release / Sprint / DAG.
You cannot, e.g., run a v3.0 plan and a v4.0 plan in parallel, lay out the task
order, and have the system drive execution. v2.9's theme is strengthening
project-management capabilities; this is the centerpiece.

## 2. Model (confirmed)

```
Project
 └── Plan (计划)            # parallel-capable; v3.0 and v4.0 are two Plans
      ├── 1:1 Conversation  # auto-created; creator + all assignees auto-added
      └── selects N Tasks (subset of the project's backlog)
           └── DAG edges: Task.depends_on → Task   (acyclic; validated)
```

- **Plan** `{ id, project_id, name (e.g. "v3.0"), description/goal, status,
  creator_ref, conversation_id, optional target_date, created_at }`.
  - **status lifecycle**: `draft` (being assembled) → `running` (started,
    orchestrator active) → `done` (all nodes terminal) ; plus `failed`/`paused`
    only if we later add gates — **v1 has no gates** (fully autonomous).
- **Plan ↔ Conversation (1:1)**: creating a Plan auto-creates a dedicated
  conversation. The creator + every selected task's assignee are auto-added as
  participants. The orchestrator posts node-ready @mentions here; humans use it
  to discuss the plan. (Reuses the Conversation BC + channel/DM membership.)
- **Task ↔ Plan**: tasks are created and **assigned first**, living in the
  project **backlog** (not in any Plan). A Plan **selects a subset** of backlog
  tasks. A task is in **0..1** Plan.
- **DAG**: dependency edges between the Plan's tasks (`depends_on`). Acyclic,
  validated at edit time. **No Stage layer in v1** (deferred — could add an
  optional grouping layer later).

### Decisions locked (@oopslink)
1. **Fully autonomous, no approval gates.** Node fails → notify the **Plan
   creator**; if the creator is an agent it self-handles (adjust the DAG /
   escalate to a human), if a human, the human handles it. No system-level retry
   policy — the creator decides.
2. **Manual assignment.** PM agent or human pre-assigns each task; the system
   does **not** auto-pick agents by skill.
3. **Lifecycle** (§3).
4. **Status is owned by the assignee** as today (existing work-item `done` /
   `failed`); the system does not auto-detect completion.

## 3. Lifecycle (end-to-end)

1. Create a batch of tasks + assign them (human / PM agent) → they sit in the
   project **backlog**.
2. Create an **empty Plan** → **select a group** of backlog tasks into it.
3. **Generate the DAG** over the selected tasks: **PM agent drafts it from the
   discussion**, and a **human can edit it in the UI** (add/remove edges).
4. **Start the Plan** (human or agent action) → status `draft → running`.
5. The **orchestrator** advances the DAG:
   - A node whose **all upstream deps are `done`** becomes **ready**.
   - On ready → the system posts an **@mention to the node's assignee in the
     Plan conversation** (@human = "you're up" notification; @agent = wakes it
     via the existing wake+mention path to do the task).
6. The assignee sets the task's status as usual. On a task → **`done`**, the
   orchestrator re-evaluates downstream and dispatches the next ready nodes.
7. On a task → **`failed`**, notify the Plan creator (per decision 1).
8. When all nodes are terminal → Plan `running → done`.

## 4. Key reuse (keeps scope bounded)

- **Dispatch = @mention in the Plan conversation.** No new dispatch channel: a
  ready node just posts `@assignee` in the plan conversation → agent woken via
  the existing **WakeProjector / mention** path (#220); human notified.
- **Advancement trigger = work-item status → done.** The orchestrator subscribes
  to the existing work-item status-change event; no new completion detection.
- **Membership / conversation** reuse the Conversation BC (auto-add participants).
- **Runaway-safe by construction.** Tasks are human/PM-agent created and
  pre-assigned; the system never auto-spawns new tasks or agents, so there is no
  cost-runaway risk — resource guardrails are **not** a v1 blocker.

## 5. New surfaces

### Backend (`internal/projectmanager` + orchestrator)
- `Plan` aggregate + repo + migration; `task.plan_id` (nullable) + `task
  depends_on` edge table; acyclic validation.
- Plan ↔ Conversation creation + participant sync.
- **Orchestrator** (event-driven): on task `done`/`failed` for a task in a
  running Plan → recompute ready set → emit dispatch @mention(s) / creator
  notification. Idempotent + replay-safe (event-sourced or status-derived).
- APIs: create/select/edit-DAG/start/get-plan + plan list (parallel plans).

### PM-agent MCP tools (authoring)
- `create_plan(project, name, task_ids[])`, `add_task_to_plan` / `remove`,
  `add_dependency(from, to)` / `remove_dependency`, `start_plan(id)`,
  `get_plan(id)` (state + ready set). PM agent drafts the DAG from discussion.

### Frontend
- **DAG view** (per Plan): node = task (status: blocked / ready / running / done
  / failed), edges = deps; human can edit edges; progress (done/total).
- **Backlog → Plan** task selection UI.
- The Plan **conversation** surfaces in the existing conversation UI.
- Parallel-plans list at project level.

## 6. Phasing

- **P1 — model + manual advance + viz.** Plan + Plan-conversation + task↔plan +
  DAG (deps + acyclic) + DAG view + **manual "advance" ** (human clicks to
  dispatch the next ready node). Proves the model + UI without the orchestrator.
- **P2 — orchestrator (full auto).** Event-driven auto-advance on status→done +
  @mention dispatch in plan conversation + failure→creator notification.
- **P3 — PM-agent authoring.** MCP tools so a PM agent drafts plan + DAG from a
  discussion; human edits in UI.

## 7. Open / deferred
- **Stage** layer (optional grouping within a Plan) — deferred past v1.
- Plan-level priority / scheduling, target-date burndown — later.
- Auto agent-selection by skill — explicitly out (manual assignment in v1).
- Cross-project Plans — v1 is within a Project (confirm if needed).

## 9. Key definitions (v1 hardening — @oopslink 2026-06-10)

1. **Plan `done` / `failed` semantics.** `Plan.done` ⟺ **every node is `done`**.
   A `failed` node keeps the Plan **`running`** (its downstream subtree blocked)
   until the creator resolves it; in v1 the Plan **never auto-enters a terminal
   `failed`** — it waits on the creator. A Plan **cannot be `done` while any node
   is `failed` or non-terminal.** UI shows a derived `has_failed` indicator.
   Resolution = re-run the task (→ `done`, subtree unblocks) or `stop` → edit →
   restart (§9.4).

2. **Task status vs Plan-node status (do not conflate).**
   - **`Task.status`** = the authoritative work-item status, **owned by the
     assignee** (todo / in_progress / in_review / done / failed / closed). Sole
     source of truth for done / failed.
   - **Plan-node status** = **derived by the orchestrator, never stored as a
     competing field**: `blocked` (some upstream not done) → `ready` (all
     upstream done, not yet dispatched) → `dispatched`/`running` (mention posted;
     task in_progress) → `done` / `failed` (mirror the task terminal).
     `node_status = f(task.status, upstream-all-done?, dispatch-record)`. The
     **only** orchestrator-owned state is the dispatch record (§9.3); everything
     else derives from `Task.status` + the DAG.

3. **Dispatch idempotency.** Each plan-node has a **dispatch record**
   (`dispatched_at` + `dispatch_message_id`), written **once** when the @mention
   is posted. ready → dispatch fires **only if not already dispatched**, so
   re-evaluation / event replay / a second upstream completing **never
   double-@mentions**. An explicit creator **re-run** of a failed node clears its
   dispatch record so it dispatches again; normal re-evaluation does not.

4. **DAG editing while `running` = disallowed (v1).** The DAG is editable **only
   in `draft`**. To restructure a running Plan the creator **`stop`s** it
   (`running → draft`, orchestration halted), edits, then re-starts. Avoids
   mid-flight consistency issues; stays gate-free.

5. **Plan-conversation participant sync = additive (v1).** On Plan create → add
   the **creator**. Whenever a task is selected into the Plan or its assignee is
   set / changed → ensure that assignee (human **or** agent) is a participant.
   **Never auto-removed** in v1 (don't yank someone mid-plan; preserve history
   access). NB: participant membership is **required for dispatch** — a mention
   only reaches conversation members, so the assignee-agent must be a participant
   to be woken (ties to the wake+mention reuse).

6. **Start validation (`draft → running`).** Reject `start` unless: (a) the DAG
   is **acyclic**; (b) the Plan has **≥ 1 task**; (c) **every task has a
   resolvable assignee** (identity exists, agent not archived/deleted) — an
   unassigned node can never be dispatched; (d) all tasks belong to the Plan's
   project. Pre-`done` tasks are allowed (counted satisfied immediately).

7. **Failure → DAG behavior.** A `failed` node **blocks only its dependent
   subtree** (downstream never becomes ready); **independent branches keep
   advancing** (full-auto). The creator is notified and resolves it. Failure does
   **not** halt the whole Plan.

8. **Plan ↔ execution-DAG = 1:1.** A Plan owns **exactly one** execution DAG
   (over its selected tasks). The DAG is **not** shared across Plans and a Plan
   has no alternative DAGs. Combined with Task ↔ Plan = 0..1 (§2), a task's
   `depends_on` edges live in exactly one Plan's DAG — so the DAG, its node set,
   and its dispatch records are all owned by, and scoped to, that single Plan.
