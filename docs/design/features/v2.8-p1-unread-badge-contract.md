> in v2.1-C (read-state tuple + monotonic/idempotent advance + `countUnread` + event). The watermark

The frontend consumes three things: **(1)** per-conversation unread/mention **counts** (sidebar
badges), **(2)** a **read-position advance** call (mark-as-read), **(3)** **thread follow/unfollow**
state + toggle. All **human-only** (Q-T1: agents are directed-wake, never accumulate unread/badges).

## 1. Read watermark = the existing ULID message `id` (NO new `seq` column — C1)

**No `Message.seq` is added.** Messages already carry a ULID `id` whose lexical order IS chronological
order, and the existing `countUnread` computes unread as `id > last_seen_message_id`. Adding a
per-conversation int `seq` would mean a schema change + full backfill on the **append-only messages
table** for zero functional gain — rejected (Dev trunk-grounding).

- The read watermark / "last seen" unit is the **ULID `last_seen_message_id` (string)**, not an int.
- `Message` is unchanged (existing `id`, `conversation_id`, `sender_identity_id`, `content_kind`,
  `content`, `direction`, `posted_at`, `context_refs?`, `attachments?`). `GET
  /conversations/:id/messages` already returns messages in `id` (chronological) order.
- **Frontend unread divider** is drawn by comparing each message's `id` against `last_seen_message_id`
  (both already in the payloads) — no new field needed on the message.

## 2. Conversation DTO gains unread/mention/followed (per current user)

Badge source = the existing `GET /conversations` list (sidebar `useConversations({kind})`) and
`GET /conversations/:id`. Add three fields, **computed for the requesting human user**:

```ts
export interface Conversation {
  // ...existing (id, kind, name, status, participants?, peer_*, owner_ref?, ...)...
  unread_count: number;   // NEW — messages with id > my last_seen_message_id; 0 when caught up
                          //   (reuses v2.1-C countUnread, 999+ cap).
  mention_count: number;  // NEW — subset of unread that @-mention me (directed); 0 when none.
  followed: boolean;      // NEW — am I following this conversation/thread (see §4).
}
```

- **human-only (Q-T1) by data-shape:** counts are for human identities only. Backend **skip-writes**
  read/follow rows for agent identities (not merely zeroes them in the DTO) → an agent session always
  gets `unread_count: 0 / mention_count: 0 / followed: false` because the row simply doesn't exist.
  Agents are directed-wake (mention → execute), never "browse unread". Strongest enforcement.
- `mention_count ≤ unread_count` always (by construction — see mention computation below).
- **mention_count = best-effort bounded scan (mention(A), v2.8):** computed by scanning only the
  unread tail (`id > last_seen`, bounded by the 999+ cap) for the requesting user's `display_name`,
  reusing the proven wake-path matcher. **No new mentions schema.** Caveat: `display_name` substring
  match (same fuzziness the wake path already accepts). A precise write-time mention-projection table
  is deferred to v2.9 (mention(B)).
- For `kind: 'channel' | 'dm'` and the work-item thread conversations (`owner_ref` set), all carry
  these fields uniformly (surface-agnostic — sidebar #258 Issues/Tasks nav + Channels/DMs all badge
  off the same fields). Embedded per-row in `GET /conversations` (sidebar loop) + `GET
  /conversations/:id`; the standalone `GET /conversations/:id/unread` stays for back-compat.

## 3. Read-position advance (mark-as-read) — existing `POST /:id/seen` (C2)

Uses the **existing v2.1-C route** (`ReadStateService.MarkSeen`), NOT a new `/read`:

```
POST /api/conversations/:id/seen?org_slug=<slug>
body: { "last_seen_message_id": "<ULID>" }     # the id of the latest message the user has seen
→ 200 { "last_seen_message_id": "<ULID>", "version": <n>, "bumped": <bool>,
        "unread_count": <N−K>, "mention_count": <m> }    # recomputed (Dev ADDS the two counts)
```

Semantics (existing engine + the added recomputed counts):
- **only-forward / never regresses:** stored marker only advances to a later ULID; an earlier id is a
  no-op (`bumped:false`). (Layer-2 never-drift analog.)
- **idempotent:** repeating the same `last_seen_message_id` is a no-op, returns the same counts.
- **recomputed, not hardcoded 0:** if the latest message is N and the user marks K (K<N), the
  response `unread_count = N−K` (precise partial-read count via `countUnread`), not a forced 0.
- **author auto-advance-on-send (Q1):** sending a message auto-advances the **human** sender's own
  `last_seen_message_id` to the new message id (you've "seen" your own message) → a user's own
  messages never count as their unread. (Cleaner than sender-exclusion in the count query; human-only
  per skip-write.) **= full catch-up (Slack-standard, R3, intended — NOT a regression):** because the
  sent message is the latest id, advancing to it clears the sender's **pre-existing** unread above it
  too (their badge → 0 on send). Acceptance must assert this is by-design (a 3→0 badge drop on send
  is correct, not a bug — Tester §2.2 gate-1d / Tester2 §2.3 mirror).
- frontend calls `/seen` when the user views/scrolls a conversation to bottom (debounced); the sidebar
  badge then clears via query invalidation.

## 4. Thread follow / unfollow

```
POST   /api/conversations/:id/follow?org_slug=<slug>     → 200 { "followed": true }
DELETE /api/conversations/:id/follow?org_slug=<slug>     → 200 { "followed": false }
```

- follow-state is the **one genuinely-new schema in #268** (migration **0050**
  `user_conversation_follow_state`) — §9.4 schema-change discipline applies (5 assertion sites +
  collectKnownKeys + backfill watermark + real cross-version upgrade smoke on non-empty DB).
- state is **per-(human user, conversation id)**; reflected in `Conversation.followed` (§2). Agents
  skip-write (Q-T1).
- **auto-follow** on participate (send a message) or being @-mentioned (backend sets `followed=true`).
- unfollow → stops delivery/counting for that thread (mirrors the CLI `slock thread unfollow`).
- threads are **child conversations** (own row with `parent_conversation_id`); the read-state PK
  `(user_id, conversation_id)` gives each thread its own independent unread/follow row — parent does
  NOT roll up thread unread (D1 thread-reuse, per-thread granularity by construction).
