## 0. What an artifact is (one paragraph)

An **artifact** is a promotable, versioned snapshot of a single rich content block of a message —
created when a user clicks "Open in panel" on an eligible `<MessageBlock>` (`onSendToSide(block)`,
#174 §3). It gives that block a **stable identity** (deeplinkable URL), an **append-only version
history** (every edit is a new version; nothing is ever destroyed), and a **side-surface home**
(the `<ArtifactPanel>`, P2 frontend). The artifact does NOT replace the inline message block — the
message stays immutable (append-only per ADR-0031); the artifact is a parallel, evolvable view of
that block's content.

## 1. Entity schema (FROZEN shape — two tables)

ULID ids (conventions §0.3), TEXT timestamps RFC3339Nano, no FK declarations (app-layer ref
integrity per conventions §9.w, consistent with conversation tables). Reference integrity
(`organization_id`→organizations, `conversation_id`→conversations, `message_id`→messages,
`created_by`→identities) enforced at the app layer.

### 1.1 `artifacts` — the stable identity + latest pointer

```sql
CREATE TABLE artifacts (
    id               TEXT PRIMARY KEY,   -- ULID. STABLE across versions = the deeplink target.
    organization_id  TEXT NOT NULL,      -- org-scope (cross-org isolation, ?org_slug=).
    conversation_id  TEXT NOT NULL,      -- source conversation (the surface it lives under).
    message_id       TEXT NOT NULL,      -- source message whose block was promoted (#174 onSendToSide).
    block_ref        TEXT NOT NULL,      -- REQUIRED non-empty stable per-block address within the message (block index or block id). Part of the idempotency key (§3.2). Never '' — a message with N promotable blocks needs N distinct block_refs (Tester §3.2 lock-1.2: '' default would collide two blocks into one artifact).
    kind             TEXT NOT NULL,      -- artifact kind enum (§2; the side-eligible subset).
    latest_version   INTEGER NOT NULL,   -- monotonic high-water; points at the current artifact_versions row.
    created_by       TEXT NOT NULL,      -- identity ref (prefixed user:/agent:, ADR-0033).
    created_at       TEXT NOT NULL,
    updated_at       TEXT NOT NULL       -- last version-append time.
);

CREATE INDEX idx_artifacts_org_conv   ON artifacts (organization_id, conversation_id);
-- One artifact per promoted block (DB-enforced idempotency, §3.2): a repeat
-- POST for the same block resolves to the existing row, never a duplicate.
CREATE UNIQUE INDEX idx_artifacts_block ON artifacts (message_id, block_ref);
```

**`block_ref` is REQUIRED + non-empty (Tester §3.2 lock-1.2).** It is the block's stable address
within its message (the block's index in the message DTO's `blocks[]`, or a block id if the message
model assigns one — #174 `MessageRow` maps `blocks[]` → N `<MessageBlock>`). A `POST` with empty/missing
`block_ref` → `400 invalid_block_ref`. The `UNIQUE (message_id, block_ref)` index is the schema-level
guarantee that one promoted block ↔ exactly one artifact.

**Why id ⊥ version (Tester2 G3.3-E):** `artifacts.id` is the permanent identity used in the
deeplink. Editing the artifact appends a new `artifact_versions` row and bumps `latest_version` — the
`id` never changes, so `/<surface>?artifact=<id>` survives every edit.

### 1.2 `artifact_versions` — append-only version history

```sql
CREATE TABLE artifact_versions (
    artifact_id   TEXT NOT NULL,         -- FK→artifacts.id (app-layer).
    version       INTEGER NOT NULL,      -- 1-based, monotonic, gapless per artifact.
    payload       TEXT NOT NULL,         -- kind-specific JSON (§2), the version's full content snapshot.
    summary       TEXT NOT NULL DEFAULT '', -- one-line collapsed-card summary (mirrors #174 MessageBlockData.summary).
    created_by    TEXT NOT NULL,         -- who authored this version (human or agent).
    created_at    TEXT NOT NULL,
    PRIMARY KEY (artifact_id, version)
);
```

**Append-only (Q-T2):** rows are only ever INSERTed. No UPDATE of an existing version's payload
(PATCH appends `version+1`), no DELETE (§5). History-back = `SELECT ... ORDER BY version DESC`.

## 2. Kind enum + payload (the side-eligible subset of `MessageBlockKind`)

The artifact `kind` enum is the **side-surface-eligible subset of `MessageBlockKind`** (#174 §2
eligibility column) — exactly the 4 kinds a block can be promoted from:

```
code | diff | tool_use | tool_result
```

> **Design decision — LOCKED by PD 2026-06-05 (4-subset).** PD's #269 scope initially said "kind enum
> 8 kinds"; Dev2 (frontend consumer) and Tester2 (UI acceptance) asked for the **4-kind side-eligible
> subset**, and **PD decided 4-subset** (PR #180 §-1). Rationale: an artifact can *only ever* hold a
> promoted block, and the other 4 MessageBlockKinds are **not promotable** (`text` always inline;
> `attachment` downloads; `image` → lightbox not side; `embed` → card) — including them would create
> dead, unrepresentable enum values; the enum should only admit values the app actually creates
> (kind expansion = a product question "what does promoting an attachment mean?", deferred together);
> forward-compat is weak for an internal schema we control (a new kind is a trivial v2.9 widening).
> The enum is therefore the explicit promotable subset — still single-source-aligned with
> `MessageBlockKind` (it *is* its eligibility subset), without the dead values. `POST /api/artifacts`
> with any other kind → `400 invalid_kind`.

`payload` is the **same per-kind shape as #174 §2** (complete-consumable, resolved — frontend does
zero assembly, #258 DTO discipline), stored as JSON text:

| kind | payload JSON | summary example |
|---|---|---|
| `code` | `{ "lang": string, "source": string }` | `Code · ts · 142 lines` |
| `diff` | `{ "path"?: string, "unified": string }` | `Edited api/x.go (+42 −7)` |
| `tool_use` | `{ "tool": string, "argsJson": <any> }` | `Searched "foo"` |
| `tool_result` | `{ "status": "running"\|"ok"\|"error", "outputJson": <any> }` | `Ran cmd · ok` |

## 3. Endpoints + complete-consumable DTOs

All routes org-scoped via `?org_slug=` (or `?org_id=`), `requireOrgMember` (JWT session), same axis
as #258/#260. All read/write fail-closed cross-org (§9). **There is deliberately NO delete route
(§5).**

### 3.1 Artifact DTO (the wire shape — complete-consumable)

```jsonc
{
  "id": "01J...",                       // bare ULID = the artifact identity + deeplink target
  "organization_id": "01J...",          // bare
  "conversation_id": "01J...",          // bare (the surface)
  "message_id": "01J...",               // bare (source message)
  "kind": "code",
  "version": 3,                          // the version represented by THIS response (latest by default)
  "latest_version": 3,                   // current high-water (so client knows if it's viewing history)
  "payload": { "lang": "ts", "source": "..." },
  "summary": "Code · ts · 142 lines",
  "created_by": { "ref": "user:01J...", "display_name": "Hayang" },  // enriched (#12.x ref-vs-id, like #260)
  "version_author": { "ref": "agent:01J...", "display_name": "AgentBeta" }, // author of THIS version
  "created_at": "2026-06-05T16:00:00Z", // artifact creation (v1)
  "updated_at": "2026-06-05T16:10:00Z", // last version append
  "deeplink": "/channel?artifact=01J..." // ready-to-use deeplink (§6), client appends nothing
}
```

### 3.2 Routes

| method + route | purpose | body | returns |
|---|---|---|---|
| `POST /api/artifacts?org_slug=` | promote a block → new artifact (v1), **idempotent** on `(message_id, block_ref)` | `{ conversation_id, message_id, block_ref, kind, payload, summary }` | Artifact DTO (version=1 on create; existing artifact on repeat) |
| `GET /api/artifacts/{id}?org_slug=` | read latest version (default) | — | Artifact DTO (latest) |
| `GET /api/artifacts/{id}?org_slug=&version=N` | read a specific version (history-back) | — | Artifact DTO (version N) |
| `GET /api/artifacts/{id}/versions?org_slug=` | version history (metadata, DESC) | — | `[{ version, summary, version_author, created_at }]` |
| `GET /api/artifacts?org_slug=&conversation_id=` | list artifacts in a conversation | — | `[Artifact DTO (latest)]` |
| `PATCH /api/artifacts/{id}?org_slug=` | append a new version (edit) | `{ payload, summary }` | Artifact DTO (version+1) |
| `GET /api/artifacts/thresholds?org_slug=&conversation_id=` | effective R2 promote thresholds | — | Threshold DTO (§7) |
| `PATCH /api/artifacts/thresholds?org_slug=&conversation_id=` | set per-org / per-conversation override (admin/owner) | `{ code_lines?, json_bytes?, doc_lines? }` | Threshold DTO (effective) |

**POST is idempotent on `(message_id, block_ref)` — FROZEN contract guarantee, by-API-shape
(Tester §3.2 lock-1.1):** a repeat `POST` for an already-promoted block returns the **existing**
artifact (its current DTO), never creates a duplicate. This is enforced server-side by the
`UNIQUE (message_id, block_ref)` index (§1.1) + an upsert-or-return handler — it does **not** rely on
the frontend remembering `artifactId` (same enforcement strength as Q-T1/Q-T2 by-API-shape). So a
direct double-`POST` (no frontend guard) yields one artifact, not two (Tester §3.2 #4 linkage).
Status: `201` on first create, `200` on idempotent return.

**PATCH = append, never clobber (Tester §3.2 #2):** `PATCH /api/artifacts/{id}` inserts
`artifact_versions(version = latest_version+1, ...)` and bumps `artifacts.latest_version`; the prior
version row is untouched and remains fetchable via `?version=N`. There is no in-place payload mutation.

**List pagination (Tester §3.2 clarification i):** `GET /api/artifacts?conversation_id=` is
**unpaginated in v2.8** — returns all artifacts in the conversation (bounded by the natural cardinality
of promoted blocks per conversation), latest version each, ordered `created_at` DESC. Pagination is a
v2.9 item if a conversation accrues many artifacts (flag, no implicit gap).

**Errors:** `404 artifact_not_found` (missing/cross-org); `400 invalid_kind` (not one of the 4
promotable kinds, §2); `400 invalid_block_ref` (empty/missing `block_ref` on POST, §1.1);
`400 invalid_payload`
(payload doesn't match the kind shape); `404 conversation_not_found` / `404 message_not_found` on POST
(linkage integrity, Tester §3.2 #4); `403 not_a_member` (org gate). No `409` for versioning — version
allocation is server-assigned + monotonic (see §4).

## 4. Versioning semantics (FROZEN)

- **Monotonic + gapless:** `version` starts at 1 (POST) and increments by exactly 1 per PATCH,
  server-assigned inside a tx (`SELECT latest_version FOR UPDATE`-equivalent under WAL serialization,
  same idiom as #245 org-sequence) — never client-supplied, never regresses.
- **Latest is default:** `GET /{id}` with no `?version=` returns `latest_version`.
- **History-back:** `GET /{id}?version=N` returns version N (1 ≤ N ≤ latest_version; out-of-range →
  `404`). `GET /{id}/versions` lists all versions DESC for the panel's history dropdown.
- **id ⊥ version (deeplink stability):** the deeplink targets `id`; editing never changes `id`, so a
  shared/bookmarked deeplink always resolves to the latest version of the same artifact (Tester2 G3.3-E).
- **Concurrent PATCH:** two concurrent edits both append (no lost update) — version allocation is
  serialized, so two PATCHes yield versions N+1 and N+2 (both preserved, append-only). No 409; this is
  intentional — append-only means concurrent edits coexist as sibling versions, the panel shows the
  latest + history.

## 5. Q-T2 — append-only, NO delete (LOCKED, by API-shape)

- **There is NO `DELETE /api/artifacts/{id}` and NO `DELETE /api/artifacts/{id}/versions/{n}` route.**
  Absence is **by API-shape, not a hidden UI control** (same enforcement strength as #176 Q-T1
  human-only "by data-shape"): the capability does not exist server-side, so no client can invoke it.
- **Implementer constraint (explicit):** the P2 implementer MUST NOT add any delete/destroy path
  (route, handler, repo method, or service op) for artifacts or artifact_versions in v2.8. "Undo" is
  served by versioning (edit → new version; the prior version remains reachable) + (future) soft-delete
  in v2.9.
- **Deferred-with-pointer (conventions §15.x, no implicit gap):** artifact **Delete** is deferred to
  **v2.9 via task #267** (@oopslink lock 2026-06-05). Rationale: preserves complete version history per
  D4 versioning intent (delete would break history-back + deeplink); minimal P2 scope; observed product
  preference for new-version-over-delete is already covered by versioning. This is bound to
  `v2.8-acceptance-plan.md` §3.4 (which must state it verbatim — Tester §3.2 #7 / Tester2 G3.3-F2
  doc-evidence gate; if §3.4 lacks it at the P2 acceptance tag, T-9 fails).
- v2.9 design re-evaluates **soft-delete/archive** (preferred if any) over hard-delete (rejected —
  breaks deeplink + history-back).

## 6. Deeplink convention (#192 id-as-content)

- **Form:** `/<surface>?artifact=<id>` where `<surface>` is the conversation route (channel/DM/task/
  issue) and `<id>` is the bare artifact ULID. Backend emits the ready-to-use `deeplink` field in the
  DTO (§3.1) so the client appends nothing.
- **id-as-content (#192-exempt):** the artifact ULID in the URL is **content, not chrome** — it is the
  one place a raw ULID legitimately appears (like a permalink), exempt from the "no raw id" rule. The
  artifact's *display* name/chrome elsewhere still follows #192 (no raw id in chrome).
- **Resolution:** loading a deeplink → `GET /api/artifacts/{id}?org_slug=` → panel opens to the latest
  version. Bad/foreign/cross-org id → clean `404` (no existence leak, Tester §3.2 #6). Reload preserves
  state because the id is stable (§4).
- **Deeplink resolves to LATEST — version-view is transient (Tester2 + Dev2 eyes-open, G3.3-E):** the
  deeplink carries only `<id>`, so reload always re-opens the artifact's **latest** version. Browsing
  history-back (`GET /{id}?version=N`) is an **in-panel transient interaction** — it is NOT encoded in
  the URL and NOT reload-persisted. Therefore "reload did not preserve the specific version N I was
  viewing" is **intended, not a bug** (G3.3-E asserts reload re-opens to latest). Sharing a specific
  version via `?artifact=<id>&version=N` is a **v2.9 option** (backend already supports `?version=N` on
  the GET; the deeplink-with-version URL form + frontend wiring is deferred).

## 7. R2 auto-promote thresholds (config, not literals — storage + effective-read)

Thresholds decide when a block auto-collapses + surfaces "Open in panel" (#174 §4). **Client default +
server override** (R2 lock), **never hardcoded** (acceptance §3.3 G3.3-C).

### 7.1 Defaults (client) + override storage (server)

- **Client defaults:** `code_lines = 20`, `json_bytes = 5120` (5 KB), `doc_lines = 200` (#174 §4).
- **Server override storage** — a small table keyed by scope:

```sql
CREATE TABLE artifact_promote_overrides (
    scope_kind   TEXT NOT NULL,   -- 'org' | 'conversation'
    scope_id     TEXT NOT NULL,   -- organization_id or conversation_id
    code_lines   INTEGER,         -- NULL = inherit
    json_bytes   INTEGER,         -- NULL = inherit
    doc_lines    INTEGER,         -- NULL = inherit
    updated_at   TEXT NOT NULL,
    PRIMARY KEY (scope_kind, scope_id)
);
```

### 7.2 Effective resolution (precedence) + read endpoint

- **Precedence (most specific wins):** conversation override → org override → client default. Each
  field resolved independently (a conversation may override only `code_lines`, inheriting the rest).
- **`GET /api/artifacts/thresholds?org_slug=&conversation_id=`** returns the **effective** values so
  both the frontend AND Tester2 (G3.3-C) can read the in-effect threshold to verify "config took
  effect, not coincidence":

```jsonc
{
  "code_lines": 40,        // effective (e.g. conversation override)
  "json_bytes": 5120,      // effective (client default; no override)
  "doc_lines": 200,
  "source": {              // provenance per field (so the verifier knows WHY)
    "code_lines": "conversation",
    "json_bytes": "default",
    "doc_lines": "default"
  }
}
```

- **`PATCH /api/artifacts/thresholds`** (admin/owner only) upserts an override row → G3.3-C can change
  server config, reload, and observe a previously-inline artifact now auto-promote (or vice-versa).

## 8. §9.4 schema-change discipline (migration planning)

`artifacts` + `artifact_versions` (+ `artifact_promote_overrides`) is a NEW backend schema = one
Schema 工作单位 (conventions §9.4). The migration (e.g. `0051_v28_artifacts` — number per trunk at
impl start) must satisfy, as its own gate independent of runtime CRUD:

1. **up + down** present (down drops the 3 tables + indexes).
2. **schema_version assertion sites bumped in lockstep** — the canonical set (verified live in #268,
   currently 7 occurrences for the prior bump): `internal/cli/handlers_migrate_v1_to_v2.go`
   (`targetSchemaVersion` const + the comment) + its `_test.go` (3 numeric strings:
   "target schema version", "new schema version", post-apply version) + `internal/persistence/
   migrator_test.go` + `internal/persistence/migration_round_trip_test.go` + `tests/integration/
   integration_test.go`. Bump all in one commit (the §9.4 plan calls it "5 sites"; the concrete trunk
   currently has 7 numeric occurrences — bump every one).
3. **`collectKnownKeys`** (config.go allowlist) updated **only if** a new config YAML key is added.
   The R2 thresholds use a DB table (§7), NOT a config key, so collectKnownKeys is **N/A for this
   migration** (documented so a reviewer doesn't expect a config bump) — unless impl chooses to also
   expose a global default via config, in which case add the key + allowlist entry.
4. **Real cross-version upgrade smoke on a NON-EMPTY database** (the #245/#251/#268 lesson, Tester
   §3.2 #8 口径 — NOT an empty-DB round-trip):
   - install a prior version (v2.7.1 / v2.8-P1 = pre-#265) carrying **real conversations + messages**
     (the artifacts' source data); the artifact tables do not yet exist.
   - upgrade across this migration → assert the 3 tables are created **on the non-empty old DB** and
     prior data is intact (schema_version advanced; conversations/messages untouched).
   - then **promote a real existing message's eligible block → artifact** (POST) + PATCH a new version
     + GET latest/by-version + threshold GET — i.e. the new CRUD works against **real pre-existing
     data**, not just a fresh seed. (Mirrors how #268 mig 0050 was verified creating a table on a
     non-empty DB + resolving old rows.)
   - This is its own gate, independent of unit migration tests + runtime CRUD smoke.

## 9. Cross-org isolation (#258 axis re-applied)

- Every route is org-scoped via `requireOrgMember(?org_slug=/?org_id=)`; the artifact's
  `organization_id` is set from the resolved org at POST (never client-supplied trust).
- `GET`/`PATCH`/list fail-closed cross-org: orgA cannot read/edit/list orgB's artifact → `404`
  (no existence leak), list scoped to caller org. Deeplink to a foreign artifact → `404` (§6).
- Linkage: POST validates `conversation_id` + `message_id` belong to the caller's org (and the
  message belongs to the conversation) → no cross-org artifact creation, no orphan (Tester §3.2 #4).

## 10. `onSendToSide(block)` → artifact mapping (the #174 §7 seam)

How a P1 `<MessageBlock>` becomes an artifact (closes #174 §7's open seam):

> **Decision: lazy-create (b), not pre-create (a)** — Dev2 day-0 touchpoint #1. An artifact is created
> on first `onSendToSide` (lazy), NOT pre-created for every promotable block at message render.
> Rationale: minimal storage (most blocks are never promoted), clean separation (an artifact = a
> *deliberately* promoted thing), and no coupling of message-write to artifact-create. Cost: an
> un-promoted inline block has no deeplink — which **Tester2 confirmed is fine** (G3.3-E only requires
> *promoted* artifacts to deeplink; both (a) and (b) pass §3.3). The universal-deeplink option (a)
> (pre-create so any block is deeplinkable) is a **v2.9 option** if product wants it later — it widens
> here without breaking this contract (the `block_ref` address + idempotent re-promote in step 5 are
> the forward-compat seam).

1. Frontend: user clicks "Open in panel" on an eligible block → `onSendToSide(block)` (#174 §1/§3).
2. If `block.artifactId` is already set → it is already promoted; GET that artifact (open panel). Else:
3. `POST /api/artifacts` `{ conversation_id, message_id, block_ref, kind: block.kind,
   payload: block.payload, summary: block.summary }` → returns the Artifact DTO with a fresh `id`.
4. Frontend stores the returned `id` back as the block's `artifactId` (the #174 `MessageBlockData.
   artifactId` seam) so re-clicking re-opens the same artifact instead of creating a duplicate.
5. `block_ref` records which block of the message was promoted (the message may have N blocks); it is
   **required + non-empty** (§1.1) and, combined with `message_id`, is the stable block address.
   Re-promotion of the same block returns the existing artifact — this idempotency is a **FROZEN
   contract guarantee** (§3.2), enforced by the `UNIQUE (message_id, block_ref)` index server-side, NOT
   a frontend-only `artifactId` guard. So the frontend's `artifactId` cache (step 4) is an
   optimization to skip the round-trip; correctness (no duplicate) holds even on a cold direct POST.

**Promotion does not mutate the message** (append-only, ADR-0031): the inline block stays; the
artifact is a parallel evolvable view.

## 12. Frozen vs impl-detail; open seams; deferred

**Frozen (cross-phase contract — changing = explicit break + re-review):**
- The two-table schema shape (§1) + id⊥version + append-only.
- The kind enum = `MessageBlockKind` 8-set, 4 promotable (§2).
- The endpoint set (§3.2) + Artifact DTO (§3.1) + **the deliberate absence of any delete route** (§5).
- **POST idempotency on `(message_id, block_ref)` by-API-shape + `block_ref` required non-empty +
  `UNIQUE (message_id, block_ref)`** (§1.1/§3.2, Tester §3.2 lock-1) — one promoted block ↔ one artifact.
- Versioning semantics (§4) + deeplink form (§6) + threshold effective-read shape (§7.2).

**P2 implementation detail (may refine freely):** repo internals, the exact migration number, tx
mechanics, whether the artifact lives in a new `artifact` BC vs a Conversation-BC sub-entity (a BC
placement decision for impl — this contract is BC-agnostic on storage), index tuning, JSON
(de)serialization of payload.

**Open seams for P2 impl:** BC placement (Conversation sub-entity vs new Artifact BC); whether
thresholds also expose a global config default (§8.3); the exact form of `block_ref` (block index vs a
block id if the message model assigns one) — either is fine as long as it's stable + non-empty + unique
per block (§1.1). (Idempotency-on-re-promote is no longer an open seam — it's a frozen guarantee, §3.2.)

**Deferred-with-pointer (no implicit gaps, §15.x):**
