## 0. Method & rigor lens

Every observation is judged through, not just described:
`ui-check-template` + `ux-standards` (§2b #192 three-layer chrome/content/id-as-content, §12 icon a11y)
+ `acceptance-methodology` (computed-truth / real usage). For each competitor behavior ask:

> Is this **usable · consistent · no-middle-state · a11y · no raw-id leak**?

Output is **学/避 (learn / avoid)**, not just "looks nice". When a competitor does something we'd
**avoid**, say why against our rules (e.g. native dialog, emoji soup, color-only state, raw ULID).

**Row format** (one line per competitor per dimension):

| Competitor | Screenshot OR public source | Approach (what they do) | 学/避 + rigor verdict |
|---|---|---|---|

**Screenshots are optional** — the deliverable value is the **学/避 rigor verdict**, which does not
depend on personally capturing a screenshot. Many of these are proprietary live apps we can't
log into from a sandbox; that's fine. Conventions (doc-honesty: source must be cited):
- Have a screenshot → commit it to `docs/design/assets/v28-ux/` **in the same PR/commit** as the
  finding (T-5 evidence-in-tree extended to design docs) and annotate per `ui-check-template`
  (chrome / content / id-as-content + a11y).
- No screenshot → write `N/A — source: <official doc / changelog / article link>` or
  `screenshot pending: @<owner>`. Public materials + our real-usage memory + the rigor lens are a
  sufficient analysis basis.
- Open-source competitors (PilotDeck, Element, Zulip) → **code/render-level analysis** is richer
  than a screenshot; the source itself is the evidence.

Screenshot capture (subject to access): Slack/Discord → Dev2 (incidental, primary scanner);
Notion/ChatGPT/GitHub Discussions → PD if access; Claude.ai/Cursor → real-usage memory + public
docs. Real-env holders (@oopslink / Dev2 dev machine) drop captures into `assets/` incrementally;
the dim owner annotates + adds the rigor verdict.

**C 维综合（deep synthesis → Slock #248 `#ref` epic 喂料）**

- **共识①「trigger-char 唤模糊 picker + 键盘优先」**：@（人/agent）/ #（频道/实体）/ /（命令）全竞品一致 = composer 输入辅助 table-stakes。Slock #248 `#ref` 语法应配 **inline picker**（输 `#` 即弹 channel/issue/task/project 模糊列表、键盘选），不要求用户手敲全 ref。
- **共识②「渲染 = chip 显名存 id」**：所有竞品 mention 渲成 chip 显**名**、底层存 **id**、点可跳、hover 出卡 = 正是 Slock **#192 EntityRef**（名见 / 裸 id 不露脸 / hover 全 id）。#248 渲染**直接复用 EntityRef**，零 raw-id 泄漏。
- **共识③（Linear）「引用 chip 带活状态」**：issue/task 引用 chip 可带 status（呼应 #258 StatusChip + #245 org_ref）→ Slock `#T34` 可渲成 `T34 · running` chip；但**状态须实时取**否则违 no-middle-state（宁可不带也不带过期）。
- **rigor 红线**：① picker 行的人/agent 要**类型区分**（icon/标签，不靠单色，呼应 §A）防选错 ② chip 渲染**不漏裸 ULID**（#192）③ agent vs human 在 `@` picker 与渲染都要可分 ④ `@everyone` 类群发要有通知治理（→ §F）。

**Slock 落地成本 / 边界**（C 深，直接支撑 #248）

- **#248 `#ref` = 后端 parse + 前端 picker + 渲染 三件套**：picker（composer 输 `#`/`@` 弹模糊列表）是**新前端组件**；渲染复用 **EntityRef**（已有 #192）成本低；后端需 ref 候选解析 endpoint（输前缀→候选实体）。proposal 标清三件套边界与成本。
- **复用既有 trunk**：渲染层 EntityRef（#192）+ #258 StatusChip（带状态 chip）+ #245 orgRefToken（I<n>/T<n>）已在 trunk → #248 渲染侧大量复用，新建主要在 **picker 组件 + 后端候选解析**。
- **identityRef 统一**：`@` 渲染人/agent 走 prefixed ref（agent:/user:），正是 deferred **#254 identityRef(kind,id) 共享 helper** 用武之地 —— #248 落地时一并提取（防 refOf 再散落，#240 根因）。
- **agent 特异**：`@` picker 要列 agent + human 且可分（icon）；`#` picker 要含 #258 跨 project 的 issue/task/project 实体。

**F 维综合（Dev2 半：unread/mention/multi-agent → Slock 喂料）**

- **共识①「mention badge 带精确计数」**：Slack/Discord mention 都显数字（非红点）= 用户可量优先级 → Slock #258 sidebar 的 Channels/DMs/Issues/Tasks 入口应带**未读/mention 计数 badge**。

**Slock 落地成本 / 边界**（F 深）
