This is the single seam between a *message* and its *rich content*. It is **render-only**: a
`MessageBlock` takes already-resolved block payload (from the message DTO) and renders it; it does
not fetch, mutate, or own data. It composes inside `<MessageRow>` (one row may have N blocks).

## 1. Interface (FROZEN)

```ts
// One renderable content block within a message. Surface-agnostic (channel/DM/task/issue thread).
export interface MessageBlockProps {
  block: MessageBlockData;            // the resolved block payload (see §2)
  isCollapsed: boolean;               // collapsed card (summary) vs expanded inline (full)
  onExpand: (collapsed: boolean) => void;  // toggle inline collapsed<->expanded
  onSendToSide: (block: MessageBlockData) => void; // open this block's artifact in Side Surface (P2)
  // context (read-only): which surface + the owning message, for #192 chrome resolution
  surface: ConversationSurface;       // 'channel' | 'dm' | 'task-thread' | 'issue-thread'
  messageId: string;
}

export type MessageBlockKind =
  | 'text'        // markdown prose (default; never collapses)
  | 'code'        // fenced code, syntax-highlighted + copy
  | 'diff'        // unified diff, +/- prefix + coloring (not color-only)
  | 'tool_use'    // agent tool invocation (name + args)  — #216 category
  | 'tool_result' // agent tool result (status + output) — #216 category
  | 'attachment'  // generic file card (icon + name + size + download)
  | 'image'       // inline image (→ lightbox on click)
  | 'embed';      // link unfurl card (OG meta) — SSRF-guarded backend (§5.7)

export interface MessageBlockData {
  kind: MessageBlockKind;
  // human-readable one-line summary for the collapsed card ("Read file X", "Ran cmd", "+42 −7").
  // Backend-provided (no-middle-state: specific, never vague like "Analyzing"). REQUIRED for
  // collapsible kinds; ignored for 'text'.
  summary?: string;
  payload: unknown;          // kind-specific (see §2); typed per-kind at the renderer
  // optional id when the block is a promotable artifact (P2): stable artifact id once promoted.
  artifactId?: string;
}
```

`ConversationSurface` is the same union P1's `<ConversationView surface>` uses (one source).

## 2. Per-kind payload shapes

| kind | payload | collapses? | summary example | side-surface eligible (P2) |
|---|---|---|---|---|
| `text` | `{ markdown: string }` | no (always inline) | — | no |
| `code` | `{ lang: string; source: string }` | yes (>20 lines → default collapsed, R2) | `Code · ts · 42 lines` | yes |
| `diff` | `{ path?: string; unified: string }` | yes | `Edited api/x.go (+42 −7)` | yes |
| `tool_use` | `{ tool: string; argsJson: unknown }` | yes (default collapsed) | `Searched "foo"` / `Read file X` | yes |
| `tool_result` | `{ status: 'running'\|'ok'\|'error'; outputJson: unknown }` | yes | `Ran cmd · ok` | yes |
| `attachment` | `{ name: string; size: number; mime: string; href: string }` | card (no expand) | — | no (download) |
| `image` | `{ alt: string; src: string; w?: number; h?: number }` | thumbnail → lightbox | — | lightbox (not side) |
| `embed` | `{ url: string; title?: string; desc?: string; image?: string }` | card | — | no |

Backend supplies `summary` + `payload` already resolved (complete-consumable; frontend does zero
assembly — mirrors the #258 DTO discipline). `tool_use`/`tool_result` reuse the **#216
`agent-activity-payload-json` JSON-viewer** for the expanded body.

## 3. Dual-state behavior (§5.4 + D4)

Three independent display states, user-controlled (per D4 "two buttons non-exclusive"):

1. **collapsed card** — `isCollapsed=true`: shows `summary` + chevron + (if eligible) "Open in panel".
   Click chevron → `onExpand(false)`.
2. **expanded inline** — `isCollapsed=false`: renders full content inline (code/diff/JSON viewer).
   Click chevron → `onExpand(true)`.
3. **side surface** — "Open in panel" → `onSendToSide(block)`. **Independent** of inline state
   (both can be open). P2 owns the panel; P1 only emits the callback. Inline expand and panel-open
   never block each other.

**Auto-promote (R2, configurable not hardcoded):** when a block exceeds the threshold
(code >20 lines / JSON >5 KB / doc >200 lines — client default + per-org/conversation server
override) it defaults to **collapsed card** + surfaces the "Open in panel" affordance. Thresholds
read from config, never literals (acceptance #263 §3.3).

## 4. Composition

```
<ConversationView surface id>
  └─ <MessageList>                    virtualized stream
        └─ <MessageRow>              author chrome (EntityRef #192) + reactions
              └─ <MessageBlock> ×N   one per content block; passes surface+messageId down
```

`<MessageRow>` maps a message DTO's `blocks[]` → N `<MessageBlock>`. A plain text message = one
`text` block. An agent turn = `text` + `tool_use` + `tool_result` (+ `code`/`diff`) blocks.

## 5. Boundaries (carried from §5.6/§5.7)

- **#192 three-layer within a block:** block *chrome* (tool name / file path / agent ref / artifact
  link) renders via **EntityRef** (name visible, raw id never, hover full id). Block *body*
  (args/result/code/diff) is **content-exempt** (raw allowed). The renderer must not leak a bare
  ULID in chrome.
- **a11y:** collapsed card is a `button` with `aria-expanded`; expanded body is the disclosure
  region. `aria-live` (if any running `tool_result`) goes on a **summary region, not per-step**
  (avoid screen-reader noise). Not-color-only (diff +/- prefix, status icon+label). Keyboard:
  Enter/Space toggles collapse; "Open in panel" is a real button.
- **no native dialog:** lightbox (image) reuses the in-app modal a11y pattern (ConfirmModal #169:
  focus-trap + Esc), NOT `<dialog>`. (Side Surface is non-modal — P2.)
- **render-only / no-middle-state:** block shows current resolved state or nothing — never a stale
  half-rendered status. Data ownership stays in the message DTO / backend.
